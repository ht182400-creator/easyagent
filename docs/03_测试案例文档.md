## 1.5 Server 集成测试 (新增)

### 测试模块：API 端点 (27 用例)

- **/api/health**: 返回200 + status/version/uptime/memory/timestamp
- **/api/status**: 返回model/tokenUsage/sessionCount/toolCount/providerCount
- **/api/version**: 返回版本号 + codename + 发布日期 + changelog
- **/api/version/check**: 检查 GitHub Release 是否有新版本
- **/api/tools**: 工具数组70项 + 分组校验(≥5分组) + 字段完整性
- **/api/plugins**: 插件列表 + 字段校验
- **/api/skills**: 技能列表 + 字段校验
- **/api/config**: GET/PUT 配置读写
- **/api/providers**: 提供商列表(含动态模型) + 连接测试
- **/api/sessions**: 会话列表 + 字段校验
- **/api/im/status**: IM状态返回
- **/api/sandbox/status**: 沙箱状态(docker/sandbox字段)
- **/api/semantic/overview**: 语义概览
- **API 404路由**: `/api/nonexistent` 返回JSON 404
- 🆕 **知识库 API — 双作用域 (9 用例)**:
  - `GET /api/knowledge?scope=project` → 项目知识库数据 + scope 字段
  - `GET /api/knowledge?scope=global` → 全局知识库数据 + scope 字段
  - `GET /api/knowledge/stats/summary` → 合并模式 (scope:merged + project + global)
  - `GET /api/knowledge/stats/summary?scope=global` → global 单域统计
  - `GET /api/knowledge/search?q=test&scope=project` → 搜索 + scope 参数
  - `GET /api/knowledge/search` → 缺少 q 参数返回 400
  - `POST /api/knowledge/import` → 缺少 filePath 返回 400
  - `POST /api/knowledge/import` → 不存在的文件返回 400 (含"文件不存在")
  - `POST /api/knowledge/upload` → 缺少文件返回 400
- 🆕 **ModelRegistry 集成 (4 用例)** (IT-09):
  - `GET /api/providers/catalog/status` → 返回 200 + ready/version/generatedAt/providers 结构验证
  - `GET /api/providers/catalog/status` → ready=true 时 version/generatedAt/providers 有效性验证
  - `POST /api/providers/catalog/refresh` → 网络不可达降级处理，返回 200 success + null version/generatedAt
  - `POST /api/providers/catalog/refresh` → 刷新后再次查询 catalog/status 确认 ready 字段
- 🆕 **插件市场 API (18 用例)** (IT-11):
  - `GET /api/plugins/market` → 市场插件列表 + 字段完整性 + ?refresh=true 强制刷新
  - `GET /api/plugins/market/:id` → 插件详情 + README HTML + 不存在不应崩溃
  - `POST /api/plugins/install` → 缺少 pluginId 返回 400 + 正常安装返回 jobId
  - `GET /api/plugins/install/:jobId` → 不存在的 jobId 返回 404 + 有效 jobId 返回进度
  - `POST /api/plugins/uninstall/:id` → 未安装也返回 200 success
  - `POST /api/plugins/update-check` → updates 数组 + id/latestVersion/hasUpdate 字段
  - `POST /api/plugins/safe-mode` → 启用/禁用/空值处理
  - 已有端点兼容性: `GET /api/plugins` 继续正常返回
  - 错误处理: 无效 JSON body 不导致 500

> 🐛 **修复记录**: 知识点 API 原返回 404，根因为 ① 旧编译产物 `src/index.js` 被 vitest 优先加载 ② 路由顺序 `:id` 在 `stats/summary` 前导致 `stats` 被 :id 捕获。修复: 删除旧 artifact + 调整路由顺序。

#### 🛡️ CORS 地址绑定机制说明

> **以下为 CORS 安全策略的架构说明，非测试用例，属于安全设计文档。**

**背景**: Desktop 版本通过 Electron `loadFile()` 加载前端页面，渲染进程发起 `fetch()` 时 `Origin` 为 `null`（`file://` 协议）。早期 CORS 配置为完全开放（`origin: '*'`），在代码评审（2026-06-24 batch-A）中被标记为 🔴 阻塞级安全漏洞，要求改为受控白名单。

**CORS 白名单策略** (`packages/server/src/index.ts` L425-453):

```
┌──────────────────────────────────────────────────────┐
│ CORS 中间件 — 两级策略                                │
├──────────────────────────────────────────────────────┤
│ 精确模式（CORS_ORIGIN 环境变量已设置）:                │
│   origin === CORS_ORIGIN → 允许                       │
│   否则 → 拒绝 "Not allowed by CORS"                   │
│                                                      │
│ 默认模式（未设 CORS_ORIGIN）:                         │
│   origin 为 null → 允许（Electron Desktop file://）   │
│   origin 以 http://127.0.0.1 开头 → 允许（本地服务）  │
│   origin 以 http://localhost 开头 → 允许（开发/Vite）  │
│   其他 → 拒绝                                         │
└──────────────────────────────────────────────────────┘
```

**允许的来源及场景**:

| 来源 (Origin)           | 场景             | 说明                        |
| ----------------------- | ---------------- | --------------------------- |
| `null`                  | Electron Desktop | `loadFile()` → file:// 协议 |
| `http://127.0.0.1:3456` | Desktop 自建服务 | Electron 主进程 Express     |
| `http://127.0.0.1:5173` | Vite 开发服务器  | Web 版本开发                |
| `http://localhost:5173` | Vite 开发服务器  | 备选域名                    |
| `http://localhost:3456` | Desktop 备选     | localhost 别名              |

**Desktop 版本地址硬编码说明**:

Desktop 版的 API 地址 `http://127.0.0.1:3456` 和 WebSocket 地址 `ws://127.0.0.1:3456/ws` 在 `packages/desktop/src/renderer/main.tsx` 中通过 `ConfigProvider` 注入前端，**不可变**。Electron 主进程 (`packages/desktop/src/main.ts`) 启动时在本地监听 `127.0.0.1:3456`，不对外暴露。该地址选择 `127.0.0.1` 而非 `0.0.0.0` 也是出于安全考量——仅限本机访问。

## 六、MCP测试在Windows下的特殊处理

MCP客户端测试在Windows环境下存在 `cmd.exe` 对 `node -e` 内联脚本的转义问题，已在所有MCP测试中采用以下方案：

1. **使用临时`.js`脚本文件**：通过 `createMcpScript()` 辅助函数将脚本写入临时文件，避免 `cmd.exe` 转义
2. **MCPClient进程退出清理**：在 `process.on('close')` 事件中 reject 所有 pending requests
3. **脚本默认响应**：所有MCP模拟脚本添加 `else` 分支处理 `shutdown` 等未预期请求
4. **测试后清理**：通过 `cleanMcpDir()` 清理临时脚本目录

### 10.2 状态机定义

```
idle ──────→ checking ──────→ available ──────→ downloading ──────→ downloaded ──────→ installing
  ↑              │                 │                  │                   │
  │              │                 │                  │                   │
  └── error ←────┴── error ────────┴──── error ───────┘                   │
       │                                                                   │
       └──→ (重试：error → checking → ...)                                 │
                                                                           │
                                                              quitAndInstall → 应用退出
```

| 状态          | 含义       | 触发条件          | 允许操作           |
| ------------- | ---------- | ----------------- | ------------------ |
| `idle`        | 无更新活动 | 启动初始 / 无更新 | 检查更新           |
| `checking`    | 正在检查   | 手动/自动触发检查 | 等待               |
| `available`   | 发现新版本 | 检查到可用更新    | 等待下载开始       |
| `downloading` | 下载中     | 下载开始          | 查看进度           |
| `downloaded`  | 下载完成   | 下载成功          | **立即重启安装**   |
| `error`       | 出错       | 检查/下载失败     | **重试**           |
| `installing`  | 安装中     | 用户确认安装      | 无（应用即将退出） |

### 10.7 关键设计决策

1. **避免 checkForUpdatesAndNotify() 重复调用冲突**: 自动检查使用 `checkForUpdatesAndNotify()`，手动检查使用 `checkForUpdates()` + 显式 `downloadUpdate()`，两者通过 `lastUpdateStatus` 互斥。

2. **"立即重启安装"按钮**: 系统弹窗点"稍后"后，用户仍可在设置页找到安装按钮。按钮调用 `quitAndInstall()` 后立即退出应用。

3. **错误状态保留版本信息**: 下载失败时保留已获取的版本号，方便前端展示"正在下载 vX.X.X 时失败"。

4. **"仅检查"模式**: `checkForUpdates()` 返回 null 表示无更新；返回结果即表示有新版本。模拟测试中 `check-only` 场景跳过了下载阶段。

5. **前端辅助函数置于组件外**: `getStatusCardStyle/getStatusIcon/getStatusTitle/getStatusDescription/getStatusAction` 均为纯函数，避免每次渲染重新创建。
