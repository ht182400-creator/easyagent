## 2026-07-02 21:55 — 方案 D 三连 Bug 修复：manifest 缺失 / 路径硬编码 / base 404

### 现象
用户截图：插件市场页面右上角反复弹出"安装失败：插件缺少 manifest.json 文件"提示横幅。手动 `GET C:\Users\ht182\.easyagent\plugins\obsidian-doc-viewer\` 确认目录为空。

### 根因（三层叠加）

#### Bug 1: `extractZip` 把平铺 plugin.zip 误判为 GitHub zipball
**位置**：`PluginMarketService.ts:551-553`  
**问题**：原代码用 `dirs.find(d => d.isDirectory() && ...)` 找第一个子目录，作为 GitHub zipball 的包装层。但我们 CI 打的 `plugin.zip` 是**平铺结构**（`manifest.json` + `plugin.js` + `dist/` 三个顶层项），`dist/` 是第一个也是唯一子目录，被误判为包装层。  
**结果**：只把 `dist/` 内容拷到 pluginDir，`manifest.json`/`plugin.js` 被遗弃在 tmpDir，然后 line 464 `existsSync(manifestPath)` 失败 → 抛"缺少 manifest.json" → line 494 catch 清理 pluginDir → 用户看到的目录是空的。

#### Bug 2: `docViewerDistPath` 硬编码 `obsidian-doc-viewer`
**位置**：`server/src/index.ts:3211`  
**问题**：插件目录名实际是 GitHub 仓库名 `easyagent-plugin-obsidian-doc-viewer`（`pluginIdToName` 取 `id.split('/')[1]`），但 Server 端硬编码 `obsidian-plugin-obsidian-doc-viewer`（即 `easyagent-plugin-obsidian-doc-viewer` 的简写），两个不一致导致 Server 永远找不到 dist。

#### Bug 3: vite `base: '/'` 导致资源 404
**位置**：插件仓库 `vite.config.ts:7`  
**问题**：HTML 里引用 `/assets/index-*.js` 绝对路径，浏览器在 `http://localhost:3456/doc-viewer/` 下解析为 `http://localhost:3456/assets/...`，但 Server 的 `app.use('/doc-viewer', ...)` 只在 `/doc-viewer` 前缀下托管 → 404。

### 修复

| 文件 | 改动 |
|------|------|
| `PluginMarketService.ts` | extractZip 新增 `isFlatLayout` 标记：检查顶层是否有 `manifest.json`/`plugin.js`；asset 查找名 `dist.zip` → `plugin.zip`；CI 打包改为 `manifest.json` + `plugin.js` + `dist/` 三件套 |
| `server/src/index.ts` | docViewerDistPath 改为动态 `readdirSync(pluginsBaseDir)` 扫描，跳过 `.cache`/`.` 前缀目录 |
| `vite.config.ts` | `base: '/'` → `base: '/doc-viewer/'`；server.open 指向 `/doc-viewer/`；README 更新访问地址 |
| `.github/workflows/build.yml` | `zip -r dist.zip dist/` → `zip -r plugin.zip manifest.json plugin.js dist/` |

### 验证（端到端）

### 关键教训
1. **`extractZip` 必须区分 zipball 模式 vs 平铺模式**：不要仅凭"第一个子目录"判断，应该用标记文件（manifest.json/package.json）区分。
2. **插件目录名是仓库名，不是产品名**：`pluginIdToName` 取 `owner/repo` 的 repo 部分，前缀 `easyagent-plugin-` 不可丢弃。任何硬编码插件目录名的代码都会因为仓库命名变化而失效。
3. **Vite base 必须与部署路径一致**：`/` 适合独立部署，`/doc-viewer/` 适合子路径托管。不要用 `/` 然后期望 Express.static 自动重写。

## 2026-07-02 13:15 — tool_call/tool_result toolCallId 不匹配 → 工具结果永远丢

### 现象
`open-doc-viewer` 工具执行成功（contentLen=18/15），但前端 UI 看不到任何提示/卡片更新，工具卡片一直停留在 "running" 状态。

### 根因
**`tool_call` 和 `tool_result` 两个事件使用了不同的 toolCallId**：

```typescript
// Agent.ts: 549-567 — 原本两个事件都未传 toolCallId
// agentAdapter.ts: 79 — tool_call 事件伪造 ID: `lg_tool_${now}`
// agentAdapter.ts: 96 — tool_result 事件伪造 ID: `lg_tool_result_${Date.now()}` (全新时间戳)
```

由于两个 ID **完全不同**，前端 chatStore `tool_result` handler 第 431 行的 `tc.toolCallId === data.toolCallId` 比较永远不匹配，工具卡片永远停留在 running 状态。

### 修复
| 文件 | 改动 |
|------|------|
| `Agent.ts` | `on_tool_start`/`on_tool_end` 映射时提取 `event.run_id`（LangChain 标准字段）作为 `toolCallId` 透传 |
| `agentAdapter.ts` | `tool_call`/`tool_result` 事件都从 `data.toolCallId` 取 ID（不再用 `Date.now()` 伪造） |

### 验证
- `pnpm --filter langgraph build` ✅
- `pnpm --filter server build` ✅

### 影响范围
修复后所有插件工具（`open-doc-viewer`/`sandbox_exec`/`git_*` 等）的执行结果都能正确回流到前端工具卡片，状态从 running → done/error。

## 2026-07-02 08:11-08:27 — Ollama 未启动导致 "fetch failed"

### 现象
用户发送消息后 AI 回复"错误: fetch failed"，Token=0，耗时 538ms。

### 排查过程
1. 后端 `localhost:3456` 正常 ✅
2. `request.ts:125` 只匹配 `'Failed to fetch'`（旧版），Chrome 100+/Node 22+ 改用 `'fetch failed'`，导致**重试逻辑不生效**

### 修复
- 错误匹配兼容两种字符串：`'Failed to fetch' \|\| 'fetch failed'`
- 方案：启动 `ollama serve`，或切换到有 API Key 的云端模型

### 关联
- MEMORY 陷阱 #45（`fetch failed` vs `Failed to fetch` 字符串差异）、#46（Ollama 未启动友好提示）

