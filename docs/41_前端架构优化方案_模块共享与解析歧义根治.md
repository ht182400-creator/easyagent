## 一、问题背景：一个小页面为何引发全站崩溃

### 1.1 触发经过

1. 新增 `DocsGuide.tsx` 页面 → 白屏，`#root` 完全为空
2. 排查确认非双重挂载、非 Vite HMR 缓存
3. **关键发现**：`/src/main.tsx` 返回的是 `frontend/src/main.tsx`（仅导出 mountApp，从未调用），而非 `web/src/main.tsx`（会调用 mountApp 渲染页面）

### 1.2 根因：同名文件解析歧义

```
web/tsconfig.json:
  "include": ["src", "../frontend/src"]     ← 将两个目录视为"同一个编译单元"
  "paths": { "@/*": ["../frontend/src/*"] } ← @/ 映射到 frontend/src

web/vite.config.ts:
  alias: { '@': '../frontend/src' }        ← Vite 层面也做同样映射

HTML:
  <script type="module" src="/src/main.tsx"> ← 绝对路径引用入口
```

当两个目录同时存在同名文件时，Vite 面临二选一：

| 候选文件                | 行为                                | 结果          |
| ----------------------- | ----------------------------------- | ------------- |
| `web/src/main.tsx`      | 导入 mountApp → **调用 mountApp()** | ✅ 正常渲染   |
| `frontend/src/main.tsx` | 仅 **export mountApp**，不调用      | ❌ #root 为空 |

**本质**：不是新增页面触发了新 Bug，而是架构中潜伏的"同名文件解析歧义"陷阱被触发。

### 2.2 引用方式对比

| 维度                 | Web                                     | Desktop                                 |
| -------------------- | --------------------------------------- | --------------------------------------- |
| **tsconfig include** | `["src", "../frontend/src"]` ❌         | `["./**/*.ts", "./**/*.tsx"]` ✅        |
| **Vite alias `@/`**  | → `../frontend/src/`                    | → `../frontend/src/`                    |
| **入口文件**         | `web/src/main.tsx`                      | `desktop/src/renderer/main.tsx`         |
| **导入语句**         | `import { mountApp } from '@/mountApp'` | `import { mountApp } from '@/mountApp'` |
| **路由模式**         | `BrowserRouter`                         | `HashRouter`                            |
| **配置注入**         | `{ isDesktop: false }`                  | `{ isDesktop: true }`                   |
| **平台特有代码**     | 无                                      | IPC 桥接 (`setupIPCBridge`)             |

### 2.3 关键发现

Desktop 的 tsconfig **没有** `include: ["../frontend/src"]`，但 `@/` alias 仍然穿透到 frontend/src。这意味着：

- **Web** 的问题更严重（include + alias 双重穿透）
- **Desktop** 依赖 alias 工作，虽然当前可用，但同名文件风险同样存在
- 两者都使用 `@/` 别名作为 `frontend/src` 的快捷方式，语义上是"借道"

## 三、结构缺陷诊断 —— 3 个层面

### 🔴 缺陷 1：模块边界被 alias+include 破坏

```
正确的依赖关系：   web → @easyagent/frontend（包依赖）
当前的实际关系：   web ⊇ frontend/src（包含关系 + 别名穿透）
```

**Web 的问题（严重）**：

```json
// packages/web/tsconfig.json
{
  "include": ["src", "../frontend/src"]           // ← 两个目录扁平化
  "paths": { "@/*": ["../frontend/src/*"] }       // ← @/ 指向外部
}
```

TypeScript 将两个目录视为**同一个项目**的源文件，任何同名文件都会产生解析歧义。

**Desktop 的问题（潜在）**：

```json
// packages/desktop/src/renderer/tsconfig.json
{
  "include": ["./**/*.ts", "./**/*.tsx"], // ← 只包含自己，无跨包引用
  "paths": { "@/*": ["../../../frontend/src/*"] } // ← 但 paths 仍然穿透
}
```

Desktop 虽然没有 include 问题，但 `@/` 的别名全部指向外部包。这带来两个隐患：

1. 如果 Desktop 自身的 renderer 目录也放了同名文件，同样产生歧义
2. IDE 的跳转、重构、自动导入都会穿透到 frontend 源码，破坏了模块封装

### 🔴 缺陷 2：两套路径系统分工混乱

| 导入方式                                | 语义                       | 实际解析                                    | 问题                                |
| --------------------------------------- | -------------------------- | ------------------------------------------- | ----------------------------------- |
| `import { mountApp } from '@/mountApp'` | "从我自己的 mountApp 导入" | 实际从 `frontend/src/mountApp.tsx`          | 语义错误，`@/` 应该是"我自己的 src" |
| `@/components/Layout`                   | "我自己的 Layout 组件"     | 实际从 `frontend/src/components/Layout.tsx` | 同上                                |
| `@/stores/appStore`                     | "我自己的 store"           | 实际从 `frontend/src/stores/appStore.ts`    | 同上                                |

**`@/` 被滥用了**：它本应表示"本包的 src 目录"，但实际被用作"frontend 共享包的快捷入口"。正确的做法是：

```ts
// ✅ 正确：包名导入（语义清晰）
import { mountApp } from '@easyagent/frontend';

// ✅ @/ 只用于本包内部引用
import { MyLocalComponent } from '@/components/MyLocalComponent';
```

### 🟡 缺陷 3：package exports 已失效

```json
// packages/frontend/package.json
{
  "main": "./src/main.tsx", // ← 文件已重命名为 mountApp.tsx
  "exports": {
    ".": "./src/main.tsx", // ← 同上，入口不存在
    "./App": "./src/App.tsx",
    "./components/*": "./src/components/*",
    "./pages/*": "./src/pages/*",
    "./stores/*": "./src/stores/*",
    "./config": "./src/config.tsx",
    "./api": "./src/api.ts",
    "./styles/*": "./src/styles/*"
  }
}
```

**影响**：任何通过包名 `import ... from '@easyagent/frontend'` 的代码都将解析失败。当前 web/desktop 使用 alias 而非包名导入，暂时规避了这个问题，但这是一个隐藏炸弹。

## 四、Desktop 构建链分析

### 4.1 Desktop 的三阶段构建

```
阶段 1: tsup                   阶段 2: vite build          阶段 3: electron-builder
┌─────────────────────┐      ┌─────────────────────┐      ┌─────────────────────┐
│ main.ts     ──→     │      │ renderer/main.tsx   │      │ dist/ + assets/ +   │
│  ├─ @easyagent/core  │      │  ──→ dist/renderer  │      │ node_modules/       │
│  └─ @easyagent/server│      │                     │      │  ──→ release/       │
│                      │      │ @/ → frontend/src   │      │  *.exe / *.dmg /    │
│ preload.ts  ──→      │      │ (Vite 打包进        │      │  *.AppImage         │
│  dist/preload.cjs    │      │  renderer bundle)   │      │                     │
└─────────────────────┘      └─────────────────────┘      └─────────────────────┘
     tsup (ESM)                  vite build                  electron-builder
```

**关键决策**：

- `@easyagent/core` + `@easyagent/server` 被 tsup 的 `noExternal` 内联进 `main.js`（避免 pnpm symlink 在 asar 中失效）
- `@easyagent/frontend` 不经过 tsup，由 Vite 打包进 renderer bundle
- frontend 作为 **源码**（而非编译产物）被 Vite 处理

### 4.2 Desktop 与 Web 的差异对比

| 维度          | Web                       | Desktop                              |
| ------------- | ------------------------- | ------------------------------------ |
| **运行环境**  | HTTP 服务器               | Electron (main + renderer + preload) |
| **路由**      | BrowserRouter（美观 URL） | HashRouter（兼容 `file://`）         |
| **API**       | 同域代理                  | 显式 `http://127.0.0.1:3456`         |
| **开发端口**  | 5173                      | 5183                                 |
| **平台代码**  | 无                        | IPC 桥接 (`setupIPCBridge`)          |
| **Vite base** | `/`                       | `./`（相对路径，file:// 兼容）       |
| **构建工具**  | 仅 Vite                   | Vite + tsup + electron-builder       |

### 4.3 影响评估

**当前架构对 Desktop 的影响**：

| 问题           | 严重程度  | 说明                                                 |
| -------------- | --------- | ---------------------------------------------------- |
| 同名文件冲突   | 🟡 潜在   | Desktop 未 include frontend/src，但 alias 仍指向外部 |
| `@/` 语义混乱  | 🟡 中等   | Desktop 自身代码几乎没有，`@/` 被完全用于穿透        |
| exports 失效   | 🟢 无影响 | Desktop 不通过包名导入 frontend                      |
| 构建链不受影响 | 🟢 正常   | Desktop 构建链独立工作                               |

**结论**：Desktop 当前运行正常，但改造后架构更清晰、更安全。改造成本低（约 3 行改动）。

## 五、目标架构：平台适配器模式

### 5.1 核心理念

```
┌─────────────────────────────────────────────────────┐
│               @easyagent/frontend                    │
│               (共享 UI 层 — 零平台依赖)              │
│                                                     │
│  mountApp(config, useHashRouter, containerId)        │
│  App / Layout / Components / Pages / Stores / Config │
│  样式、路由配置、状态管理                            │
│                                                     │
│  出口：mountApp.tsx + 子路径导出                      │
│  依赖：react, react-dom, react-router-dom, zustand   │
└──────────┬──────────────┬──────────────┬────────────┘
           │              │              │
    包名导入          包名导入        (未来) 包名导入
  @easyagent/      @easyagent/     @easyagent/
  frontend         frontend        frontend
           │              │              │
    ┌──────▼──────┐ ┌────▼─────┐ ┌─────▼──────┐
    │ web (适配器) │ │ desktop  │ │ mobile-web │
    │             │ │ (适配器) │ │ (适配器)   │
    │             │ │          │ │            │
    │ 平台配置：   │ │ 平台配置 │ │ 平台配置：  │
    │ isDesktop:  │ │ isDesktop│ │ isPWA: true│
    │   false     │ │   : true │ │            │
    │             │ │          │ │            │
    │ Browser     │ │ Hash     │ │ Hash       │
    │ Router      │ │ Router   │ │ Router     │
    │             │ │          │ │            │
    │ 平台代码：   │ │ 平台代码 │ │ 平台代码：  │
    │ 无          │ │ IPC 桥接 │ │ Service    │
    │             │ │ Auto     │ │ Worker     │
    │             │ │ Updater  │ │            │
    └─────────────┘ └──────────┘ └────────────┘
```

### 5.2 设计原则

**原则 1：共享层零平台依赖**

- `@easyagent/frontend` 不引用任何平台 API（Electron、window 全局等）
- 平台特有行为通过 `ConfigProvider` 的 config 注入
- 平台特有组件通过 `children` / `slots` 模式注入（预留）

**原则 2：适配器是薄层**

- 每个适配器包只有 `main.tsx` + 平台配置 + 可选的平台初始化代码
- 平均代码量：30-50 行
- 无业务逻辑，只做装配

**原则 3：统一包名导入**

```ts
// ✅ 所有平台统一导入方式
import { mountApp } from '@easyagent/frontend';

// ❌ 不再使用 @/ 穿透到外部包
// @/ 回归本意：指向本包 src 目录
```

**原则 4：独立编译边界**

- 每个包的 tsconfig 只 `include` 自己的 `src/`
- 类型信息通过 package.json `exports` + `types` 字段声明
- 不使用跨包 `include` 或 `paths` 穿透

### 5.3 目录结构

```

│   ├── src/
```

### 5.4 关键配置文件

#### `packages/frontend/package.json` — 共享包导出

```json
{
  "name": "@easyagent/frontend",
  "version": "0.4.1",
  "type": "module",
  "main": "./src/mountApp.tsx",
  "exports": {
    ".": "./src/mountApp.tsx",
    "./App": "./src/App.tsx",
    "./components/*": "./src/components/*",
    "./pages/*": "./src/pages/*",
    "./stores/*": "./src/stores/*",
    "./config": "./src/config.tsx",
    "./api": "./src/api.ts",
    "./styles/*": "./src/styles/*"
  }
}
```

> **关键**：`"main"` 和 `"exports[\".\"]"` 必须指向 `mountApp.tsx`（当前指向已删除的 `main.tsx`）。

#### `packages/frontend/tsconfig.json` — 独立编译边界

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "esModuleInterop": true,
    "declaration": true,
    "declarationMap": true,
    "outDir": "dist",
    "rootDir": "src",
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "include": ["src"]
}
```

> **关键改变**：`@/` 指向 `./src/*`（自己的源码），不再被外部穿透。

#### `packages/web/vite.config.ts` — Web 适配器

```ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { resolve } from 'path';

export default defineConfig({
  plugins: [react()],
  root: '.',
  resolve: {
    alias: {
      // 包名导入 → 指向 frontend 源码（Vite 实时编译）
      '@easyagent/frontend': resolve(__dirname, '../frontend/src'),
      // @/ 回归本意 → 指向 web 自己的 src
      '@': resolve(__dirname, 'src'),
    },
  },
  server: {
    host: true,
    port: 5173,
    proxy: {
      '/api': { target: 'http://localhost:3456', changeOrigin: true },
      '/ws': { target: 'ws://localhost:3456', ws: true },
    },
  },
  build: {
    outDir: 'dist',
    emptyOutDir: true,
  },
});
```

#### `packages/web/tsconfig.json` — Web 适配器

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "esModuleInterop": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],
      "@easyagent/frontend": ["../frontend/src"]
    }
  },
  "include": ["src"]
}
```

> **关键改变**：移除 `"../frontend/src"`，`@/` 改为 `"./src/*"`，frontend 的类型通过 `paths` 中的**完整包名**映射。

#### `packages/web/src/main.tsx` — Web 入口（薄适配器）

```tsx
/**
 * EasyAgent Web - 平台适配器入口
 * 职责：提供 Web 平台配置 → 调用统一前端 mountApp()
 */
import { mountApp } from '@easyagent/frontend';

// Web 平台配置
const webConfig = {
  apiBase: '',
  wsBase: '/ws',
  isDesktop: false,
};

mountApp(webConfig, false); // BrowserRouter
```

> **改变**：`@/mountApp` → `@easyagent/frontend`，语义清晰。

#### `packages/desktop/vite.config.ts` — Desktop 适配器

```ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { resolve } from 'path';

export default defineConfig({
  plugins: [react()],
  root: '.',
  base: './', // file:// 协议兼容（关键！）
  resolve: {
    alias: {
      // 包名导入 → 指向 frontend 源码
      '@easyagent/frontend': resolve(__dirname, '../frontend/src'),
      // @/ 回归本意 → 指向 desktop 自己的 renderer 目录
      '@': resolve(__dirname, 'src/renderer'),
    },
  },
  build: {
    outDir: 'dist/renderer',
    emptyOutDir: true,
    sourcemap: true,
    assetsInlineLimit: 0,
  },
  server: {
    port: 5183,
```

```ts
/**
 * Desktop 平台 IPC 桥接
 * 将 Electron IPC 事件桥接到前端 store
 */
export function setupIPCBridge(): void {
  const api = (window as any).easyAgent;
  if (!api) {
    console.log('[EasyAgent Desktop] 开发模式 - IPC 桥接不可用，使用 HTTP API');
    return;
  }

  console.log('[EasyAgent Desktop] IPC 桥接已初始化');

  // 动态导入避免循环依赖
  import('@easyagent/frontend/stores/appStore').then(({ useAppStore }) => {
    api.onWorkspaceChanged((workspacePath: string) => {
      console.log('[EasyAgent Desktop] 工作区切换:', workspacePath);
      useAppStore.getState().addNotification({
        type: 'info',
        message: `工作区已切换: ${workspacePath}`,
        duration: 3000,
      });
    });

    api.onUpdateStatus((status: any) => {
      if (status.status === 'downloaded') {
        useAppStore.getState().addNotification({
          type: 'success',
          message: '新版本已下载，重启应用以安装更新',
          duration: 0,
        });
      } else if (status.status === 'error') {
        console.warn('[EasyAgent Desktop] 更新检查失败:', status.message);
      }
    });
  });
}
```

> **改变**：从 `@/stores/appStore`（alias 穿透）→ `@easyagent/frontend/stores/appStore`（包名导入，清晰）。

| 原则         | 说明                       | 体现                                          |

### 9.1 各维度评估

| 维度                            | 当前状态                | 平台适配器模式后           | 改善  |

### 9.2 核心结论

1. **架构思路是对的**：统一前端包 + `mountApp()` 注入隔离平台差异 → 业界标准。
2. **实现方式有缺陷**：`@/` 别名穿透 + `include` 跨包引用 → 反模式。
3. **Desktop 是干净的**：Desktop 没有 include 问题，但别名穿透同样存在。改造后更安全。
4. **平台适配器模式**是当前阶段的最优解：
   - 改动最小（~30 行）
   - 风险最低
   - 天然可扩展（新平台 = `packages/<name>/src/main.tsx`）
   - 为后续 Library Mode 迁移提供平滑路径
5. **修复优先级**：
   - P0（今日）：修复 `package.json` exports
   - P1（本周）：实施平台适配器模式
   - P2（按需）：Vite Library Mode
