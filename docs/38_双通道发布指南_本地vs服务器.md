# EasyAgent 双通道发布指南 —— 本地构建 vs 服务器构建

> **目标读者**：EasyAgent 项目维护者、初次参与发布的开发者  
> **更新日期**：2026-06-27  
> **适用版本**：v0.6.6+

---

## 一、概述

EasyAgent 支持 **两种发布方式**，对应两个 `.bat` 脚本：

| 脚本                  | 类型                   | 适用场景                                         |
| --------------------- | ---------------------- | ------------------------------------------------ |
| `release-publish.bat` | **本地构建**           | 完整构建流水线，在开发者本机完成所有步骤         |
| `release-server.bat`  | **服务器构建 (CI/CD)** | 仅推送版本标签，由 GitHub Actions 自动构建和发布 |

> **核心区别**: 谁来完成耗时最长的"构建 EXE"这一步？是开发者电脑还是 GitHub 服务器。

---

## 二、两种方式全景对比

```
┌──────────────────────────────────────────────────────────────────┐
│                    release-publish.bat (本地)                      │
│                                                                   │
│  开发者电脑                                                        │
│  ┌─────────┐    ┌─────────┐    ┌───────────┐    ┌───────────┐   │
│  │ 版本升级  │ → │ DTS检查  │ → │ 构建EXE    │ → │ 上传Release│   │
│  │ ~30s     │    │ ~20s     │    │ ~3-5min    │    │ ~1min     │   │
│  └─────────┘    └─────────┘    └───────────┘    └───────────┘   │
│                    需要 MSVC + electron-builder                    │
│                    产物在本地，立即可验证                            │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│                    release-server.bat (服务器)                     │
│                                                                   │
│  开发者电脑                        GitHub 服务器                    │
│  ┌─────────┐                      ┌───────────────────────────┐  │
│  │ 版本升级  │  ──git push tag──→   │ GitHub Actions 自动执行:   │  │
│  │ + push  │                      │ 构建Desktop EXE (Windows)  │  │
│  │ ~30s    │                      │ 构建Web Dashboard (Ubuntu) │  │
│  └─────────┘                      │ 合并产物 → 创建Release      │  │
│     无需 MSVC/electron-builder    │ ~5-10min                    │  │
│     无需本地编译                    └───────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 详细对比表

| 维度                | `release-publish.bat` (本地)                | `release-server.bat` (服务器)          |
| ------------------- | ------------------------------------------- | -------------------------------------- |
| **本地耗时**        | ~5-8 分钟 (含构建)                          | ~30 秒 (仅版本 + push)                 |
| **总耗时 (含构建)** | ~5-8 分钟                                   | ~5-10 分钟 (后台自动)                  |
| **本地环境要求**    | Node + Git + pnpm + MSVC + electron-builder | Node + Git                             |
| **磁盘占用 (本地)** | ~2GB (构建产物+中间文件)                    | ~0 (仅源码更改)                        |
| **CPU/内存占用**    | 高 (编译原生模块 + 打包)                    | 低 (仅文本操作)                        |
| **网络要求**        | 上传 EXE (~105MB) 需稳定网络                | 仅推送 Git (很小)                      |
| **产物立即可验证**  | ✅ 可以 (EXE 在本地)                        | ❌ 需要等 CI/CD 完成                   |
| **需手动上传**      | 可选 (支持 gh CLI / token / 手动)           | ❌ 自动完成                            |
| **失败恢复**        | 手动重试，灵活性高                          | 重跑 CI Job 或重新推送                 |
| **多人协作**        | ❌ 各自环境可能不同                         | ✅ CI 环境一致                         |
| **Pipeline 同步**   | ✅ 自动执行 (Step 6)                        | ❌ 需手动执行 `pipeline-auto-sync.ps1` |

## 三、`release-server.bat` 详解 （服务器构建）

### 3.1 设计理念

**"你只管版本升级和推送，剩下的交给 GitHub"**

这是最轻量的发布方式——开发者只需要在本地执行版本号升级和 Git 推送，其余构建、打包、上传全部由 GitHub Actions 在云端完成。

### 3.2 执行流程

```
release-server.bat
  │
  ├── Step 0: 环境检查
  │    ├─ Git 是否安装
  │    ├─ Node.js 是否安装
  │    ├─ 远程仓库是否可连接
  │    └─ CI/CD 配置是否就绪 (.github/workflows/release.yml)
  │
  ├── Step 1-2: 版本号选择
  │    ├─ 读取当前版本 (version.json)
  │    ├─ 选择 patch/minor/major/custom
  │    └─ 确认目标版本
  │
  ├── Step 3: 检查 CHANGELOG
  │    └─ 验证当前版本条目存在 (不存在则自动生成)
  │
  ├── Step 4: 执行 release.mjs
  │    ├─ 更新 version.json → 新版本号
  │    ├─ 同步版本到 7 个 package.json（含 frontend + vscode）
  │    ├─ 更新 CHANGELOG.md (从 git log 自动生成)
  │    ├─ git commit -m "release: v{version}"
  │    ├─ git tag -a v{version}
  │    └─ git push origin main --follow-tags
  │
  ├── Step 5: 验证远程标签
  │    └─ git ls-remote 确认标签已推送
  │
  └── Step 6: 提示 CI/CD 监控链接
       └─ 可选自动打开浏览器查看 CI 进度
```

### 3.3 参数说明

| 参数     | 说明                                | 示例                        |
| -------- | ----------------------------------- | --------------------------- |
| 无参数   | 交互式模式，逐步确认                | `release-server.bat`        |
| `--auto` | 自动模式，默认 patch 升级，减少交互 | `release-server.bat --auto` |
| `--help` | 显示帮助信息                        | `release-server.bat --help` |

### 3.4 推送后的自动化流程

推送 `v{version}` 标签后，`.github/workflows/release.yml` 自动接管：

```
push v0.6.6 标签
     │
     ┌── (并行)
     │
     ├─ build-desktop (windows-2022, ~5min)
     │    core tsup → server tsup → desktop tsup (CI 独立编译)
     │    → preload CJS → vite → verify-build
     │    → sqlite3 双ABI编译+切换
     │    → electron-builder --win --x64
     │    → 恢复 sqlite3 → 上传 artifact
     │
     ├─ build-web (ubuntu-latest, ~2min)
     │    core tsup → server tsup → vite build (CI 独立编译)
     │    → tar → 上传 artifact
     │
     └─ release (ubuntu-latest)
          下载所有 artifact → 验证 → 生成Release Notes
          → 创建 GitHub Release → 上传 4 个 Assets
```

## 四、`release-publish.bat` 详解 （本地构建）

### 4.1 设计理念

**"一站式发布，所有步骤在本地完成"**

适合需要立即验证构建产物或需要完全掌控发布流程的开发者。

### 4.2 执行流程

```
release-publish.bat
  │
  ├── Step 1: 版本升级类型选择 (patch/minor/major/custom/skip)
  ├── Step 2: 预检 (Git/Remote/Node/工作区状态)
  ├── Step 3: 版本升级 (release.mjs → version.json + package.json + CHANGELOG + push)
  ├── Step 3.5: DTS 类型检查 (npx tsup --dts) — 质量门禁
  ├── Step 4: 构建 Desktop EXE (build.bat --release) — 约3-5分钟
  ├── Step 5: 上传到 GitHub Release
  │    ├─ [1] gh CLI (推荐)
  │    ├─ [2] Token + _release.mjs (自动)
  │    └─ [3] 手动上传 (不推荐，文件超限)
  └── Step 6: Pipeline 数据自动同步
       └─ CI vitest 结果下载 + pipeline 数据更新 + 服务器重启
```

### 4.3 参数说明

| 参数     | 说明                           | 示例                         |
| -------- | ------------------------------ | ---------------------------- |
| 无参数   | 交互式模式                     | `release-publish.bat`        |
| `--auto` | 自动模式，patch 升级，减少确认 | `release-publish.bat --auto` |
| `--help` | 显示帮助信息                   | `release-publish.bat --help` |

### 4.4 Step 5 三种上传方式对比

| 方式                   | 速度  | 可靠性    | 前提条件                                      |
| ---------------------- | ----- | --------- | --------------------------------------------- |
| `gh CLI`               | ⚡ 快 | ⭐⭐⭐    | `winget install GitHub.cli` + `gh auth login` |
| `_release.mjs` (Token) | ⚡ 快 | ⭐⭐⭐    | `scripts/.release_token` 文件 (推荐)          |
| 手动上传               | 🐢 慢 | ⚠️ 不可用 | 浏览器限制 25MB，EXE 超限！                   |

## 五、如何选择？

### 5.1 决策流程图

```
需要发版
  │
  ├── 电脑配置足够 (MSVC + 16GB RAM + 充足磁盘)
  │   ├── 需要立即验证 EXE → release-publish.bat
  │   └── 可以等待几分钟 → release-server.bat
  │
  ├── 电脑配置不足 (轻量级/无 MSVC)
  │   └── → release-server.bat (唯一选择)
  │
  ├── 网络不稳定 (上传大文件容易失败)
  │   └── → release-server.bat (GitHub 服务器上传)
  │
  └── 需要完全掌控流程、即时反馈
      └── → release-publish.bat
```

### 5.2 场景推荐

| 场景                   | 推荐方式              | 理由                                   |
| ---------------------- | --------------------- | -------------------------------------- |
| **日常 Bug 修复发版**  | `release-server.bat`  | 轻量快速，无需本地编译                 |
| **首次参与发版的新人** | `release-server.bat`  | 环境要求最低，不会因 MSVC 配置问题卡住 |
| **修改了构建链代码**   | `release-publish.bat` | 需要本地验证构建是否成功再推送         |
| **紧急热修复**         | `release-publish.bat` | 本地直接构建上传，无需等 CI            |
| **网络不稳定**         | `release-server.bat`  | 本地只需推送很小的 Git 数据            |
| **发布 major 版本**    | `release-publish.bat` | 重大更新建议本地验证后发布             |
| **CI 端不可用**        | `release-publish.bat` | GitHub Actions 故障时的备选方案        |
| **CI 端正常**          | `release-server.bat`  | ⭐ **默认推荐**                        |

### 5.3 给初学者的建议

**首选 `release-server.bat` (服务器构建)**，原因：

1. **零环境依赖**：不需要安装 MSVC、electron-builder、better-sqlite3 编译工具链
2. **不会卡在环境问题**：MSVC 配置、Windows SDK、NSIS 打包等问题全部由 CI 环境解决
3. **操作最简单**：选择版本号 → 确认 → 等待 → 完成，全程约 30 秒操作
4. **产物一致**：CI 环境固定，不会出现"我电脑上可以但你电脑上不行"的问题

**何时需要本地构建**：

- 改了 `packages/desktop/` 下的构建相关代码，想本地验证
- GitHub Actions 宕机/限流
- 需要立即给用户发更新包（不能等 5-10 分钟）

### 5.4 给项目维护者的建议

| 维护者类型     | 建议                                                                                           |
| -------------- | ---------------------------------------------------------------------------------------------- |
| **主要维护者** | 两者都要熟悉。日常用 `release-server.bat`，重大更新或修改构建链后用 `release-publish.bat` 验证 |
| **偶尔贡献者** | 只用 `release-server.bat`，把构建环境维护成本交给 CI                                           |

## 六、操作手册

### 6.0 发版前置步骤：更新 CHANGELOG.md

在运行任何发布脚本之前，应先更新 `CHANGELOG.md`。本项目的 CHANGELOG 基于 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.0.0/) 规范。

#### 6.0.1 CHANGELOG 格式规范

```
## [x.y.z] - YYYY-MM-DD

### Added          ← 新增的功能
- feat: xxx

### Changed        ← 功能变更/重构
- refactor: xxx

### Fixed          ← Bug 修复
- fix: xxx

### Removed        ← 移除的功能（如有）
```

**示例**（来自实际 `CHANGELOG.md`）：

```markdown
## [0.5.31] - 2026-06-26

### Fixed

- fix: hasUpdate=false 时错误状态未清除导致 UI 误显"更新失败" (ht182400-creator)
- fix: 修复 GitHub Release v0.5.30 缺少 latest.yml 导致 electron-updater 检查 404 (ht182400-creator)
```

#### 6.0.2 更新方式（两种）

**方式一：手动编写（推荐 ⭐⭐⭐）**

在 `CHANGELOG.md` 第一个 `## [` 条目上方，按规范格式手写当前版本的变更内容：

```bash
# 1. 用编辑器打开 CHANGELOG.md
# 2. 在第一个 ## [ 上方插入新条目，例如:
#    ## [0.6.2] - 2026-06-27
#    ...你的变更内容...

# 3. 保存文件
```

> **为什么推荐手动编写**：
>
> - 可以写清晰、面向用户的中文描述
> - 可以筛选掉内部 commit（如 `chore: 临时保存`）
> - 可以合并相关改动、突出重要变更
> - 自动生成的条目质量取决于 commit message 是否规范

**方式二：自动生成（省事但质量一般）**

不手动编辑 `CHANGELOG.md`，让 `release.mjs` 自动从 Git 提交记录生成。脚本逻辑：

1. 找到上一个 tag → 收集此后的所有 commit message
2. 自动分类：含 `feat`/`add` → `Added`，含 `fix`/`bug` → `Fixed`，含 `change`/`refactor` → `Changed`
3. 插入到 `CHANGELOG.md` 顶部

```bash
# 什么都不用做，release 脚本会自动生成
# → 自动生成效果示例:
#   ## [0.6.2] - 2026-06-27
#   ### Added
#   - feat: 新增双通道发布功能 (ht182400-creator)
#   ### Fixed
#   - fix: 修复 xxx 问题 (ht182400-creator)
```

> ⚠️ **注意**：自动生成的分类依赖于 commit message 是否包含 `feat`/`fix`/`refactor` 等关键词。如果 commit message 不规范（如"改了个 bug"），会全部归入 `Changed`。

#### 6.0.3 发布脚本如何处理 CHANGELOG

两个脚本内调用的 `node scripts/release.mjs` 都会**自动更新 CHANGELOG.md**：

```
release.mjs 执行时:
  ├─ 读取 CHANGELOG.md
  ├─ 调用 generateChangelogEntry(version)  ← 从 git log 生成新条目
  ├─ 将新条目插入到文件头部 (第一个 ## [ 之前)
  └─ 写入 CHANGELOG.md
```

**如果你已经手动写好了当前版本的条目**：脚本生成的条目会插入到你的条目**上方**，造成重复。此时需要：

- 方案 A：手动写之前，先用 `git tag` 打个临时标签，确保脚本只收集上一个正式 tag 之后的 commit
- 方案 B（推荐）：手动写好后，**直接跳过脚本的 CHANGELOG 步骤**——但由于 `release.mjs` 强制执行，发布后手动删除重复条目再 commit

> 💡 **最佳实践**：如果手动写 CHANGELOG，请在运行 `release-server.bat` 或 `release-publish.bat` **之前**就写好并 commit，这样脚本拿到的 commit 列表中就不会包含 `changelog` 相关条目，减少重复。

**当前行为说明**（v0.6.6 版本）：

| 你的 CHANGELOG 状态          | release.mjs 会怎么做                                     |
| ---------------------------- | -------------------------------------------------------- |
| 已手动写好 `## [0.6.6]` 条目 | 会在上方再插入一个自动生成的 `## [0.6.6]` 条目→ 可能重复 |
| 仅存在空标题 `## [0.6.6]`    | 会在上方插入自动生成条目（空标题仍在下方）               |
| 完全不存在 `[0.6.6]`         | 自动生成并插入                                           |

```bash
# 如果你手动写了 CHANGELOG，发布后的处理:
# 1. 发布完成后，打开 CHANGELOG.md
# 2. 删除自动生成的重复条目
# 3. git add CHANGELOG.md && git commit -m "docs: 清理重复CHANGELOG" && git push
```

> 📋 **后续优化**：考虑让 `release.mjs` 支持 `--skip-changelog` 参数，当手动写好时跳过自动生成。

### 6.1 使用 `release-server.bat` 发布（推荐日常使用）

```bash
# 方式一：交互式（推荐）
release-server.bat
# → 选择版本类型 → 确认 → 等待推送完成

# 方式二：自动模式（patch 升级）
release-server.bat --auto

# 完成后：
# 1. 等待 5-10 分钟
# 2. 打开 https://github.com/ht182400-creator/easyagent/actions 查看进度
# 3. Release 就绪: https://github.com/ht182400-creator/easyagent/releases
```

### 6.2 使用 `release-publish.bat` 发布（完整本地流程）

```bash
# 前提：
#   1. 已安装 MSVC (Visual Studio 2022 Build Tools)
#   2. scripts/.release_token 文件已配置
#   3. gh CLI 已安装并登录 (可选)

# 交互式模式（推荐首次使用）
release-publish.bat
# → 选择版本 → 确认 → 等待 DTS 检查 → 等待构建 → 等待上传

# 自动模式
release-publish.bat --auto
```

### 6.3 手动回退步骤（出问题时）

```bash
# 如果 server 发布推送了错误标签：
git tag -d v0.6.6           # 删除本地标签
git push origin :v0.6.6     # 删除远程标签
# 然后修正代码，重新运行 release-server.bat

# 如果本地发布构建失败：
# 1. 修复问题
# 2. 直接运行 build.bat --release (跳过版本升级，仅构建)
# 3. 运行 node _release.mjs 0.6.6 --create (上传产物)
```

## 七、注意事项与常见陷阱

### 7.1 通用注意事项

| 序号 | 注意事项                    | 说明                                                                                                                                                |
| ---- | --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1    | ⚠️ **CHANGELOG 重复条目**   | 手动写好 CHANGELOG 后运行脚本，`release.mjs` 仍会从 git log 自动生成新条目插入到顶部，可能造成重复。详见 [6.0 节](#60-发版前置步骤更新-changelogmd) |
| 2    | ⚠️ **确认 worktree 干净**   | 脚本会执行 `git add .`，所有未跟踪文件都会被提交                                                                                                    |
| 3    | ⚠️ **不要同时推送多个标签** | 每次只推送一个版本标签，否则 CI 会多次触发                                                                                                          |
| 4    | ⚠️ **pre-release 版本**     | `release-server.bat` 默认 `prerelease=false`，若需预发布请在 GitHub Actions 页面手动触发并勾选                                                      |

### 7.2 本地构建特有陷阱

| 序号 | 注意事项                  | 说明                                                                       |
| ---- | ------------------------- | -------------------------------------------------------------------------- |
| 1    | ⚠️ **Windows Defender**   | NSIS 打包工具 `makensis.exe` 可能被杀软拦截，需添加排除                    |
| 2    | ⚠️ **sqlite3 ABI 版本**   | `build.bat` 会自动进行双版本编译+切换，但如果手动中断可能导致 ABI 状态异常 |
| 3    | ⚠️ **dist/renderer 缓存** | 构建前必须 `rmdir /s /q dist\renderer`，否则 Vite 复用旧产物               |
| 4    | ⚠️ **上传 EXE 超 25MB**   | 浏览器直接上传 GitHub Release 限制 25MB，必须用 `gh CLI` 或 `_release.mjs` |

### 7.3 服务器构建特有陷阱

| 序号 | 注意事项                     | 说明                                                                                                                  |
| ---- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| 1    | ⚠️ **GitHub Actions 限额**   | 免费账户每月 2000 分钟，Desktop 构建约 5 分钟/次，够用但要注意                                                        |
| 2    | ⚠️ **CI 需要 `release.yml`** | 如果没有 `.github/workflows/release.yml`，推送标签不会触发任何操作                                                    |
| 3    | ⚠️ **等待时间**              | 推送后不是立即可用，需 5-10 分钟。不要频繁推送标签                                                                    |
| 4    | ⚠️ **Pipeline Sync 需手动**  | `release-server.bat` 不包含 Step 6（Pipeline 数据同步），需手动执行 `powershell -File scripts/pipeline-auto-sync.ps1` |

## 八、故障排除

### 8.1 `release-server.bat` 常见问题

| 问题                        | 原因                                           | 解决方法                             |
| --------------------------- | ---------------------------------------------- | ------------------------------------ |
| 推送标签后 CI 没有触发      | `release.yml` 不存在或 `on.push.tags` 配置错误 | 检查 `.github/workflows/release.yml` |
| CI 构建失败                 | better-sqlite3 编译问题或网络超时              | 查看 Actions 日志，重跑失败的 Job    |
| Release 创建了但没有 Assets | `build-desktop` 或 `build-web` Job 失败        | 检查构建日志，修复后重新推送标签     |
| 远程标签推送失败            | 网络问题或权限不足                             | 检查 Git remote 配置和 GitHub 认证   |

### 8.2 `release-publish.bat` 常见问题

| 问题                   | 原因                          | 解决方法                                                     |
| ---------------------- | ----------------------------- | ------------------------------------------------------------ |
| DTS 检查失败           | TypeScript 类型错误           | 修复类型错误，常见: `as ISkill[]` → `as unknown as ISkill[]` |
| electron-builder 失败  | NSIS 打包被拦截 或 文件被占用 | 关闭 EasyAgent 进程，检查 Windows Defender                   |
| \_release.mjs 上传失败 | Token 过期或权限不足          | 重新生成 `scripts/.release_token` (scope: repo)              |
| "Release not found"    | 标签未推送或 Release 未创建   | 添加 `--create` 参数: `node _release.mjs 0.6.2 --create`     |

## 九、后续优化建议

### 9.1 CI/CD 增强（推荐 ⭐⭐⭐⭐）

**当前**: 只有 Windows x64 安装包。

**优化**: 增加更多平台支持。

```yaml
# release.yml 中增加 macOS 和 Linux 构建
build-desktop-mac:
  runs-on: macos-14
  # ... 构建 macOS .dmg 安装包

build-desktop-linux:
  runs-on: ubuntu-latest
  # ... 构建 Linux .AppImage
```

- **难度**: 中
- **收益**: 扩展用户群

### 9.2 自动生成 CHANGELOG 增强（推荐 ⭐⭐⭐）

**当前**: `release.mjs` 的自动 CHANGELOG 生成逻辑较简单，分类不够精准。

**优化**: 使用 Conventional Commits 规范 + 自动分类。

```bash
# 约定提交格式:
feat: 新功能
fix: Bug修复
docs: 文档更新
chore: 构建/工具链
refactor: 重构
```

- **难度**: 低
- **收益**: CHANGELOG 质量大幅提升

### 9.3 `release-server.bat` 增加 Pipeline 同步（推荐 ⭐⭐⭐）

**当前**: `release-server.bat` 不包含 Pipeline 同步，需要手动执行。

**优化**: 在 CI/CD 完成后自动触发 Pipeline 同步。

```bash
# release-server.bat 末尾增加
echo 等待 CI/CD 完成后执行:
echo   powershell -File scripts/pipeline-auto-sync.ps1

# 或集成到 CI 中: release.yml 最后加一个 job
pipeline-sync:
  needs: release
  runs-on: ubuntu-latest
  steps:
    - run: # 触发 pipeline 同步
```

- **难度**: 中
- **收益**: 减少手动操作步骤

### 9.4 发布前自动测试门禁（推荐 ⭐⭐⭐⭐）

**当前**: 本地发布有 DTS 检查，但缺少真正的测试门禁。

**优化**: 在 `release.mjs` 中增加可选测试步骤。

```bash
# release-server.bat 中增加 --test 选项
release-server.bat --test
# → 推送前先运行 pnpm test:all，失败则阻止发布

# 或在 CI 中增加测试 Job (release.yml)
test:
  runs-on: ubuntu-latest
  steps:
    - run: cd packages/core && npx vitest run
    - run: cd packages/server && npx vitest run
```

- **难度**: 中
- **收益**: 阻止有测试失败的版本发布

## 十、总结

### 一句话总结

| 你是...             | 用这个                  |
| ------------------- | ----------------------- |
| 初学者 / 环境不完整 | `release-server.bat`    |
| 日常 Bug 修复发版   | `release-server.bat`    |
| 需要本地验证构建    | `release-publish.bat`   |
| 紧急热修复          | `release-publish.bat`   |
| CI 不可用           | `release-publish.bat`   |
| 其他所有情况        | `release-server.bat` ⭐ |

### 核心原则

> **让机器做机器擅长的事（构建、打包、上传），让人做人擅长的事（代码审查、CHANGELOG 编写、质量把关）。**

`release-server.bat` 的设计哲学正是如此——开发者专注于代码和版本管理，CI/CD 负责繁重的构建和发布工作。
