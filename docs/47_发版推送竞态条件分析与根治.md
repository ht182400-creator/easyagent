# 47_发版推送竞态条件分析与根治

> **日期**: 2026-06-28  
> **版本**: v0.6.12  
> **问题级别**: 🔴 高 — 每次发版必现，阻断发布流程

---

## 一、问题现象

执行 `release-publish.bat` 时，在 Git push 阶段反复失败：

```
To github.com:ht182400-creator/easyagent.git
 * [new tag]         v0.6.12 -> v0.6.12
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'github.com:ht182400-creator/easyagent.git'
hint: Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart.
```

**特征**：
- Tag 推送成功，但 main 分支被拒绝
- 每次发版都会遇到
- 重新执行发版脚本依旧失败

---

## 二、根因分析

### 2.1 竞态窗口

发版流程中存在一个天然的 **竞态窗口**：

```
时间线：
  T0: Step 3 版本递增 → git commit + tag（本地）
  T1: Step 4 EXE 构建（3-5 分钟，CI 可在此期间推管线数据）
  T2: Step 7 最终 commit → git push（失败！）
```

### 2.2 竞态双方

| 竞态方 | 修改的文件 | 触发时机 |
|--------|-----------|---------|
| **发版脚本** | `version.json`, `CHANGELOG.md`, 7个 `package.json` | Step 3 & Step 7 |
| **CI 管线同步** | `docs/pipeline/pipeline-data.json`, `issue-data.json`, `test-case-mapping.json`, `project-progress-data.json`, `dashboard-data.json`, `_issue_cache.json` | 每次 CI 运行自动推送 [skip ci] commit |

**两者的修改文件完全不重叠**，但都修改了 main 分支的 HEAD。

### 2.3 竞态时序图

```
  本地: A → B(release commit)
                   ↕ 竞态窗口（构建期间）
  远程: A → C(CI pipeline sync) 
  
  结果: push 时发现远程 HEAD→C 领先于本地 HEAD→B
        → non-fast-forward，拒绝
```

---

## 三、为什么会反复失败

问题不仅存在于第一次失败，修复后重新执行依旧会失败：

### 第一层：发版脚本 push vs CI push

手动 `git pull --rebase` 修复后，重新运行脚本 → Step 3 再次 `git push` → CI 又在期间推了新数据 → 再次 rejected。

### 第二层：Git hooks 导致 unstaged changes

更隐蔽的问题：git hooks（如 `post-commit` / `pre-push`）在 commit/tag 期间自动运行 `unified-sync` 管线同步，修改 `docs/pipeline/*.json` 文件。

**错误信息**：
```
error: cannot pull with rebase: You have unstaged changes.
error: Please commit or stash them.
```

**原因链**：
```
git commit -m "release: v0.6.12"
  ↓ 触发 git hooks
  ↓ unified-sync 修改 docs/pipeline/*.json
  ↓ 文件已保存到磁盘，但未 git add
  ↓ 工作区状态: 非干净（unstaged changes）
  
git pull --rebase origin main
  ↓ ❌ 失败！rebase 要求工作区干净
```

---

## 四、修复方案

### 4.1 核心思路

在每个 `git push origin main` 之前，插入：

```bash
git stash           # 暂存 hooks 修改的脏文件
git pull --rebase   # rebase 远程 CI 管线数据
git stash pop       # 恢复 hooks 修改
git push origin main
```

### 4.2 为什么 `--rebase` 安全（不会覆盖源码）

| 远程 CI 修改的文件 | 发版本地修改的文件 |
|--------------------|-------------------|
| `docs/pipeline/pipeline-data.json` | `version.json` |
| `docs/pipeline/issue-data.json` | `package.json` × 7 |
| `docs/pipeline/test-case-mapping.json` | `CHANGELOG.md` |
| `docs/pipeline/project-progress-data.json` | `.codebuddy/memory/*.md` |
| `docs/pipeline/dashboard-data.json` | |
| `docs/pipeline/_issue_cache.json` | |

**两者文件集合完全不相交**，rebase 只会产生干净的线性历史，不会产生代码冲突。

即使极端情况下同一文件被修改，rebase 会自动暂停并要求手动解决冲突，**绝不会静默覆盖代码**。

### 4.3 修改位置

#### 位置 1：`scripts/release.mjs` — Step 3 版本递增后的 push

```javascript
// 修改前（第 368 行）
execSync('git push origin main --follow-tags', { cwd: root, stdio: 'inherit' });

// 修改后
// git hooks 可能在 commit/tag 后修改了管线文件 → stash 保护
execSync('git stash', { cwd: root, stdio: 'inherit' });
execSync('git pull --rebase origin main', { cwd: root, stdio: 'inherit' });
execSync('git stash pop', { cwd: root, stdio: 'inherit' });
execSync('git push origin main --follow-tags', { cwd: root, stdio: 'inherit' });
```

#### 位置 2：`release-publish.bat` — Step 7 管线产物最终提交后的 push

```batch
rem 修改前（原第 632 行）
git push origin main

rem 修改后
git stash
git pull --rebase origin main
git stash pop
git push origin main
```

## 五、修复后的流程

```
Step 3: 版本递增
  git add → git commit → git tag
  git stash               ← 暂存 hooks 修改的管线文件
  git pull --rebase       ← rebase 远程 CI 数据
  git stash pop           ← 恢复 hooks 修改
  git push --follow-tags  ← ✅ 成功

       ↓  3-5 分钟构建期间，CI 可能推送管线数据

Step 7: 最终提交
  git add → git commit
  git stash               ← 暂存 hooks 修改
  git pull --rebase       ← rebase 远程 CI 数据
  git stash pop           ← 恢复 hooks 修改
  git push                ← ✅ 成功
```

## 七、教训

1. **发版脚本必须假设有外部推送者**：CI pipeline sync 可能在发版任何阶段推送新 commit
2. **Git hooks 是隐形的文件修改者**：`git stash` 是保护 rebase 不被 hooks 打断的必要手段
3. **push 前加 `pull --rebase`** 应成为所有自动化 git push 脚本的标准做法
4. **文件隔离 ≠ 竞态免疫**：即使修改不同文件，push 仍然会因为分支指针落后而被拒绝
