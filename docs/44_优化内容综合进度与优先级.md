#### #16 延期说明 — Vite Library Mode 详细分析

> **结论**: 正确方案，错误时机。当前项目规模下投入产出比极低，延期至消费端≥5 或源文件>200 时重新评估。

##### 16.1 什么是 Vite Library Mode

将 `packages/frontend` 从"源码直出"改为"独立编译产物"——用 Vite 的 `build.lib` 模式把 frontend 构建成 ESM/UMD 输出，web 和 desktop 不再直接 import 源码，而是 import 编译后的 `dist/` 产物。

```
源码直出模式（当前）：                  Vite Library Mode：
  web ──→ frontend/src/*.tsx              web ──→ frontend/dist/*.js
  desktop ──→ frontend/src/*.tsx          desktop ──→ frontend/dist/*.js
  构建时 Vite 内联源码                    构建时只引用已编译产物
```

##### 16.2 当前项目硬数据 vs 触发阈值

| 指标 | 当前值 | Library Mode 触发阈值 | 差距 |
|------|:------:|:---------------------:|:----:|
| **消费端数量** | **2**（web + desktop） | ≥5 | 差 2.5 倍 |
| **frontend 源文件** | **45** 个 TS/TSX | >200 | 差 4.4 倍 |
| **源码总行数** | ~3,000 行 | >15,000 | 差 5 倍 |

> CLI 包已验证不依赖 `@easyagent/frontend`，不导入任何 frontend 模块。

##### 16.3 五维度不划算分析

**维度一：边际收益递减** — 阶段 2（平台适配器模式）已解决核心问题

```
问题维度                 当前做法          平台适配器模式(已完成)     Library Mode
────────────────────────────────────────────────────────────────────────────
模块边界                 🔴 alias 穿透     → 🟢 包名导入              → 🟢 产物隔离     (已绿)
同名冲突风险             🔴 高             → 🟢 极低                  → 🟢 零           (已绿→更低)
package exports          🔴 指向不存在文件  → 🟢 正确                  → 🟢 不变         (已绿)
扩展性(新增平台)         🔴 改 alias 配置   → 🟢 建包+main.tsx         → 🟢 不变         (已绿)
Desktop 兼容性           🟡 可用不安全      → 🟢 明确安全              → 🟢 不变         (已绿)
```

所有 🔴 在 P0+P1 已经变 🟢。Library Mode 是把"极低风险"变成"零风险"，收益约为 **5% 的边际改进**。

**维度二：开发体验降级**

```
源码直出模式（当前）:                   Library Mode:
  改 frontend/src/xxx.tsx                 改 frontend/src/xxx.tsx
  ↓ Vite HMR 即时生效 (<100ms)            ↓ 需 watch rebuild (1-3s)
  立即看到效果                             ↓ web/desktop HMR 再刷新
  🟢 即时反馈                             🟡 延迟反馈 (2-5s)
```

对日常开发的响应速度有 **3-10 倍的退化**。

**维度三：构建链新增步骤**

```
当前 5 步:  core tsup → server tsup → desktop tsup → vite build → electron-builder

新增后 6 步: core tsup → server tsup → frontend vite build(新增!) → desktop tsup → vite build → electron-builder
```

frontend 必须先于 web/desktop 构建，增加构建失败点。

##### 16.5 重新评估的硬触发条件

当以下**任一**条件满足时，应重新评估 #16：

1. **消费端 ≥ 5 个**（如新增：小程序端、VSCode WebView、CLI GUI、移动端 WebView、第三方 SDK 接入等）
2. **frontend 源文件 > 200 个**（当前 45，需增长 4 倍以上）
3. **出现 Tree-shaking 刚需**（如某消费端只需 20% 的组件但当前被全部打包）
4. **出现构建隔离需求**（如不同消费端需要不同的 Babel/PostCSS 配置）

##### 16.6 实施预案（待触发后使用）

当触发条件满足时，实施方案：

1. **`packages/frontend/vite.config.ts`** — 新增 `build.lib` 配置（entry: `src/mountApp.tsx`，formats: `['es']`）
2. **`packages/frontend/package.json`** — `exports["."]` 指向 `dist/mountApp.mjs`
3. **web/desktop 移除 alias** — `@easyagent/frontend` 不再需要 Vite alias，直接走 node_modules 解析
4. **构建脚本** — 在 tsup core 之后插入 `pnpm --filter @easyagent/frontend exec vite build`
5. **本地开发** — 可选 `vite build --watch` 或继续源码直出（两种模式通过环境变量切换）

### #08 解决方案说明

Desktop 依赖从 **119 → 36 项**（删除 83 项）。方案:

1. **tsup 内联 express/cors/ws/multer**：Desktop tsup `noExternal` 增加 `express/cors/ws/multer`，这些 server 运行时依赖被直接编码进 `main.js`
2. **根因**：pnpm isolated 模式下，electron-builder 打包只看到 `packages/desktop/node_modules/` 的直接 symlink。express 的 55 个传递依赖在根 `.pnpm/` 中，电子打包器无法访问
3. **pino 例外**：pino 内部使用 CJS `require('pino-pretty')`，esbuild 无法处理 → 保持 external + 保留其 10 个传递依赖
4. **保留项** (36): 3 workspace + 6 前端(Vite) + 11 pino + 2 electron-updater + 3 native + 4 工具 + 其他
