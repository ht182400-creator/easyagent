# v0.6.21 CI 修复全记录：package 版本对齐与 Lint 错误清零

> **日期**: 2026-06-28  
> **影响范围**: GitHub Actions CI / Release Workflow 全部失败  
> **修复后状态**: CI 全绿通过，0 errors

## 一、问题发现

v0.6.21 发版后，GitHub Actions 中 **CI 和 Release 两个 Workflow 全部失败**，所有 Job 都在 `Install dependencies` 步骤挂掉。

```
CI → All Tests → ci (ubuntu) | failure | Install dependencies
CI → All Tests → ci (windows) | failure | Install dependencies
Release → Build & Release   | failure | Install dependencies
```

---

## 二、根因分析：package.json 与 pnpm-lock.yaml 版本不一致

### 2.1 错误日志

```
ERR_PNPM_LOCKFILE_MISMATCH  
Cannot proceed with the frozen lockfile.  

  @eslint/js:  
    lockfile specifier: ^9.39.4  version: 9.39.4  
    manifest specifier: ^9.15.0  

  eslint:  
    lockfile specifier: ^9.39.4  version: 9.39.4  
    manifest specifier: ^9.15.0  

  typescript-eslint:  
    lockfile specifier: ^8.62.0  version: 8.62.0  
    manifest specifier: ^8.15.0
```

### 2.2 根因链

```
之前添加 eslint 依赖时手动写了最低版本 ^9.15.0 / ^8.15.0
    ↓
pnpm install 解析时 lockfile 记录了实际安装的最高版本 9.39.4 / 8.62.0
    ↓
package.json 未同步更新 → package.json 和 lockfile 不一致
    ↓
CI 中 pnpm install --frozen-lockfile 检测到不匹配 → 拒绝安装
    ↓
所有 Job 的依赖安装步骤失败
```

### 2.3 关键教训

- `package.json` 中的版本号应始终与 `pnpm-lock.yaml` 中的 specifier 一致
- 添加依赖时，让 pnpm 自动写入 package.json（`pnpm add xxx`），避免手动编辑版本号
- Lockfile 中的 specifier 是 `pnpm install` 解析后的实际值，是版本一致性的权威来源


## 四、后续发现：Lint 错误积累

CI 恢复后，Lint & Format Check 步骤出现大量 `no-empty` Error（Annotation 显示 11 errors）。

### 4.1 问题

12 个测试文件中存在 **51 处空 catch 块**：

```typescript
// 问题代码
try {
  rmSync(testDir, { recursive: true, force: true });
} catch (err) {}
```

全部是测试清理 `rmSync` 的 catch 块。清理失败不应影响测试结果，所以 catch 为空是**有意的**，但缺少注释违反 ESLint `no-empty` 规则。

### 4.2 修复

统一改为带注释的 catch 块：

```typescript
// 修复后
try {
  rmSync(testDir, { recursive: true, force: true });
} catch (_) { /* 测试清理失败不影响结果 */ }
```

影响文件及修复数：

| 文件 | 修复处数 |
|------|---------|
| `config-manager.test.ts` | 10 |
| `project-tools.test.ts` | 7 |
| `session-manager.test.ts` | 6 |
| `preview-media-tools.test.ts` | 6 |
| `subagent-db-tools.test.ts` | 5 |
| `knowledge-tools.test.ts` | 5 |
| `code-tools.test.ts` | 4 |
| `file-extra-tools.test.ts` | 4 |
| `builtin-tools.test.ts` | 1 |
| `git-advanced-tools.test.ts` | 1 |
| `file-tools-security.test.ts` | 1 |
| `mcp-client.test.ts` | 1 |
| **合计** | **51** |

处理方式：编写 `temp/fix-empty-catch.mjs` 脚本批量替换，避免手动 51 次重复操作。

**验证结果**: `no-empty count: 0` ✅

## 五、残余 Error 清零

修复 no-empty 后，仍有 **10 个 errors**（来自 4 个文件），是之前就积累的：

### 5.1 错误分布

| 文件 | 错误类型 | 行号 | 说明 |
|------|---------|------|------|
| `plugin-manager.test.ts` | `prefer-const` | 220 | `let registerCalled` 从未被重新赋值 |
| `plugin-sandbox.test.ts` | `no-unsafe-function-type` | 35,36,177,178 | 使用宽泛的 `Function` 类型 |
| `project-tools.test.ts` | `no-require-imports` | 294,305,322 | 使用 `require('fs').readdirSync` |
| `semantic.test.ts` | `no-misleading-character-class` | 485 | emoji 在正则字符类中需 `u` 标志 |

### 5.2 修复详情

**① prefer-const** (`plugin-manager.test.ts:220`)

```typescript
// 修复前
let registerCalled = false;

// 修复后
const registerCalled = false;
```

**② no-unsafe-function-type** (`plugin-sandbox.test.ts`)

```typescript
// 修复前
let checkPermissions: Function;
let getDangerousPermissions: Function;
let loadManifest: Function;
let getManifestPath: Function;

// 修复后
let checkPermissions: (...args: unknown[]) => unknown;
let getDangerousPermissions: (...args: unknown[]) => unknown;
let loadManifest: (...args: unknown[]) => unknown;
let getManifestPath: (...args: unknown[]) => unknown;
```

**③ no-require-imports** (`project-tools.test.ts`)

```typescript
// 修复前
import { existsSync, mkdirSync, rmSync, writeFileSync } from 'node:fs';
// ...
const files = require('fs').readdirSync(memDir);

// 修复后
import { existsSync, mkdirSync, readdirSync, rmSync, writeFileSync } from 'node:fs';
// ...
const files = readdirSync(memDir);
```

**④ no-misleading-character-class** (`semantic.test.ts:485`)

```typescript
// 修复前
expect(overview.fileTree).toMatch(/[📁📄]/);

// 修复后 — emoji 是代理对，正则字符类需 u 标志
expect(overview.fileTree).toMatch(/[📁📄]/u);
```

## 七、经验总结

| 序号 | 教训 | 预防措施 |
|------|------|---------|
| 1 | package.json 版本与 lockfile 不一致会导致 CI 完全阻塞 | 始终用 `pnpm add` 添加依赖，避免手动编辑版本号 |
| 2 | 空的 catch 块虽然有意为之，但需加注释告知 ESLint | 测试清理 catch 加 `/* 测试清理失败不影响结果 */` |
| 3 | `Function` 类型过于宽泛，应使用具体签名的函数类型 | `(...args: unknown[]) => unknown` |
| 4 | emoji 在正则字符类中需要 `u` 标志 | Unicode 正则加 `/u` |
| 5 | `require()` 在 TS 项目中应替换为 ESM import | 顶层 import，测试中也可用 |
| 6 | CI 中 lint error 积累要尽早修复，越积越多 | 发版前跑一次全量 lint 检查 |
