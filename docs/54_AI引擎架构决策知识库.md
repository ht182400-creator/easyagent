## 1. 提示词工程决策

### 1.1 系统提示词对 7B 模型行为的量化影响

#### 背景

2026-06-30 至 07-01，在 LangGraph 引擎调试过程中观察到：相同的 qwen2.5:7b 模型，通过 `ollama run` CLI 直接对话返回自然回复，而通过 EasyAgent 引擎返回工具化语气（如 "您可以告诉我需要使用哪个工具或功能"）。

#### 对比实验数据

| 调用方式 | System Prompt | 对"你好"的回复 |
|---------|---------------|---------------|
| `ollama run qwen2.5:7b` | Ollama CLI 默认 system prompt | "你好！有什么问题或者需要帮助的吗？" |
| EasyAgent LangGraph | 角色设定 + 工具准则 + 66 工具定义 | "您好！有什么我可以帮助您做的吗？您可以告诉我需要使用哪个工具或功能。" |

**差异分析**：

- CLI 回复：自然、人性化、无工具提及
- Engine 回复：正式、工具导向、主动提及"工具/功能"
- 根因：模型看到 66 个工具定义后，自行推断"我是个工具型助手"，改变了语气

#### 决策

```
系统提示词中工具使用准则的措辞，会显著影响小模型的角色认知。
越长的工具规则 + 越多的工具定义 → 模型越倾向使用"工具助手"语气。
```

#### 理由

7B 模型指令跟随能力弱于 70B+ 模型，context 中任何结构化信息（JSON schema、工具列表）都会影响其输出分布。这不是 bug，而是模型规模限制下的必然行为。

#### 适用建议

- **自己项目**：对 7B 模型，系统提示词 ≤ 200 tokens，工具准则 1-2 句话
- **其他项目借鉴**：对 ≤ 8B 的模型做 A/B 测试，对比有无工具定义下的问候/闲聊回复，确认语气偏移后针对性调整

### 1.2 工具定义污染上下文的机制

#### 背景

LangGraph 的 thinkNode 在每次 LLM 调用时，将**全部 66 个工具的定义**（每个工具的 JSON schema：name + description + parameters.properties）注入到 messages 中。

#### 机制分析

```
用户发送: "请给我写个1+100的C语言算法函数"

↓ System Prompt 包含:
  "你是一个有帮助的 AI 助手"
  "只有在需要执行具体操作时才调用工具"
  
↓ 66 个工具定义（JSON schema）同时存在于上下文:
  {
    "name": "read_file", "parameters": { "type": "object", "properties": {...} }
  }
  {
    "name": "write_file", "parameters": { ... }
  }
  ...
  
↓ qwen2.5:7b 的注意力被 JSON schema 影响
↓ 输出的代码中附带: "在JSON格式中请求这个函数内容可能不太合适..."
```

**污染路径**：工具 schema → 上下文结构化信息 → 模型关联"JSON"概念 → 渗透到不相关的输出中。

#### 决策

```
对 7B 模型，工具定义是上下文噪声，必须控制数量。
对大模型（≥ 70B），工具定义是能力信息，可以全量暴露。
```

#### 量化估算

`qwen2.5:7b` 的上下文窗口为 32K tokens。66 个工具定义的 JSON schema 约占 **6,000-8,000 tokens**（占 19%-25%），系统提示词约占 500-1,000 tokens。剩余给对话的空间不足 75%。

如果用户连续多轮对话，每轮历史消息（含上轮的 tool_calls JSON）会继续膨胀上下文。

#### 适用建议

- 对 7B 模型，工具定义总数控制在 **15-20 个以内**
- 工具定义的 `description` 字段保持 ≤ 50 字符
- 参数 `properties` 只保留必需字段，删除 `default`、`examples` 等辅助字段
- 其他项目可参考：在构造 LLM 请求前计算工具 schema 的 token 量，超过上下文的 25% 时触发告警

### 1.3 小模型 vs 大模型的提示词差异化策略

#### 背景

讨论中提出的核心问题：如果针对 qwen2.5:7b 做了提示词优化（约束更多、提示更短），对 GLM5.2、GPT5.5 等先进模型会有什么影响？

#### 分析矩阵

| 提示词策略 | 对小模型 (≤ 8B) | 对大模型 (≥ 70B) | 结论 |
|-----------|:---:|:---:|------|
| "普通聊天不调工具" | ✅ 必须 | ✅ 无害 | **通用安全** |
| 过滤 benchmark_* 工具 | ✅ 必须 | ✅ 无害 | **通用安全** |
| 简化 role description | ✅ 帮助理解 | ⚠️ 可能丢失细节 | 需区分处理 |
| 减少工具定义到 20 个 | ✅ 降低噪声 | ❌ 限制能力 | **必须区分** |
| 连续失败 3 次停止 | ✅ 安全兜底 | ✅ 安全兜底 | **通用安全** |
| "不要说 JSON/工具" patch | ✅ 纠正语气 | ⚠️ 可能多余 | 需区分处理 |

#### 决策

```
采用"模型自适应提示词"策略：
- 检测模型规模/能力
- 小模型 → 精简提示词 + 裁剪工具 + 补充 patch
- 大模型 → 完整提示词 + 全量工具 + 无 patch
```

#### 实现方案（未实施，架构设计阶段）

```typescript
// 伪代码：在 AgentFactory.ts 中根据模型名选择配置
function getPromptConfig(modelName: string): PromptConfig {
  const SMALL_MODEL_PATTERNS = /qwen.*:(7b|3b|1\.5b)|llama.*:(8b|3b)/i;
  
  if (SMALL_MODEL_PATTERNS.test(modelName)) {
    return {
      systemPrompt: SHORT_SYSTEM_PROMPT,        // ≤ 200 tokens
      toolFilter: (def) => CORE_TOOLS.has(def.name), // 仅核心 20 个
      patches: ['no-json-mention', 'natural-tone'],
    };
  }
  
  return {
    systemPrompt: FULL_SYSTEM_PROMPT,
    toolFilter: () => true,  // 全量
    patches: [],
  };
}
```

#### 代价/权衡

- **小模型**：能力受限（工具少），但行为稳定可控
- **大模型**：能力完整，但配置复杂度不增加
- **维护成本**：需要维护两套提示词模板

#### 适用建议

- 如果项目只用一种模型 → 为那个模型调优即可，不需要自适应
- 如果项目需支持多种模型 → **必须实现自适应**，否则小模型行为不可控
- 检测方式可以是：模型名正则匹配 或 从模型元数据读取参数量

### 1.4 普通聊天不应触发工具调用

#### 背景

2026-06-30，用户发送"你好"，LangGraph 引擎却进入 think→act→observe 循环，反复调用 benchmark_report/benchmark_run/benchmark_load，最终 `Recursion limit of 25 reached`。

#### 根因链

```
用户输入: "你好"
    ↓ thinkNode: LLM 看到 69 个工具（含 benchmark_*）
    ↓ LLM 误判: "这是个技术场景，我应该用 benchmark 工具"
    ↓ actNode: benchmark_report 执行失败（无评测会话）
    ↓ observeNode: 无条件继续循环
    ↓ thinkNode: LLM 收到失败结果，换 benchmark_run
    ↓ ... 重复 25 轮 → Recursion limit
```

#### 决策

**多层防御，而非单点修复**：

| 层级 | 措施 | 效果 |
|------|------|------|
| 提示词层 | 系统提示词明确："普通聊天、问候、解释性问题时，直接回复文本，不要调用工具" | 源头引导 |
| 工具层 | AgentFactory 默认过滤 `benchmark_*` 工具 | 消除误导源 |
| 状态层 | AgentState 新增 `consecutiveFailures` 字段 | 跟踪失败 |
| 节点层 | observeNode 检测连续失败 ≥ 3 次 → 停止循环 | 循环刹车 |
| 运行时层 | `streamEvents` 传入 `recursionLimit = maxTurns * 3 + 10` | 防止 LangGraph 先行截断 |

#### 理由

单点修复（如只改提示词）不可靠，因为：
- 模型可能忽略提示词（特别是指令跟随弱的小模型）
- 工具定义的存在本身就诱导工具调用
- LangGraph 框架的默认 recursionLimit 可能先于业务 maxTurns 触发
- 需要最后防线（连续失败计数器），无论什么原因导致的循环都能终止

#### 适用建议

这是**可复用的防御模式**：任何基于 LLM 的工具调用系统，都应该有类似的**多层防御**。

## 2. 工具暴露策略决策

### 2.1 全量暴露 66 个工具的风险

#### 上下文成本

| 项目 | 数量 |
|------|------|
| 工具总数（过滤前） | 69 |
| 工具总数（过滤 benchmark_* 后） | 66 |
| 每个工具 JSON schema 平均 tokens | ~90-130 |
| 66 个工具总 tokens | ~6,000-8,000 |
| qwen2.5:7b 上下文窗口 | 32,768 tokens |
| 工具占比 | **19%-25%** |

#### 非成本风险

1. **诱导误调用**：模型看到"benchmark_report"→ "用户可能想让我跑评测" → 误触发
2. **语气偏移**：模型看到大量工具 → 自我认知变为"工具执行器" → 回复包含工具术语
3. **选择困难**：66 个工具中，名称相近的（如 `find_imports` vs `find_definitions` vs `code_find_references`）增加模型选错的概率

#### 决策

```
工具暴露 = 能力暴露 ≠ 越多越好。

工具集大小应与模型能力匹配：
- 7B: 15-20 个核心工具
- 13B-30B: 30-40 个工具
- 70B+: 全量 66 个工具
```

## 3. 引擎循环控制决策

### 3.1 recursionLimit 与 maxTurns 的语义差异

#### 问题

LangGraph 的 `recursionLimit`（默认 25）按**图步数**计算，而业务层的 `maxTurns` 按**对话轮次**计算。两者单位不同：

```
1 轮对话 (turn) = think → routeAfterThink → (act → observe)? 
                ≈ 1-3 个图步 (step)

maxTurns = 25 → 需要 25-75 个图步
recursionLimit = 25 → 最多 25 个图步 → 在 maxTurns 生效前截断
```

#### 决策

```typescript
// 在 Agent.ts stream() 中
const recursionLimit = this.maxTurns * 3 + 10;
//                                    ↑     ↑
//                          每次循环约3步  10步安全余量
```

#### 理由

- `maxTurns * 3`：覆盖 think→act→observe 的每轮最大步数
- `+10`：覆盖初始化（__start__ → think 的额外步）
- 确保 `maxTurns` 是真正的限制者，而非 LangGraph 默认值

#### 适用建议

任何使用 LangGraph 的项目，如果业务层有自己的轮次控制，都必须**同步调整 recursionLimit**，避免框架限制先于业务限制触发。

### 3.2 连续工具失败停止机制

#### 状态设计

```typescript
// AgentState.ts
consecutiveFailures: Annotation<number>({
  reducer: (_, next) => next,
  default: () => 0,
})
```

#### 计数逻辑

```typescript
// actNode.ts
// 每轮执行后：
//   - 有任意工具成功 → consecutiveFailures = 0
//   - 全部失败 → consecutiveFailures += 本轮失败数
```

#### 停止判断

```typescript
// observeNode.ts
const maxConsecutiveFailures = 3;
if ((state.consecutiveFailures || 0) >= maxConsecutiveFailures) {
  return {
    shouldContinue: false,
    finalResponse: `已连续 ${state.consecutiveFailures} 次工具调用失败，停止循环以避免无限重试。请检查工具配置或重试。`,
  };
}
```

#### 为什么是 3 次？

| 阈值 | 效果 |
|:---:|------|
| 1 | 太敏感，偶发网络波动就中断 |
| 2 | 仍然敏感，一次重试不够 |
| **3** | **合理**：给 LLM 2 次自我纠正机会 |
| 5+ | 太宽松，浪费 token 和时间 |

#### 适用建议

这个模式适用于**所有工具调用系统**，不限于 EasyAgent。核心思想：

```
工具调用 = 可能失败
失败循环 = 工具失败 → LLM 重试 → 再次失败 → ...
防护 = 连续失败计数 + 阈值停止 + 用户友好提示
```

### 3.3 流式输出兜底：finalResponse 优先级

#### 背景

LangGraph 的 `streamEvents()` 以流式方式产生事件，但如果底层 LLM 调用是非流式的（如通过 adapterBridge），`on_chat_model_stream` 事件不会触发，导致 `chunkCount=0`。

#### 兜底链（优先级从高到低）

```
1. finalResponse（observeNode 设置的停止/完成消息）
   ↓ 不存在则
2. checkpoint 最后一条 AI 消息的 content
   ↓ 不存在则
3. 空（前端显示空回复）
```

#### 实现

```typescript
// Agent.ts stream()
if (!hasResponseContent) {
  const finalState = await this.checkpointer.getLatestState(sessionId);
  
  // 优先：observeNode 的停止信号
  const finalResponse = finalState?.finalResponse as string;
  if (finalResponse) {
    yield { type: 'response', data: { content: finalResponse } };
    return;
  }
  
  // 其次：checkpoint 中的 AI 消息
  const lastMessage = stateMessages?.[stateMessages.length - 1];
  const finalContent = extractMessageContent(lastMessage);
  if (finalContent) {
    yield { type: 'response', data: { content: finalContent } };
  }
}
```

#### 理由

`finalResponse` 优先级高于 checkpoint 消息，因为：
- 它是 observeNode 有意设置的**停止+解释**
- 包含上下文信息（如 "已连续 3 次失败"）
- checkpoint 最后一条消息可能是工具调用结果，不是对用户有用的回复

## 4. 模型适配层决策

### 4.1 API 调用 vs CLI 调用的行为差异

#### 差异来源

| 维度 | `ollama run` CLI | EasyAgent API 调用 |
|------|-----------------|-------------------|
| System Prompt | Ollama 内置默认 | EasyAgent 自定义 |
| 工具定义 | 无 | 66 个 JSON schema |
| temperature | CLI 默认（通常 0.7） | 取决于 adapter 配置 |
| top_p / top_k | CLI 默认 | 取决于 adapter 配置 |
| 消息格式 | [user] 单条 | [system + user] 多条 |

#### 结论

```
同样的模型，通过不同"外壳"（CLI vs 自定义 API）调用，
回复内容差异可达 30-50%。这不是模型的问题，
是调用方馈入的上下文不同。
```

#### 调试方法

如果要对比：构造与 CLI 完全相同的 messages 数组（只含 user message，无 system prompt，无工具定义），通过 API 发送，对比回复。如果此时回复一致 → 差异来自上下文，非模型本身。

### 4.3 小模型优化的通用性原则

```
1. 上下文是稀缺资源 — 每 1 token 都要有明确目的
2. 规则要显式、具体、不矛盾 — 小模型不会"理解意图"
3. 工具 schema 是双刃剑 — 提供能力同时制造噪声
4. 多次短提示 > 一次长提示 — 小模型处理长 prompt 时容易丢失中段信息
5. 示例 (few-shot) 比规则 (zero-shot) 更有效 — 给 1-2 个正确行为示例
```

## 5. 上下文管理决策

### 5.1 上下文窗口预算模型

#### qwen2.5:7b (32K 窗口) 预算分配建议

```
┌────────────────────────────────────────────┐
│ 系统提示词:     200 tokens   (0.6%)        │
│ 工具定义:      2,000 tokens   (6.3%)       │ ← 15 个精简工具
│ 对话历史:     20,000 tokens  (62.5%)       │ ← 最近 N 轮
│ 当前用户输入:   1,000 tokens   (3.1%)       │
│ 预留输出:      4,000 tokens  (12.5%)       │ ← 模型回复空间
│ 余量:         4,800 tokens  (15.0%)       │ ← 安全缓冲
└────────────────────────────────────────────┘
```

#### 对比：全量 66 个工具时的预算

```
┌────────────────────────────────────────────┐
│ 系统提示词:     200 tokens   (0.6%)        │
│ 工具定义:      8,000 tokens  (25.0%)       │ ← ⚠️ 占比过高
│ 对话历史:     14,000 tokens  (43.8%)       │ ← 被压缩
│ 当前用户输入:   1,000 tokens   (3.1%)       │
│ 预留输出:      4,000 tokens  (12.5%)       │
│ 余量:         4,800 tokens  (15.0%)       │
└────────────────────────────────────────────┘
```

工具定义从 6.3% 膨胀到 25%，吃掉的是对话历史的配额。多轮对话时更容易触发截断。

### 5.2 工具 schema 在上下文中的 token 成本

#### 单工具 token 成本估算

以 `read_file` 为例：

```json
{
  "type": "function",
  "function": {
    "name": "read_file",
    "description": "读取指定路径的文件内容",
    "parameters": {
      "type": "object",
      "properties": {
        "filePath": { "type": "string", "description": "文件路径" },
        "offset": { "type": "number", "description": "起始行" },
        "limit": { "type": "number", "description": "读取行数" }
      },
      "required": ["filePath"]
    }
  }
}
```

**约 90-110 tokens**。

66 个类似工具 → 6,000-8,000 tokens。

#### 优化方向

1. **description 精简化**：当前部分工具 description 过长（50+ 字），可压缩到 20 字
2. **properties 瘦身**：删除不必要的参数描述，仅保留 `type`
3. **移除 default/examples**：这些字段增加 token 但对小模型选工具帮助有限
4. **动态注入**：只在需要时注入特定工具组，而非全量

## 6. 失败恢复模式决策

### 6.1 Checkpoint 兜底提取

#### 背景

LangGraph 的 SqliteCheckpointer 使用 `JSON.stringify/parse` 持久化 state，导致 messages 数组中的 `BaseMessage` 实例（AIMessage/HumanMessage 等）丢失原型方法。

#### 问题现象

```
state.messages[0]?.getType is not a function
msg instanceof AIMessage  → false (checkpoint 还原后)
msg.tool_calls  → undefined (需要通过 kwargs.tool_calls 访问)
```

#### 决策

所有消息处理代码使用**工具函数**而非直接原型方法调用：

```typescript
// ❌ 不安全（checkpoint 还原后失效）
if (msg instanceof AIMessage) { ... }
const type = msg.getType();

// ✅ 安全（兼容序列化后的普通对象）
const type = getMessageType(msg);  // 优先 getType()，否则 msg.type，兜底 constructor.name
const hasToolCalls = msg.tool_calls?.length > 0 || msg.kwargs?.tool_calls?.length > 0;
```

#### 工具函数参考

`packages/langgraph/src/graph/messageUtils.ts`

### 6.2 连续失败计数器设计

（详见 §3.2）

#### 设计原则

```
1. 成功 → 重置为 0（表示"恢复健康"）
2. 失败 → 累加（表示"持续恶化"）
3. 达到阈值 → 停止 + 告知用户
4. 阈值可配置，默认 3
```
