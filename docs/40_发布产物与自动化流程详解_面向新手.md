## 一、本文解决什么问题？

在第一次接触 EasyAgent 发布流程时，新手通常会问：

> _"latest.yml 是什么？跟 .github/workflows/release.yml 是同一个东西吗？"_
> _"release.yml 是自动创建的吗？我需要手动维护它吗？"_
> _"上传说的是 release.yml 还是 latest.yml？到底有几个 yml 文件？"_
> _"GitHub Release Notes 要不要手动填？"_
> _"发布到底上传了几个文件？各自有什么用？"_

**本文用一张全景图 + 逐个拆解的方式解答上述所有问题。**

## 三、名称澄清：两个最易混淆的 yml 文件

| 文件名                              | 在哪                        | 谁创建的                   | 用途                                                            |    需要手动维护？     |
| ----------------------------------- | --------------------------- | -------------------------- | --------------------------------------------------------------- | :-------------------: |
| **`latest.yml`**                    | `packages/desktop/release/` | electron-builder 自动生成  | Desktop EXE 自动更新元数据，客户端读取它判断"是否有新版本"      |       ❌ 全自动       |
| **`.github/workflows/release.yml`** | `.github/workflows/`        | 开发者手动编写并提交到仓库 | GitHub Actions CI/CD 流水线定义（何时构建、怎么构建、上传到哪） | ✅ 改构建配置时手动改 |

### 3.1 `latest.yml` —— 桌面端自动更新的"版本号公告板"

每次构建 Desktop EXE 时，electron-builder 自动生成此文件。内容示例：

```yaml
version: 0.6.6
files:
  - url: EasyAgent-0.6.6-win-x64.exe
    sha512: S3+4u5h6BNigETeZzu4...
    size: 110463941
path: EasyAgent-0.6.6-win-x64.exe
releaseDate: '2026-06-27T02:55:11.130Z'
```

**工作流程**：

```
用户打开 EasyAgent Desktop
  → 应用读取 GitHub Release 上的 latest.yml
    → 比对本地版本号
      → 如果远端版本 > 本地版本 → 弹出"发现新版本"通知
        → 下载新 EXE → 安装 → 重启
```

> **关键理解**：`latest.yml` 是为 Desktop EXE 服务的，与 server 包（`@easyagent/server`）无关。Server 是纯 Node.js 服务，不使用 electron 自动更新。

### 3.2 `.github/workflows/release.yml` —— CI/CD 流水线的"操作手册"

这是一个 **GitHub Actions 工作流文件**，定义：当有人推送 `v*` 标签时，GitHub 服务器应该做什么。

**它不是自动创建的，而是手动编写后提交到仓库的**。当前内容概要：

```yaml
name: Release

on:
  push:
    tags: ['v*'] # 推送 v0.6.6 等标签时自动触发
  workflow_dispatch: # 也支持在 GitHub 网页上手动触发

jobs:
  version: # 提取版本号
  desktop: # Windows Runner 上构建 Desktop EXE（约 5 分钟）
  web: # Ubuntu Runner 上构建 Web Dashboard（约 2 分钟）
  release: # 合并产物 → 创建 GitHub Release → 上传 Assets
```

**什么时候需要修改它**：

| 场景                    | 需要改吗  |
| ----------------------- | :-------: |
| 日常发布                | ❌ 不用管 |
| 加 macOS/Linux 构建平台 | ✅ 需要改 |
| 升级 Node.js 版本       | ✅ 需要改 |
| 修改构建参数/环境变量   | ✅ 需要改 |
| 添加新的上传产物        | ✅ 需要改 |

## 四、一次发布到底上传了什么？

每次 GitHub Release 包含 **3 个 Assets**：

### ① `EasyAgent-{version}-win-x64.exe`（~105 MB）

NSIS 安装包。用户下载后双击即可安装 EasyAgent Desktop。

- 格式：Windows 安装程序
- 包含：Electron + 后端 server + 前端 UI（全内联）
- 生成工具：electron-builder（NSIS 模式）

### ② `EasyAgent-{version}-win-x64.exe.blockmap`（~200 KB）

块映射文件。配合自动更新使用，让用户**只下载 EXE 中变化的部分**（增量更新），而非每次重下 105 MB 的完整包。

- 格式：二进制块映射（electron-builder blockmap）
- 用途：自动更新时计算差异块，实现增量下载
- 没有它：自动更新仍能工作，但每次都是全量下载

### ③ `latest.yml`（~200 B）

见上文 3.1。自动更新元数据文件。

### ④（仅 CI/CD 模式额外）`web-dist.tar.gz`

Web Dashboard 静态文件压缩包，用于部署到 Web 服务器。仅在 `release-server.bat` 触发的 GitHub Actions 构建中产生，本地 `release-publish.bat` 不产生此文件。

## 五、GitHub Release Notes —— 手动还是自动？

### 5.1 两种 Release Notes 策略

| 上传方式                                          | Release Notes 来源                               |   效果好？    |
| ------------------------------------------------- | ------------------------------------------------ | :-----------: |
| **gh CLI**（`release-publish.bat` 选项1）         | 读取本地 `CHANGELOG.md` 作为 Release Notes       |    ✅ 完整    |
| **`_release.mjs`**（`release-publish.bat` 选项2） | 固定的占位文字（`EasyAgent v0.6.6 - verify...`） | ⚠️ 需手动补充 |
| **GitHub Actions**（`release-server.bat`）        | 从 git tag message 自动生成                      |   ⚠️ 较简略   |

### 5.2 如果 Release Notes 不完整

去 Release 页面点 **"Edit release"**，把 `CHANGELOG.md` 中对应版本条目的内容粘贴进去即可：

```
https://github.com/ht182400-creator/easyagent/releases/tag/v0.6.6
```

### 5.3 那 release.mjs 末尾的提示什么意思？

运行 `release.mjs`（版本升级）时末尾会显示：

> _"下一步: 在 GitHub Release 页面手动补充 Release Notes"_

这是因为 `release.mjs` 只负责：版本号更新 → git commit → tag → push。它**不创建** GitHub Release 页面。

当你用 `release-publish.bat` 全流程时，Step 5 会自动创建 Release 并上传产物——这条提示针对的是**只运行了 release.mjs 但没继续执行后续步骤**的场景。

## 六、关键结论速查表

| 问题                                             | 答案                                   |
| ------------------------------------------------ | -------------------------------------- |
| `latest.yml` 是自动生成的吗？                    | ✅ 每次构建自动生成                    |
| `.github/workflows/release.yml` 是自动生成的吗？ | ❌ 手动编写并提交的                    |
| 平时发布需要改 `release.yml` 吗？                | ❌ 不需要                              |
| `latest.yml` 跟 server 有关系吗？                | ❌ 没关系，只用于 Desktop EXE 自动更新 |
| 每次发布必须上传 `latest.yml` 吗？               | ✅ 必须，否则客户端自动更新失效        |
| 发布上传几个文件？                               | 3 个：EXE + blockmap + latest.yml      |
| Release Notes 要手动填吗？                       | 用 gh CLI 则自动，否则需手动补充       |

## 七、新手检查清单

第一次参与发布前，确认以下两点：

- [ ] 已阅读 `38_双通道发布指南_本地vs服务器.md`（了解两种发布通道）
- [ ] 已理解 `latest.yml` ≠ `.github/workflows/release.yml`（名称相似但完全不同）

如果你是**日常发布**：

- [ ] 运行 `release-publish.bat` 或 `release-server.bat` 二选一
- [ ] 观察输出日志，确认 3 个产物上传成功
- [ ] 检查 GitHub Release 页面，补充 Release Notes（如果需要）

如果你是**修改 CI/CD 配置**：

- [ ] 编辑 `.github/workflows/release.yml`
- [ ] `git commit` + `git push` 提交修改
- [ ] 下一个版本发布时生效
