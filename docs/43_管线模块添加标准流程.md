# 管线模块添加标准流程

> **目的**: 将"每次分析管线结构再决定怎么加模块"变成一个机械操作，零决策成本。
> **适用版本**: v0.6.6+
> **最后更新**: 2026-06-27

---

## 决策树：先判断你要加什么

```
你要添加什么？
├── 新功能模块（如 "知识库RAG"、"MCP协议"）
│   → 场景A: 功能模块
├── 新优化分支（如 "前端架构优化"、"性能优化"）
│   → 场景B: 优化分支
├── 已有模块增加测试数据
│   → 场景C: 测试数据更新
└── 文档索引更新
    → 场景D: 文档登记
```

**80% 的日常操作只涉及场景 A 或场景 B。**

> **💡 组合操作**: 以上场景可组合使用。例如：
> - 加模块 + 同时加了测试 → **A + C**
> - 加分支 + 改了文档 → **B + D**
> - 加模块 + 新建阶段 + 测试数据 → **A(新阶段) + C**
>
> 执行时按各场景的"需要修改的文件"清单合并操作，共用一次 `replace_in_file` 批次。

## 场景A：新功能模块

### 判断条件

- 新增了一个业务功能模块（如"知识库RAG"、"消息队列"）
- 模块有独立的 `packages/` 子目录或明确的功能边界
- 该模块会被管线看板展示为独立节点

### 需要修改的文件（4 个）

#### A1. `docs/pipeline/pipeline-data.json` → `pipeline.phases[所属阶段].nodes[]`

**追加位置**: 所属 Phase 的 `nodes` 数组末尾。

**定位命令**（不用肉眼翻 10000+ 行）：

```bash
# 1. 找到所属 Phase 的 nodes 最后一个元素
node -e "const d=require('./pipeline-data.json');
  const phase=d.pipeline.phases.find(p=>p.id==='P4');
  console.log('Phase:', phase.label, '| 节点数:', phase.nodes.length);
  const last=phase.nodes[phase.nodes.length-1];
  console.log('最后节点:', JSON.stringify(last,null,2));"

# 2. 用 replace_in_file，old_str = 最后一个 node 的完整 JSON + 后续 "],"
#    例如：如果 P4 最后一个节点是 f13，old_str = 包含 f13 完整对象及其后的 "],"
```

**新增条目模板**:

```json
{
  "id": "f{N}", // ← 找当前最大的 f 编号 +1
  "icon": "🔧", // ← 选一个相关 emoji
  "label": "模块中文名",
  "desc": "一行描述\n可换行写更多",
  "status": "done", // done | pending | in-progress
  "testCount": 0 // 有测试数据后更新
}
```

**id 编号规则**: 查找 `pipeline.phases[*].nodes[*].id` 中最大的 `f{N}`，`+1`。

- 当前最大: `f16` → 新增为 `f17`

**所属阶段选择**:
| 阶段 | 适用模块类型 |
|------|------------|
| P0 基础层 | 核心依赖、底层设施 |
| P1 架构层 | 模块间关系、架构改进 |
| P2 体验层 | 用户可感知的功能优化 |
| P3 分发层 | 安装/部署/分发渠道 |
| P4 发布层 | 发布/升级/版本管理 |
| P5 运维层 | 管线/监控/CI |
| P6+ | 新增阶段，按需创建 |

#### A2. `docs/pipeline/project-progress-data.json` → `phases[]`

**判断**: 新模块属于已有阶段还是新阶段？

**已有阶段**: 在对应 Phase 的 `tasks[]` 追加任务条目:

```json
{
  "id": "{p}{阶段号}-{简称}",
  "name": "任务中文名",
  "desc": "一句话描述",
  "status": "done", // pending | done | skipped
  "time": "2h", // 预估工时
  "detect": {
    // 如何检测任务完成
    "files": ["packages/core/src/新模块/入口文件.ts"]
  }
}
```

**新阶段**: 先追加 Phase 对象，再加 tasks:

```json
{
  "id": "P{N}", // 当前最大 P 编号 +1
  "name": "阶段中文名",
  "status": "in-progress",
  "priority": "p{N}", // 小写 p + 编号
  "tasks": [
    /* 按上方的任务模板填 */
  ]
}
```

#### A3. `docs/pipeline/test-case-mapping.json` → `modules`

**追加位置**: 在 `"modules"` 对象内，`"_utils"` 之前插入。

**定位最后一个业务键**:

```bash
# 找 modules 中最后一个非 _utils 的 key（作为 old_str 的一部分）
node -e "const d=require('./test-case-mapping.json');
  const keys=Object.keys(d.modules).filter(k=>!k.startsWith('_'));
  console.log('LAST_KEY:', keys[keys.length-1]);"
# 然后用 replace_in_file，old_str = 该 key 的完整 JSON 对象 + 逗号 + 换行 + "_utils"
```

**新增条目模板**:

```json
"f{N}": {
  "moduleId": "f{N}",
  "totalCases": 0,
  "subModules": {
    "包名": {
      "name": "模块中文描述",
      "cases": 0,
      "files": []
    }
  }
}
```

**moduleId 必须与 A1 中的 `id` 一致**。

#### A4. 评分更新（仅新阶段时）`docs/pipeline/project-progress-data.json` → `metadata`

**仅当新建了 Phase（场景A 的新阶段变体）时才需要**。

更新 `metadata.kpi._scoreDimensions` 中的 `"分支优化"` 维度:

- `score` 调整（完成模块数 / 总模块数）
- `note` 更新为最新状态

更新 `metadata.score.scoreHistory[]` → 追加一条:

```json
{
  "version": "v0.X.X",
  "date": "2026-XX-XX",
  "score": {新分数},
  "note": "新增XX模块"
}
```

## 场景B：新优化分支

### 判断条件

- 这是一个跨模块的改进任务组（如 "前端架构优化"）
- 包含多个子任务，但不太适合作为独立 Phase
- 会出现在管线的"分支优化"视图

### 需要修改的文件（4 个）

#### B1. `docs/pipeline/pipeline-data.json` → `pipeline.branches[]`

**追加位置**: `branches` 数组末尾。

**新增条目模板**:

```json
{
  "id": "B{N}", // ← 当前最大 B 编号 +1
  "label": "分支中文名",
  "source": "P{N}", // ← 所属 Phase ID
  "nodes": [
    {
      "id": "b{N}{字母}", // ← 分支号 + 小写字母 a,b,c...
      "icon": "🚑",
      "label": "子任务中文名",
      "desc": "描述\n可选换行",
      "status": "pending",
      "testCount": 0
    }
  ]
}
```

**id 编号规则**:

- Branch ID: 找最大 `B{N}`，`+1`。当前最大 `B4` → 新增为 `B5`
- Node ID: `b{分支号}{a|b|c...}`, 从 `a` 开始递增

#### B2. `docs/pipeline/project-progress-data.json` → `phases[]`

**判断**: 分支属于已有阶段还是新阶段？

**已有阶段**: 在该 Phase 的 `tasks[]` 追加每个子节点对应的任务。

**新阶段**: 先追加 Phase 对象（同场景A的新阶段模板），再加 tasks。

**任务 ID 命名**: `p{阶段号}-{简短关键词}`

#### B3. `docs/pipeline/test-case-mapping.json` → `modules`

**追加位置**: `"modules"` 内，`"_utils"` 之前。

每个分支节点都要一个映射条目:

```json
"b{N}{字母}": {
  "moduleId": "b{N}{字母}",
  "totalCases": 0,
  "phase": "P{N}",
  "branch": "B{N}",
  "subModules": {
    "子模块key": {
      "name": "子模块中文描述",
      "cases": 0,
      "files": []
    }
  }
}
```

**注意**: 分支节点的 `moduleId` 有小写的 `phase` 和 `branch` 字段（功能模块的 `f{N}` 没有这些字段）。

#### B4. 评分更新 `docs/pipeline/project-progress-data.json` → `metadata`

必须更新:

1. `metadata.kpi._scoreDimensions` → `"分支优化"` 维度:
   - `score`: 调整为 `{完成分支数}/{总分支数} * 100`
   - `note`: 如 `"10完成/1进行中 — B5新增XXX"`

2. `metadata.score.dimensions` → 同上同步

3. `metadata.score.scoreHistory[]` → 追加里程碑条目

4. `scoreHistory[]` (顶层) → 追加可视化条目:

```json
{
  "label": "v0.X.X B5新增XXX (2026-XX-XX)",
  "value": {新分数}
}
```

## 场景C：测试数据更新

### 判断条件

- 已有的模块新增了测试用例
- 已有模块的测试数量变化
- **不需要新增模块**

### 需要修改的文件（3 个，含同步更新）

#### C1. `docs/pipeline/test-case-mapping.json`

更新对应 `moduleId` 条目:

- `totalCases`: 更新总数
- `subModules.{key}.cases`: 更新子模块用例数
- `subModules.{key}.files`: 追加新测试文件

#### C1b. `docs/pipeline/pipeline-data.json` → `node.testCount`（同步更新）

> ⚠️ 容易遗漏！`test-case-mapping.json` 的 `totalCases` 和 `pipeline-data.json` 的 `testCount` 是**同一数据的两份拷贝**。更新测试后必须两处同步，否则看板节点显示的测试数量与映射数据不一致。

定位要更新的 node:

```bash
# 找到 pipeline-data.json 中对应节点的 testCount
node -e "const d=require('./pipeline-data.json');
  const allNodes=[...d.pipeline.phases.flatMap(p=>p.nodes),...d.pipeline.branches.flatMap(b=>b.nodes)];
  const node=allNodes.find(n=>n.id==='f1');
  console.log('当前 testCount:', node.testCount);"
```

#### C2. `docs/pipeline/project-progress-data.json` → `metadata.kpi`

更新:

- `testCases`: 总用例数
- `testPassRate` / `testPassed` / `testFailed` / `testSkipped`: 按实际结果
- `_scoreDimensions` → `"测试覆盖"` note

```bash

# 2. ID 交叉校验
node -e "
const p = JSON.parse(require('fs').readFileSync('docs/pipeline/pipeline-data.json','utf8'));
const t = JSON.parse(require('fs').readFileSync('docs/pipeline/test-case-mapping.json','utf8'));
const pIds = new Set();
p.pipeline.phases.forEach(ph=>ph.nodes.forEach(n=>pIds.add(n.id)));
p.pipeline.branches.forEach(br=>br.nodes.forEach(n=>pIds.add(n.id)));
const tIds = new Set(Object.keys(t.modules));
const missing = [...pIds].filter(id=>!tIds.has(id));
const extra = [...tIds].filter(id=>!pIds.has(id) && id!=='_utils');
if(missing.length) console.log('❌ pipeline-data.json 有但 test-case-mapping.json 缺少:', missing);
if(extra.length) console.log('❌ test-case-mapping.json 有但 pipeline-data.json 缺少:', extra);
if(!missing.length && !extra.length) console.log('✅ 所有模块 ID 交叉匹配一致');
"
```

## 格式对照表

### scoreHistory：两种格式

| 位置 | 数组路径 | 字段格式 | 用途 | 当前条目数 |
|------|---------|---------|------|:--------:|
| 顶层 | `scoreHistory[]` | `{ label: string, value: number }` | 看板柱状图可视化 | 8 条 |
| metadata 内 | `metadata.score.scoreHistory[]` | `{ version: string, date: string, score: number, note: string }` | 版本里程碑时间线 | 8 条 |

> ⚠️ **格式不兼容**：写入时严格区分。用错格式会导致前端渲染崩溃。

### test-case-mapping.json：两种节点格式

| 类型 | 示例 ID | 字段 | 说明 |
|------|---------|------|------|
| 功能模块 | f1-f16 | `moduleId, totalCases, subModules` | 无 phase/branch |
| 分支节点 | b1a-b4d | `moduleId, totalCases, phase, branch, subModules` | 有 phase/branch |

> 分支节点必须包含 `phase` 和 `branch` 字段（v1.1 已统一补充）。

### testCount 与 totalCases 双维护

| 文件 | 字段 | 同步规则 |
|------|------|---------|
| `pipeline-data.json` 的 `node.testCount` | 看板节点显示 | 更新测试后**必须同步** |
| `test-case-mapping.json` 的 `module.totalCases` | 测试映射计数 | 更新测试后**必须同步** |

> ⚠️ 两处必须同步更新，否则看板与映射数据不一致。

### Phase 插入位置

Phase 在 `project-progress-data.json` 中按 `id` 字母/数字序排列，**不要盲追加到数组末尾**。
- 读出 `phases[]` → 按 `id` 排序 → 找到正确位置 → `splice` 插入
- 或追加后运行 `phases.sort((a,b) => a.id.localeCompare(b.id))`

## 精确评分公式

以下公式来自 `docs/pipeline/lib/pipeline-config.mjs` 的 `calculateScore()` 函数：

### 1. 功能完整度（满分 100）

```
functionalScore = round((doneNodes + inProgressNodes*0.5) / totalNodes * 100)
```

- `doneNodes`: status === 'done' 的主线节点数
- `inProgressNodes`: status === 'in-progress' 的主线节点数（按 0.5 计）
- `totalNodes`: 所有 `PHASES.flatMap(p => p.nodeIds)` 的节点数

### 2. 测试通过率（满分 100）

```
passRate = round(totalPassed / (totalPassed + totalFailed + totalSkipped) * 100)
```

- 数据来源：Vitest JSON 报告（`_vitest-core.json`, `_vitest-server.json` 等）
- `testCases` 始终取 `test-case-mapping.json` 的 `_meta.totalTestCases`（权威值）

### 3. 综合评分

```
scoreTotal = weightedAvg(functionalScore, testScore, branchScore, ...)
```

权重与维度定义见 `pipeline-config.mjs` 的 `_scoreDimensions`。

# 附录 A：v1.0 可行性审计报告

> **审计日期**: 2026-06-27
> **审计方法**: 逐条对照实际数据文件验证指南中的每条指令，模拟操作者视角执行。

## 审计结论

指南的**思路框架正确**（决策树 → 场景模板 → 校验），但**可操作性约 60%**。核心问题：指南假设操作者"打开文件→肉眼定位→手动编辑"，但在 10000+ 行的 JSON 文件面前，这种假设不成立。以下 8 处不足按严重度排序。

---

## 不足 1：Phase 顺序不按数字排列，盲追加会导致乱序

### 证据

```json
// project-progress-data.json 中的实际顺序:
"phases": [
  { "id": "P0" },  // line 12
  { "id": "P1" },  // line 53
  { "id": "P2" },  // line 136
  { "id": "P4" },  // line 215  ← 跳号! P3 去哪了?
  { "id": "P3" },  // line 295  ← P3 在 P4 之后!
  { "id": "P5" },  // line 328
  { "id": "P6" },  // line 372
]
```

### 问题

指南说"追加到 phases 数组末尾"。当前 P3 已经在 P4 之后，再盲追加 P7 到末尾虽然碰巧正确，但如果未来有人要在 P3 和 P4 之间插入一个阶段，按指南追加就会乱序。

### 影响程度

🟡 **中等** — 当前碰巧适用，但不可靠。

### 修复方向

改为"按 `id` 字段排序插入"（读出数组 → 找到目标位置 → insert），或改为始终追加+在末尾备注"如需保持顺序，请手动调整位置"。

---

## 不足 2：两个 `scoreHistory` 数组格式不同，指南混用

### 证据

```json
// project-progress-data.json 实际有两个 scoreHistory:

// 1. 顶层 scoreHistory[] (line 416) — 给管线看板柱状图用
{ "label": "审查前", "value": 68.4 }           // 字段: label, value

// 2. 根 nodes[].metadata.score.scoreHistory[] (line 553) — 给评分历史时间线用
{ "version": "v0.1.0", "date": "2026-06-18", "score": 30, "note": "..." }
                                                // 字段: version, date, score, note
```

### 问题

指南 B4 列出了两种 scoreHistory 但没有给出字段对照表。操作者很容易把 `{label, value}` 格式写入到 `metadata.score.scoreHistory[]`（该数组期望 `{version, date, score, note}`），或者反过来。两种格式**完全不兼容**。

### 实际数据量

| 数组                            | 条目数 | 用途                       |
| ------------------------------- | :----: | -------------------------- |
| 顶层 `scoreHistory[]`           |  7 条  | 看板柱状图可视化的评分演进 |
| `metadata.score.scoreHistory[]` |  8 条  | 详细版本里程碑记录         |

### 影响程度

🔴 **严重** — 格式写错会导致前端渲染崩溃。

### 修复方向

在指南 B4 处添加格式对照表格，标明哪个数组配哪种对象格式。

## 不足 3：10000+ 行文件中"找 nodes 末尾"不可操作

### 证据

```
pipeline-data.json 结构:
  Line 1-346:   pipeline.phases[] + pipeline.branches[]  ← 要修改的部分
  Line 348-447: kpi + scoreHistory
  Line 448-10798: dashboard.tests + .pass + .tools + ...  ← 巨大的冗余数据
  Line 10800-10802: lastUpdated / source
```

当你需要"在 P4 的 nodes 末尾加一个新节点"时，你必须：

1. 在这 10802 行中找到 P4 的位置（line 133-173）
2. 找到 P4 的 nodes 数组的最后一个元素
3. 在它之后、`]` 之前插入新节点

### 问题

指南只说"追加到所属 Phase 的 nodes 数组末尾"，没有给出：

- 如何用 `search_content` 定位 P4
- 插入时 `old_str` 应该是什么（最后一个 node 的完整 JSON 还是 `],`？）
- `replace_in_file` 的具体参数

### 影响程度

🔴 **严重** — 这是场景 A 最频繁的操作，指南在这步失效。

### 修复方向

给出可复制的操作序列，如：


## 不足 6："在 `_utils` 之前插入" 在 `replace_in_file` 下不可精确操作

### 证据

```json
// test-case-mapping.json 的 modules 对象结构:
"modules": {
  "f1": {...},
  "f2": {...},
  // ... 30+ 个条目 ...
  "b4d": {...},       ← 最后一条业务模块, 内容可能变化
  "_utils": {...}     ← 要在这之前插入
}
```

### 问题

用 `replace_in_file` 插入新模块到 `_utils` 之前，需要精确的 `old_str`。

两种方式都有缺陷：

1. `old_str = '"b4d": {...},'` — 但 b4d 的内容未来会变（status 从 pending→done）
2. `old_str = '"_utils":'` — 这样新条目会**替换** `_utils` 而不是插在它前面

JSON 对象的键序在规范中是无序的（虽然 JS 引擎通常保持插入序）。一旦有人重格式化文件，`_utils` 可能不在最后。

### 影响程度

🔴 **严重** — 这是添加新模块最常用的操作，指南在这步失效。

### 修复方向

改为"在 modules 最后一个非 `_` 前缀的键之后插入"，并给出定位最后一个业务键的方法：

```bash
# 找最后一个非 _utils 的模块 key
node -e "const d=require('./test-case-mapping.json');
  const keys=Object.keys(d.modules).filter(k=>!k.startsWith('_'));
  console.log('LAST_KEY:', keys[keys.length-1]);"
```

# 附录 B：pipeline-data.json 肥大问题分析与拆分方案

> **分析日期**: 2026-06-27
> **问题源头**: 文档 `43` 中涉及的所有修改操作都在一个 10802 行的巨型 JSON 文件中进行。

## B.1 当前体量

| 指标     | 值         |
| -------- | :--------- |
| 总行数   | **10,802** |
| 文件大小 | **330 KB** |
| 编码     | UTF-8      |

## B.2 各区段行数占比

```text
pipeline-data.json
├── pipeline.phases[]              ~340 行  (3.1%)  ← 6 phases, 19 nodes
├── pipeline.branches[]            ~145 行  (1.3%)  ← 4 branches, 14 nodes
├── kpi                            ~55 行   (0.5%)  ← _scoreDimensions 细节

│   ├── dashboard.tools             ~300 行
```

## B.3 肥胖根因分析

### 根因：`TEST_LEVEL3_MAP` 硬编码测试详情

`pipeline-config.mjs` 第 761 行开始，有一个 800+ 行的 `TEST_LEVEL3_MAP` 常量：

```js
const TEST_LEVEL3_MAP = {
  // F1: 多模型适配器
  'DeepSeek 适配': [
    { label: '流式对话测试', val: '4' },
    { label: 'Function Calling', val: '3' },
    // ... 5-6 条 per 适配器, 10 个适配器 = ~50 条
  ],
  // F2: Agent 系统 ... 5 个子模块 × 4-5 条 = ~25 条
  // ... 每个功能模块都有类似结构 ...
};
```

`generateTestItems()` 函数将这份硬编码数据展开为 4-5 层嵌套的 JSON 树：

```
Phase 级条目 (label: "f1", val: "54 tests")
  └── Level2 子模块 (label: "DeepSeek 适配", val: "18")
        └── Level3 测试分类 (label: "流式对话测试", val: "4")
              └── Level4 测试用例 (label: "SSE 流式响应完整性", val: "✅")
```

每个叶子节点序列化成 JSON 约 4-5 行，33 个模块 × 平均 5 个子模块 × 5 个测试分类 × 4 个用例 = **约 3300 个叶子节点** → **约 9000+ 行 JSON**。

### 次要根因：`dashboard` 中还嵌入了工具列表、模型列表等展示数据

这些同样是硬编码在 `pipeline-config.mjs` 中的常量展开。

## B.4 消费关系分析

```text
pipeline-data.json 的所有消费者:

生产者:
  scripts/unified-sync.mjs          → 调用 generateDashboardDetails() 生成 dashboard 段
                                      → 调用 MODULE_REGISTRY 生成 pipeline 段

消费者:
  scripts/update-progress.mjs       → 只读/写 pipeline/kpi/scoreHistory
                                      → 不碰 dashboard 段
  docs/pipeline/_check_data.mjs     → 只读 dashboard.tests.items（调试用）
  docs/pipeline/lib/pipeline-config.mjs → 不直接读此文件（是数据源）

  应用包 (server/core/frontend/desktop):
    → 完全不读 pipeline-data.json
    → 这是纯文档产物，非运行时依赖
```

**关键发现**：`pipeline-data.json` 不是应用运行时依赖，修改其结构不影响应用功能。

## B.5 拆分方案

### 方案 A：按"可变/不可变"拆分为两个文件

```
pipeline-data.json              (~500 行)  ← 频繁修改的核心数据
├── pipeline.phases[]           (19 nodes, ~340 行)
├── pipeline.branches[]         (14 nodes, ~145 行)
├── kpi                         (~55 行)
└── scoreHistory[]              (~50 行)

pipeline-dashboard.json         (~10,200 行) ← 几乎不变的展示数据
└── dashboard                   (tests/pass/tools/models/score/modes/progress/issues)
```

**优点**：

- pipeline-data.json 从 10802→500 行，编辑/搜索/定位瞬间完成
- dashboard 数据几乎不变（硬编码常量为数据源），拆分后 Git diff 大幅减少
- 模块添加场景（场景 A/B）只需改 500 行文件

**缺点**：

- 需更新 2 个消费脚本（`update-progress.mjs`、`_check_data.mjs`）
- 多维护一个文件路径

**改动点**：

1. `unified-sync.mjs` — 输出流拆分为两个 `writeFileSync`
2. `update-progress.mjs` — 只读写 `pipeline-data.json`（已不碰 dashboard，无需改）
3. `_check_data.mjs` — 读取来源改为 `pipeline-dashboard.json`

---

### 方案 B：从 JSON 中删除 dashboard 段，按需动态生成（推荐）

```
pipeline-data.json              (~500 行)  ← 只保留核心数据
├── pipeline.phases[]
├── pipeline.branches[]
├── kpi
├── scoreHistory[]
└── lastUpdated / source

dashboard 数据 ← 不存磁盘，generateDashboardDetails() 按需生成
```

**原理**：`dashboard` 数据的生成逻辑已在 `pipeline-config.mjs` 中，数据源（`TEST_LEVEL3_MAP` 等）是硬编码常量。将 dashboard 写入 JSON 只是**序列化了一份冗余副本**。

**改动点（仅 1 处）**：

```diff
// scripts/unified-sync.mjs
const pipelineData = {
  pipeline: { phases, branches },
  kpi: mappingKPI,
  scoreHistory,
- dashboard: rawDashboard,
  lastUpdated: new Date().toISOString(),
  source: 'module-registry.mjs (统一注册表同步)',
};
```

`_check_data.mjs` 改为直接调用 `generateDashboardDetails()` 函数而非读 JSON 文件。

**优点**：

- pipeline-data.json 缩减至 ~500 行（缩减 **95%**）
- 零数据冗余 — dashboard 变化自动同步（因为动态生成）
- 不需要维护额外的 dashboard JSON 文件
- 改动量最小（1 行删除 + 1 个调试脚本调整）

**缺点**：

- 失去"离线查看 dashboard 数据"的能力（但脚本可重新生成）
- `_check_data.mjs` 需做简单调整

---

## B.6 推荐方案：方案 B

**理由**：

1. `dashboard` 数据是纯展示数据，不做业务逻辑判断
2. 生成逻辑已在代码中完整定义（复用优于复制）
3. 10000+ 行冗余消除比拆分成两个文件更彻底
4. **改动量最小**：1 行删除 + 1 个调试脚本调整
5. 零数据一致性风险 — 不存在 "pipeline-data.json 的 dashboard 过期了但没更新" 的情况

3. 重新运行 `node scripts/unified-sync.mjs --force` 生成精简后的 pipeline-data.json
4. 确认 pipeline-data.json 从 10802 行降至 ~500 行
5. 运行校验脚本确认 JSON 格式正确
