## 第一章：问题全景

### 1.1 触发背景

v0.6.17 发布后，计划发布 v0.6.18 以修复 `eslint.config.cjs` 中的 scoped 包路径 bug。发版过程中先后暴露两个独立问题。v0.6.20 发版时暴露了第三个问题——管线数据文件进入 release commit 导致 rebase 冲突。修复后 CI 运行又暴露了第四个和第五个问题。

| # | 问题 | 暴露版本 | 严重度 |
|---|------|---------|--------|
| 1 | `eslint.config.cjs` 的 `pnpmRequire` 无法找到 `@eslint/js` | v0.6.18 CI | 🟡 不阻塞（`continue-on-error: true`） |
| 2 | tag push 后 `release.yml` 没有自动触发 | v0.6.18 发版 | 🔴 阻断发版 |
| 3 | `git pull --rebase` 时管线数据文件冲突 | v0.6.20 发版 | 🔴 阻断发版 |
| 4 | release.yml sync-pipeline `git checkout main` 失败 | v0.6.20 CI | 🟡 sync 失败但不阻塞 Release 构建 |
| 5 | eslint 核心依赖（`@eslint/js` 等）从未在 package.json 中声明 | 长期隐藏，v0.6.20 CI 暴露 | 🟡 不阻塞（`continue-on-error: true`） |

### 1.2 问题之间的关联

问题① 是 D 方案之前就存在的隐藏 bug，被 `continue-on-error: true` 掩盖；v0.6.17 发版时为了修复问题②，引入了 `[skip ci]` 标记——结果 `[skip ci]` 自身又成了**问题② 的根因**。

**问题③ 与问题② 的关联**：问题② 修复去掉了 `[skip ci]`，导致 release commit 不再抑制 CI。post-commit hook 会在每次 commit 后修改管线 JSON 产生脏数据，而 `git add .` 把这些脏数据一起提交到了 release commit 中，与 CI auto-sync 推送的管线数据版本不同 → rebase 冲突。**三个问题共享同一段触发链（`release.mjs` 的发版流程）**。

**问题④ 与问题② 的关联**：问题② 修复去掉 `[skip ci]` 后 release.yml 正常触发，但 `actions/checkout@v4` 在 `push: tags` 事件下默认检出 tag → detached HEAD。sync-pipeline job 中 `git checkout main` 因本地无 `main` 分支引用而失败。**修复：checkout 步骤直接指定 `ref: main`**。

**问题⑤ 与问题① 的关联**：问题① 修复了 `pnpmRequire` 的路径解析逻辑，但 eslint 能运行的前提是 `eslint`、`@eslint/js`、`typescript-eslint` 三个包实际被安装。这三包从未在 `package.json` 中声明——本地能工作是因为 pnpm hoisting 侥幸解析到，但 CI 严格隔离环境下完全不可用。**问题① 修的是"能找到但路径不对"，问题⑤ 修的是"根本没安装"**。

### 1.3 涉及文件

| 文件 | 修改内容 |
|------|---------|
| `packages/frontend/eslint.config.cjs` | 问题① 修复：`pkgName.split('/').pop()` → `pkgName`；后续根治：双策略回退（createRequire + .pnpm 扫描） |
| `scripts/release.mjs` | 问题② 修复：去掉 commit 中的 `[skip ci]`；问题③ 修复：`git add .` 前先 `git checkout HEAD -- docs/pipeline/` |
| `.github/workflows/release.yml` | 问题④ 修复：checkout 步骤加 `ref: main`，避免 tag checkout → detached HEAD → `git checkout main` 失败 |
| `package.json` + `pnpm-lock.yaml` | 问题⑤ 修复：添加 `eslint`, `@eslint/js`, `typescript-eslint` 到 root devDependencies |

## 第二章：问题① —— eslint.config.cjs scoped 包路径解析错误

### 2.1 错误内容

```
pnpmRequire: 找不到包 @eslint/js
```

CI 的 `Lint & Format Check` job 中 `npx eslint` 配置加载阶段报错，但 job 本身显示 `success`（因为 `continue-on-error: true`）。

### 2.2 根因分析

**触发链**：

```
D方案引入 _test.yml 可复用 workflow
  → ci.yml 引用 _test.yml 的 tests job
  → lint-and-format job 在 windows-2022 runner 上执行
  → eslint 加载 eslint.config.cjs
  → pnpmRequire('@eslint/js') 找不到包 ❌
```

**根本原因**：`eslint.config.cjs` 第 26 行（修复前）：

```javascript
// ❌ 错误：对 scoped 包处理错误
foundDir = path.join(pnpmDir, dirent.name, 'node_modules', pkgName.split('/').pop());
```

| 包名 | `.split('/').pop()` | 拼接后的路径 | 是否正确 |
|------|---------------------|-------------|---------|
| `@eslint/js` | `"js"` | `.../@eslint+js@10.0.1_.../node_modules/js/` | ❌ 不存在 |
| `typescript-eslint` | `"typescript-eslint"` | `.../typescript-eslint@8.0.0_.../node_modules/typescript-eslint/` | ✅ 正确 |

scoped 包的 `node_modules` 下是 `@eslint/js` 两级目录结构，`.pop()` 只取了 `js`，丢失了 `@eslint` scope 目录。

### 2.3 修复方案

```javascript
// ✅ 修复后：直接使用完整 pkgName，path.join 自动处理跨平台分隔符
foundDir = path.join(pnpmDir, dirent.name, 'node_modules', pkgName);
```

同时补充了代码注释说明 pnpm 目录命名格式。

### 2.4 为什么之前没暴露

D方案之前 CI 直接运行 `ci.yml` 中的 eslint（而非通过 `_test.yml` 引用），环境差异（ubuntu vs windows runner）或 pnpm 解析行为可能导致 `@eslint/js` 能被全局 `node_modules` 找到。D方案上线后 runner 环境变化，`pnpmRequire` 才被实际执行并暴露 bug。

### 2.5 后续优化建议

**注意**：问题① 仅修复了路径解析逻辑，后续发现问题⑤ — 这三个包从未在 `package.json` 中声明 → CI 环境根本未安装 → 任何路径解析策略都无效。详见[第九章](#第九章问题--eslint-核心依赖从未在-packagejson-中声明)。

`lint-and-format` job 的 `continue-on-error: true`（第 143 行）掩盖了此错误。虽然历史 lint 债务存在，但**配置加载失败**与**业务代码 Warning** 是不同性质的错误。建议：
- 将 eslint 配置加载失败的退出码与 lint warning 的退出码分开处理
- 或者在 job 开头单独验证 `node -e "require('./eslint.config.cjs')"`，配置加载失败时让 job 失败

## 第三章：问题② —— [skip ci] 双重抑制导致 Release workflow 不触发

### 3.1 错误表现

v0.6.18 tag (`v0.6.18`) push 到 GitHub 后，`release.yml` **没有自动触发**。`gh run list` 只显示常规 push 触发的 CI，没有 tag 触发的 Release。

### 3.2 根因分析

**GitHub Actions 的 `[skip ci]` 规则**：

> 如果 push 事件的 HEAD commit 的 commit message 中包含 `[skip ci]` / `[ci skip]` / `[no ci]` 等关键字，GitHub Actions 会跳过**该次 push 事件关联的所有 workflow**，包括：
> - commit push 触发的 workflow（`push: branches`）
> - tag push 触发的 workflow（`push: tags`） —— **如果 tag 指向的 commit 含 `[skip ci]`**

**v0.6.18 的触发链**：

```
release.mjs 生成 commit → "release: v0.6.18 [skip ci]"
  → git commit + git tag v0.6.18（tag 指向该 commit）
  → 分两次 push: git push origin main → git push origin v0.6.18
  → tag push 事件: HEAD commit = "release: v0.6.18 [skip ci]"
  → GitHub 检测到 [skip ci] → 跳过 release.yml ❌
```

**关键误区**：之前认为"分两次 push 能绕过 `[skip ci]` 对 release.yml 的抑制"——这是错误的。`[skip ci]` 检查的是**tag 所指向的 commit 的 message**，与 push 的次数无关。

### 3.3 v0.6.17 为什么也失败了

回顾历史：v0.6.17 发布时 `release.yml` 也没有自动触发，当时**手动 `workflow_dispatch`** 触发了 Release。当时错误地将其归因于"没有分两次 push"，修复为分两次 push + 保留 `[skip ci]`。现在确认：**v0.6.17 的 release.yml 不触发同样是 `[skip ci]` 双重抑制**，分两次 push 并没有解决问题。

### 3.4 修复方案

```diff
- // [skip ci] 避免 CI workflow 为发版 commit 重复跑
- const commitMsg = `release: v${targetVersion} [skip ci]\n\n${changelogSummary}`;
+ // 注意：不能加 [skip ci]，因为 tag 指向的 commit 含 [skip ci] 会导致 GitHub 连 tag 触发的
+ // release.yml 也一并跳过。ci.yml 重复跑一次无害，Release workflow 能正常触发才关键。
+ const commitMsg = `release: v${targetVersion}\n\n${changelogSummary}`;
```

同时简化 push 注释（去掉关于 `[skip ci]` 能选择性抑制的错误叙述）：

```diff
- // 分两次 push，避免 [skip ci] 抑制 tag 触发的 release.yml
- // 第一次 push commit（含 [skip ci] → 跳过 ci.yml）
- // 第二次单独 push tag（无 [skip ci] → 触发 release.yml）
+ // 分两次 push：先 commit 再 tag
+ // 第二次 push tag 会触发 release.yml（基于 push: tags: ['v*'] 事件）
```

### 3.5 副作用评估

| 影响 | 评估 |
|------|------|
| ci.yml 在发版 commit 上多跑一次 | ✅ 可接受，约 5 分钟，不做额外工作（管线数据无变化） |
| Release workflow 正常触发 | ✅ 目标达成 |
| 其他 workflow 被误抑制 | ✅ 不存在，去掉 `[skip ci]` 后所有 push 事件正常触发 |

### 4.2 Rebase 冲突处理

发版期间（约 3 分钟），远程 CI 的 `sync-pipeline` job 完成并推送了一个管线数据 commit。本地 `git pull --rebase` 时在 4 个管线文件上冲突：

| 文件 | 处理方式 |
|------|---------|
| `docs/pipeline/_issue_cache.json` | 用远程版本 |
| `docs/pipeline/issue-data.json` | 用远程版本 |
| `docs/pipeline/pipeline-data.json` | 用远程版本 |
| `docs/pipeline/test-case-mapping.json` | 用远程版本 |

**决策依据**：管线数据是全量生成的（`unified-sync.mjs`），远程 CI 先完成 → 其数据包含最新的测试报告 + 本地发版 commit 没有修改源代码 → 远程数据是最完整的。

### 4.3 Post-commit Hook 干扰

每次 `git commit` 后，`post-commit` hook 自动运行 `unified-sync.mjs`，修改 `docs/pipeline/*.json`。这导致 commit → 脏文件 → 无法 rebase 的死循环。

**处理方法**：
- 管线文件产生后立即 `git checkout HEAD -- docs/pipeline/` 回退
- `git pull --rebase` 前 stash 所有修改
- 这些步骤已固化在 `scripts/release.mjs` 的推送前逻辑中

## 第六章：经验教训

### 6.1 [skip ci] 使用规则

| 场景 | 能否使用 [skip ci] | 原因 |
|------|-------------------|------|
| 管线数据自动同步 commit | ✅ 可以 | CI 不会为此 commit 再次 trigger pipeline sync |
| 发版 commit（需要 tag 触发的 workflow） | ❌ 禁止 | tag 指向的 commit 含 `[skip ci]` → 连 tag 触发的 workflow 一起抑制 |
| 日常文档/配置修改 commit | ✅ 可以 | 不需要触发 CI |

### 6.2 continue-on-error 使用警告

`continue-on-error: true` 会掩盖**配置加载失败**这类严重错误。建议对关键步骤（如 eslint 配置加载、依赖安装）单独验证，配置错误时 fail-fast。

### 6.3 发版流程检查清单

发版后应立即验证：
1. `gh run list --workflow=release.yml --limit 3` → 确认有新的 Release run
2. `gh run list --limit 5` → 确认 tag push 没有被 `[skip ci]` 抑制
3. 如果 Release 未触发 → 立即检查 commit message 是否含 `[skip ci]`

### 6.4 分两次 push 的真正作用

分两次 push（先 commit 再 tag）**不是为了绕过 `[skip ci]`**——它实际上绕不过。其真正作用是：
- 确保 commit 先到达远程（tag 指向有效的 commit）
- 避免 `--follow-tags` 导致的一次性 push 中 tag 先于 commit 到达的时序问题
- 使日志更清晰：哪个 push 触发了哪个 workflow

### 6.5 修复后的工作流全景 (v0.6.20)

```
release.mjs (修复后)
  ├─ git checkout HEAD -- docs/pipeline/ (恢复 hook 遗留脏文件) ← 新增
  ├─ git add . (此时管线文件已恢复到 HEAD 状态)
  ├─ commit "release: vX.Y.Z" (无 [skip ci]，无管线数据 ✅)
  ├─ git tag vX.Y.Z
  ├─ git checkout HEAD -- docs/pipeline/ (清理 commit 后 hook 新产生的变动)
  ├─ git pull --rebase origin main (管线数据一致 → 不会冲突 ✅)
  ├─ git push origin main (commit push → 触发 ci.yml + sync-pipeline)
  └─ git push origin vX.Y.Z (tag push → 触发 release.yml ✅)
```

### 6.6 [skip ci] 使用规则补充

| 场景 | 能否使用 [skip ci] | 原因 |
|------|-------------------|------|
| 管线数据自动同步 commit (CI) | ✅ 可以 | CI 不会为此 commit 再次 trigger pipeline sync |
| release-publish.bat Step 7 结尾 artifacts commit | ✅ 可以 | 纯产物记录，不需要触发 CI |
| **发版 commit（需要 tag 触发的 workflow）** | ❌ **禁止** | tag 指向 commit 含 `[skip ci]` → tag 触发 workflow 一起抑制 |
| 日常文档/配置修改 commit | ✅ 可以 | 不需要触发 CI |

### 6.7 actions/checkout 在 tag push 事件下的行为

`actions/checkout@v4` 在不指定 `ref` 时，默认检出触发事件对应的 Git 引用：
- `push: branches` → 检出分支（正常 HEAD）
- `push: tags` → 检出 tag → **detached HEAD**

如果后续步骤需要 `git checkout <branch>` 或 `git push`，必须显式指定 `ref: main`（或其他目标分支）。

### 6.8 依赖声明的完整性检查

**教训**：本地能运行 ≠ 依赖声明正确。pnpm 的 hoisting 可能让未声明的依赖通过其他包的 peer dependency 被"偶然安装"。排查方法：

1. 检查所有 `require()` / `import` 的来源是否在 `package.json` 中声明
2. 在 CI 环境中验证（或使用 `pnpm install --frozen-lockfile` 严格模式）
3. 对于 scoped 包（如 `@eslint/js`），特别注意路径解析的跨平台兼容性

## 第七章：问题③ —— 管线数据文件进入 release commit 导致 rebase 冲突 (v0.6.20)

### 7.1 错误表现

执行 `release-publish.bat` 时，在 Step 3 (version bump) 的 `git pull --rebase origin main` 阶段报错：

```
Auto-merging docs/pipeline/_issue_cache.json
CONFLICT (content): Merge conflict in docs/pipeline/_issue_cache.json
Auto-merging docs/pipeline/issue-data.json
CONFLICT (content): Merge conflict in docs/pipeline/issue-data.json
Auto-merging docs/pipeline/pipeline-data.json
CONFLICT (content): Merge conflict in docs/pipeline/pipeline-data.json
Auto-merging docs/pipeline/test-case-mapping.json
CONFLICT (content): Merge conflict in docs/pipeline/test-case-mapping.json
error: could not apply 57cfcd0... release: v0.6.20
```

4 个管线数据文件全部冲突，导致发版流程中断。

### 7.2 表象 vs 根因

**表象**（最初错误判断）：用户之前处于 rebase 中间状态，管线文件本地/远程都有修改。

**真根因**：release.mjs 使用 `git add .`，将 post-commit hook 遗留的脏管线文件纳入了 release commit，与 CI auto-sync 推送的管线数据版本不同。

### 7.3 完整调用链分析

#### 阶段一：脏文件产生

```
日常开发中某次 git commit
  └─ post-commit hook 触发
       └─ scripts/update-progress.mjs
            └─ unified-sync.mjs --force
                 └─ 修改 docs/pipeline/_issue_cache.json
                 └─ 修改 docs/pipeline/issue-data.json
                 └─ 修改 docs/pipeline/pipeline-data.json
                 └─ 修改 docs/pipeline/test-case-mapping.json
                 └─ ❌ 4个文件留在工作区成为脏文件（未被提交）
```

#### 阶段二：脏文件被吞入 release commit

```
用户运行 release-publish.bat
  └─ release.mjs (Step 3)
       ├─ git add .                           ← ❌ 脏管线文件被暂存
       ├─ git commit "release: v0.6.20"       ← ❌ 管线数据进入 release commit
       │   └─ post-commit hook 再次触发！
       │        └─ unified-sync --force 又改了管线文件
       ├─ git restore docs/pipeline/          ← ⚠️ 只清理工作区，commit 里已包含！
       └─ git pull --rebase origin main       ← 💥 冲突！
            release commit 的管线数据 ≠ origin/main 的管线数据（CI auto-sync 推送）
```

#### 关键点：`git restore docs/pipeline/` 为什么无效

原代码第 372 行的 `git restore` 是一个**补救措施**，它只能清理 post-commit hook 在 commit **之后**产生的新脏文件。但管线数据**已经进入 commit 了**，`git restore` 改的是工作区，不影响 commit 内容。rebase 比较的是**两个 commit 的管线数据差异**，所以必然冲突。

### 7.4 设计层面的问题分析

这不仅仅是代码 bug，更是**设计层面的缺陷**——两个 Writer 写同一组 Git 跟踪文件：

```
                    docs/pipeline/*.json (Git 跟踪)
                              │
            ┌─────────────────┼─────────────────┐
            ▼                                    ▼
    本地 post-commit hook                    CI sync-pipeline job
    (每次 commit 后修改)                    (每次 CI 后 push)
            │                                    │
            ▼                                    ▼
    工作区变脏，                 推送到 origin/main
    被 git add . 误吞
```

三个设计缺陷：

| 层面 | 问题 | 严重度 |
|------|------|--------|
| **架构层** | 自动生成的数据产物 (`_issue_cache.json` 等) 不应该被 Git 跟踪。应当跟源码分离存储（artifact/CDN/数据库） | 🔴 架构级 |
| **Hook 层** | post-commit 修改被跟踪文件后没有自清理机制，脏文件留在工作区成为"地雷" | 🟡 工程级 |
| **工具层** | `release.mjs` 的 `git add .` 过于激进，应该使用白名单而非通配 | 🟡 工程级 |

### 7.5 修复方案

#### 修复位置：`scripts/release.mjs` 第 359-381 行

```javascript
// ✅ 修复后
const commitMsg = `release: v${targetVersion}\n\n${changelogSummary}`;

// ⚠️ 关键：管线数据文件由 CI sync-pipeline job 自动维护，绝不能被 release commit 包含
// 否则与 origin/main 的 CI 自动推送数据冲突，导致 rebase 失败。
// post-commit hook（update-progress.mjs → unified-sync.mjs）会在每次 commit 后修改
// 这些文件，形成脏数据。必须先恢复到 HEAD 状态，阻止脏管线数据进入 commit。
execSync('git checkout HEAD -- docs/pipeline/', { cwd: root, stdio: 'pipe' });  // ← 新增
execSync('git add .', { cwd: root, stdio: 'inherit' });
execSync(`git commit -m "${commitMsg.replace(/"/g, '\\"')}"`, { cwd: root, stdio: 'inherit' });

// 回退 post-commit hook 本次触发可能产生的管线文件修改
try {
  execSync('git checkout HEAD -- docs/pipeline/', { cwd: root, stdio: 'pipe' });  // ← 改用 checkout
} catch { /* 没有管线文件修改则跳过 */ }
// 推送前先 rebase 远程（CI 管线同步可能在此期间推了新 commit）
// 由于管线数据不再包含在 release commit 中，rebase 不会冲突
execSync('git pull --rebase origin main', { cwd: root, stdio: 'inherit' });
```

**两个关键改动**：

| 位置 | 修改前 | 修改后 | 效果 |
|------|--------|--------|------|
| `git add .` 之前 | 无操作 | `git checkout HEAD -- docs/pipeline/` | **阻止脏管线文件进入 commit** |
| commit 之后 | `git restore docs/pipeline/` | `git checkout HEAD -- docs/pipeline/` | 恢复到 HEAD 状态（更可靠） |

原代码只有 commit 之后的 `git restore`，治标不治本。新代码在 `git add .` **之前**先恢复管线文件，确保管线数据不会进入 release commit。

### 7.6 修复后的效果验证

修复后的 commit 内容：

```
修复前 release commit (v0.6.20):
  21 files changed    ← 包含 4 个管线 JSON

修复后 release commit (预期):
  ~17 files changed   ← 仅含版本变更 + CHANGELOG，不含管线 JSON
```

rebase 行为：

```
修复前:
  release commit (含管线数据 v1) → rebase on origin/main (管线数据 v2)
  → CONFLICT 💥  因为两边都修改了同一组文件

修复后:
  release commit (不含管线数据) → rebase on origin/main
  → Fast-forward / clean merge ✅  管线文件不再冲突
```

### 7.7 架构级根治方案（未来）

当前修复是**工程层面**的，阻止了管线数据进入 release commit。但**架构层面的问题**尚未解决：

只要 `docs/pipeline/*.json` 还在 Git 仓库中被追踪，任何人本地运行 `unified-sync` 后 pull 远程都可能遇到管线数据冲突。根治方案：

| 方案 | 改动范围 | 风险 |
|------|---------|------|
| A. 管线 JSON 加入 `.gitignore`，CI 以 artifact 上传 | 中等（需改 CI、管线页面加载逻辑） | 本地开发无法查看管线数据 |
| B. 管线 JSON 不再被 CI push，仅作为 CI artifact + 本地生成 | 小（改 CI sync-pipeline job） | 历史管线数据丢失 |
| C. 使用独立 Git 仓库存储管线数据（submodule 或独立 repo） | 大 | 增加仓库管理复杂度 |

**建议**：当前修复已能在不改架构的情况下避免 release 冲突。如果未来同类冲突在其他场景出现，优先考虑方案 A（`.gitignore` + artifact）。

### 7.8 v0.6.20 冲突解决过程

由于在修复代码前，冲突已经发生，当时使用了以下应急处理：

```bash
git checkout --ours docs/pipeline/_issue_cache.json
git checkout --ours docs/pipeline/issue-data.json
git checkout --ours docs/pipeline/pipeline-data.json
git checkout --ours docs/pipeline/test-case-mapping.json
git add docs/pipeline/_issue_cache.json docs/pipeline/issue-data.json \
        docs/pipeline/pipeline-data.json docs/pipeline/test-case-mapping.json

# post-commit hook 导致 rebase --continue 超时
ren .git\hooks\post-commit post-commit.bak
git rebase --continue
ren .git\hooks\post-commit.bak post-commit

# tag 指向旧 commit，需要迁移
git tag -f v0.6.20 HEAD
git push origin main && git push origin v0.6.20
```

## 第八章：问题④ —— release.yml sync-pipeline 的 `git checkout main` 失败

### 8.1 错误表现

v0.6.20 Release workflow 运行后，`sync-pipeline-data` job 报错：

```
fatal: not currently on a branch
error: pathspec 'main' did not match any file(s) known to git
```

job 失败但不阻塞 Release 构建（`continue-on-error: false` 会导致整个 job 标红，但不影响前面的 Build Web / Build Desktop / Publish 等步骤）。

### 8.2 根因分析

**触发链**：

```
release-publish.bat 推送 tag v0.6.20 到 GitHub
  → push: tags: ['v*'] 事件触发 release.yml
  → sync-pipeline-data job 中:
      actions/checkout@v4 (无 ref 参数)
        → GitHub push: tags 事件 → 默认检出 tag v0.6.20
        → 进入 detached HEAD 状态（HEAD 指向 v0.6.20，不在任何分支上）
      git checkout main
        → 本地仓库从 tag 检出，没有 main 分支引用
        → fatal: pathspec 'main' did not match ❌
```

**根本原因**：`actions/checkout@v4` 在 `push: tags` 事件下默认检出触发 tag（不是分支），导致进入 detached HEAD 状态。后续 `git checkout main` 因本地无 `main` 分支引用而失败。

**为什么之前没暴露**：问题② 期间 release.yml 被 `[skip ci]` 抑制，从未被 tag push 自动触发过。v0.6.19 首次自动触发时，可能因为需要 `git checkout main` 的 sync-pipeline job 是后续新增的，或当时使用了 `workflow_dispatch` 手动触发（此时 checkout 默认检出默认分支）。

### 8.3 修复方案

在 `actions/checkout@v4` 步骤中直接指定检出 `main` 分支，避免进入 detached HEAD：

```yaml
# ❌ 修复前
- uses: actions/checkout@v4
  with:
    token: ${{ secrets.GITHUB_TOKEN }}

# ✅ 修复后
- uses: actions/checkout@v4
  with:
    token: ${{ secrets.GITHUB_TOKEN }}
    ref: main                # 直接检出 main 分支，避免 tag checkout → detached HEAD
```

**文件**：`.github/workflows/release.yml` sync-pipeline job 第 388-390 行。

### 8.4 副作用评估

| 影响 | 评估 |
|------|------|
| sync-pipeline job 从 main 分支检出 | ✅ 正确行为，管线数据应基于 main 的最新状态 |
| tag 信息在 job 中是否可访问 | `github.ref` 仍为 `refs/tags/v0.6.20`，tag 信息通过 GitHub Actions 上下文获取，不受 checkout ref 影响 |
| 其他 job 受影响 | ✅ sync-pipeline 是 release.yml 中唯一需要 `git push` 的 job |

## 第九章：问题⑤ —— eslint 核心依赖从未在 package.json 中声明

### 9.1 错误表现

v0.6.20 CI 的 `lint-and-format` job 报错：

```
pnpmRequire: 找不到包 @eslint/js
```

与问题① 相同的错误信息，但**根因完全不同**。

### 9.2 根因分析

**问题① vs 问题⑤ 的区别**：

| 维度 | 问题① | 问题⑤ |
|------|--------|--------|
| 阶段 | 包已安装，但 `pnpmRequire` 路径解析错误 | 包未安装，根本不存在于 `node_modules` 中 |
| 根因 | `.split('/').pop()` 丢失 `@eslint` scope | `eslint`/`@eslint/js`/`typescript-eslint` 从未写入任何 `package.json` |
| 本地表现 | 可能偶然工作（取决于 pnpm hoisting 行为） | 本地可能工作（被其他 workspace 包的 peer dep 间接安装） |
| CI 表现 | 必然失败 | 必然失败 |

**本地为什么能工作**：

```
本地开发环境 (.npmrc 有 public-hoist-pattern):
  pnpm 的 hoisting 机制可能将 eslint 提升到根 node_modules
  其他 workspace 包（如 frontend）的 peer dep 可能间接触发安装

CI 环境（严格的 pnpm install --frozen-lockfile）:
  pnpm 严格按 lockfile 安装，不存在的依赖不会被安装
  → node_modules/@eslint/js/ 不存在
  → pnpmRequire 找不到包 ❌
```

**依赖缺失清单**：

`packages/frontend/eslint.config.cjs` 第 5-6 行需要以下 import：

```javascript
const tseslint = require('typescript-eslint');
const eslintJs = require('@eslint/js');
```

但三个包（`eslint` 作为 eslint 运行时应由 `npx eslint` 提供）**从未在任何 `package.json` 的 devDependencies 中声明**。

### 9.3 修复方案

在根 `package.json` 的 devDependencies 中添加三个包：

```json
{
  "devDependencies": {
    "@eslint/js": "^9.15.0",
    "eslint": "^9.15.0",
    "prettier": "^3.8.4",
    "typescript-eslint": "^8.15.0"
  }
}
```

**为什么放根 package.json**：`.npmrc` 已配 `public-hoist-pattern[]=*eslint*`，确保 eslint 相关包被提升到根 `node_modules/`，所有 workspace 包共享一份。

### 9.4 验证结果

```bash
# 本地验证 eslint 配置能否正确加载
$ node -e "require('./packages/frontend/eslint.config.cjs')"
✅ eslint config loaded successfully

# 确认依赖已安装
$ ls node_modules/eslint/package.json       → exists
$ ls node_modules/@eslint/js/package.json   → exists
$ ls node_modules/typescript-eslint/package.json → exists
```

### 9.5 为什么问题① 的修复不彻底

问题① 只修复了 `pnpmRequire` 的**路径解析逻辑**（用 `pkgName` 代替 `.pop()`），但没有解决**依赖未声明**问题。即使 v0.6.19 的 `pnpmRequire` 使用了双策略回退（createRequire + .pnpm 扫描），如果包根本不存在，任何策略都找不到。

**教训**：排查 "找不到包" 错误时，先确认包是否在 `package.json` 中声明，再排查路径解析。
