## 第一章：问题全景

### 1.1 触发背景

v0.6.13 发布后，GitHub Actions 同时触发了两个 workflow：
- **CI workflow**（`ci.yml`）—— main 分支 push 触发
- **Release workflow**（`release.yml`）—— `v0.6.13` tag push 触发

两个 workflow 各有两个 job 失败，共 4 个截图、4 个错误。

## 第二章：截图① —— Build Desktop GH_TOKEN 缺失

### 2.1 错误内容

```
Error: GitHub Personal Access Token is not set, neither programmatically, 
nor using env "GH_TOKEN"
    at PublishManager.artifactCreatedWithoutExplicitPublishConfig
    at throwError (asyncTaskManager.ts:88:11)
```

### 2.2 根因分析

`packages/desktop/package.json` 的 `build` 配置中定义了：

```json
"publish": {
  "provider": "github",
  "owner": "ht182400-creator",
  "repo": "easyagent"
}
```

当 `electron-builder --win --x64` 执行时，electron-builder 检测到 `publish` 配置，会尝试自动发布构建产物到 GitHub Release。但 CI 中的这一步**没有设置 `GH_TOKEN` 环境变量**，导致 electron-builder 无法认证。

**关键机制**：electron-builder 的默认行为是 **自动发布**，除非显式传递 `--publish never` 参数。CI 中调用的是 `npx electron-builder --win --x64`，没有 `--publish never`，也没有 `GH_TOKEN`，双重缺失导致报错。

### 2.3 修复方案

在 `release.yml` 的 Package 步骤中添加 `GH_TOKEN` 环境变量：

```yaml
- name: Package (electron-builder)
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}   # ← 新增
  run: |
    cd packages/desktop
    npx electron-builder --win --x64
```

### 2.4 为何之前没暴露

本地 `release-publish.bat` 中，`_release.mjs` 上传 EXE 使用的是独立的 GitHub API 调用（通过 Node.js `fs` + `https`），不经过 electron-builder 的发布机制。本地甚至 `electron-builder --win` 时也不会去发布（因为没有环境变量，electron-builder 只会警告但继续构建）。

**CI 与本地差异**：CI 中 electron-builder 检测到了 GitHub Actions 的运行环境（`GITHUB_ACTIONS=true`），触发了更严格的发布检查，**有 publish 配置就必须有 token**。

## 第三章：截图② —— Build Web react/jsx-runtime 解析失败

### 3.1 错误内容

```
error during build:
Error: [vite]: Rollup failed to resolve import "react/jsx-runtime" 
from "/home/runner/work/easyagent/easyagent/packages/frontend/src/mountApp.tsx".
If you do want to externalize this module explicitly add it to 
`build.rollupOptions.external`
```

### 3.2 根因分析

这是一个**依赖传递链断裂**问题，涉及三个包：

```
packages/web  ──────────→  @easyagent/frontend (workspace:*)
  vite build                  ├── react: ^18.3.0        ← 在这里定义了
  resolve alias @ →           └── react-dom: ^18.3.0
  ../frontend/src/
```

**关键链路**：
1. `packages/web/vite.config.ts` 配置了别名 `@: ../frontend/src`
2. Vite 构建时解析 `mountApp.tsx`，其中 `import React from 'react'` 和 JSX 编译后的 `react/jsx-runtime`
3. **但在 pnpm workspace 中**，Vite/Rollup 在 `packages/web` 上下文中解析 `react/jsx-runtime` 时，依赖的 node_modules 解析链是：
   - `packages/web/node_modules/` → 没有 react
   - `packages/web` 的 `pnpm` 依赖图中也没有 react（web 的 `dependencies` 只有 `@easyagent/frontend`）

4. 虽然 `react` 是 `@easyagent/frontend` 的依赖，但 **pnpm 的严格隔离机制**下，`packages/web` 无法直接访问 `packages/frontend/node_modules/` 中的 react 实例。

5. 本地开发时 `pnpm install` 可能会 hoist react 到根目录 `node_modules`，所以**本地 `vite build` 不报错**。但 CI 使用 `--frozen-lockfile`，解析更严格。

### 3.3 对比：Desktop 为什么没问题

| 对比项 | Desktop | Web |
|--------|---------|-----|
| react 依赖 | `packages/desktop/package.json` → `dependencies` ✅ | `packages/web/package.json` → **无** ❌ |
| 构建方式 | `vite build` + tsup | `vite build`（含 tsc） |
| 解析能力 | 有 react 在 deps，直接解析 | 依赖传递解析，失败 |

### 3.4 修复方案

在 `packages/web/package.json` 的 `dependencies` 中显式添加 `react` 和 `react-dom`：

```json
"dependencies": {
  "@easyagent/frontend": "workspace:*",
  "react": "^18.3.0",
  "react-dom": "^18.3.0"
}
```

### 3.5 设计原则

- **显式依赖原则**：如果包 A 直接 import 了包 B 的模块，那么包 A 应该显式声明 B 为依赖
- 即使是 workspace 别名引用源码，Vite/Rollup 的模块解析不一定能穿透 node_modules 边界
- 这条原则同样适用于 future 的 web 端增加任何其他直接 import 的第三方库

## 第四章：截图③④ —— Sync Pipeline unstaged 残留阻塞 rebase

### 4.1 错误内容（两次触发，同一错误）

```
error: cannot pull with rebase: You have unstaged changes.
error: please commit or stash them.
```

截图③ 出现在 **tag 触发**的 CI run，截图④ 出现在 **push 触发**的 CI run。

### 4.2 根因分析（三层递进）

#### 第一层：`git add` 文件列表不完整

ci.yml 的 `Commit pipeline data` 步骤原本使用了**显式文件列举**：

```bash
git add docs/pipeline/test-case-mapping.json
git add docs/pipeline/pipeline-data.json
git add docs/pipeline/issue-data.json
git add docs/pipeline/project-progress-data.json
git add docs/pipeline/.pipeline-cache.json
```

但 `scripts/unified-sync.mjs` 实际生成的文件列表**更大**：

| git add 覆盖 | unified-sync 实际产物 | 覆盖? |
|-------------|---------------------|------|
| `test-case-mapping.json` | ✅ | ✅ |
| `pipeline-data.json` | ✅ | ✅ |
| `issue-data.json` | ✅ | ✅ |
| `project-progress-data.json` | ✅ | ✅ |
| `.pipeline-cache.json` | ✅ | ✅ |
| — | `_issue_cache.json` | ❌ 遗漏 |
| — | `dashboard-data.json` | ❌ 遗漏 |

结果：`git commit` 只提交了 5 个文件，但 `_issue_cache.json` 和 `dashboard-data.json` 保持 **unstaged 修改状态**，阻碍了后续的 `git pull --rebase`。

#### 第二层：git hooks 干扰

即使在 `git add` 之后立即执行 `git commit`，post-commit hook 会自动运行 `unified-sync.mjs`，这会在 commit 后**立即生成新的文件差异**。时间线：

```
00:00.0  git add docs/pipeline/    # 暂存当前状态
00:00.1  git commit -m "..."       # 提交 → 触发 post-commit hook
00:00.2  [hook] unified-sync.mjs  # 重新扫描，可能产生新差异
00:00.5  git pull --rebase        # 💥 Unstaged changes!
```

#### 第三层：tag 和 push 两次触发导致并发竞态

v0.6.13 发版时：
- `release-publish.bat` push tag + main 分支
- GitHub 收到 `v0.6.13` tag → 触发 `release.yml`  
- GitHub 收到 main push → 触发 `ci.yml` → sync-pipeline job

两个 CI 的 sync-pipeline job 会在**同一时间窗口竞争 push**，进一步加剧 rebase 冲突。

### 4.3 修复方案（两步）

#### 步骤一：改用目录级 `git add`

```bash
# 旧：显式列举（易遗漏）
git add docs/pipeline/test-case-mapping.json
git add docs/pipeline/pipeline-data.json
# ... 5个文件

# 新：整个目录
git add docs/pipeline/
```

#### 步骤二：添加 stash 保护链

```bash
git commit -m "ci: auto-sync pipeline data [skip ci]"

# 推送前 stash → rebase 远程 → pop stash → push
git stash                     # 暂存 hook 产生的任何残留修改
git pull --rebase origin main # 拉取远程最新
git stash pop                 # 恢复本地修改
git push                      # 成功推送
```

### 4.4 修复范围

此竞态问题影响三个位置，全部修复：

| # | 位置 | 修复 |
|---|------|------|
| ① | `scripts/release.mjs` | `git stash` → `pull --rebase` → `stash pop` |
| ② | `release-publish.bat` | 同上 |
| ③ | `.github/workflows/ci.yml` | `git add docs/pipeline/` + `git stash` → `pull --rebase` → `stash pop` |

## 第五章：关联问题 —— 发版 push 被 rejected（竞态）

### 5.1 问题背景

与截图③④属于同一根因家族（参见 `docs/47_发版推送竞态条件分析与根治.md`）。

### 5.2 问题链路

```
release-publish.bat 执行（3-5分钟）
  ├── git tag v0.6.13 → push tag       # tag 触发 release.yml (CI)
  ├── ... 构建 EXE ... (2分钟)
  │     ← CI sync-pipeline push 管线数据到 main  ← 远程产生新 commit
  ├── git commit version/CHANGELOG
  └── git push origin main             # 💥 rejected! (远程多了一个 commit)
```

### 5.3 修复

同样使用 `git stash → pull --rebase → stash pop → push` 防护链。详见 `docs/47_发版推送竞态条件分析与根治.md`。

## 第七章：经验教训

### 7.1 CI 与本地环境的差异陷阱

> **永远不要假设 CI 环境与本地行为一致。**

| 差异点 | 本地 Windows | CI (ubuntu/windows) |
|--------|-------------|---------------------|
| pnpm 模块解析 | 可能 hoist 到根 node_modules | `--frozen-lockfile` 更严格 |
| electron-builder | 不检测 GITHUB_ACTIONS，跳过发布 | 检测到 GITHUB_ACTIONS=true，严格检查 token |
| git hooks | post-commit 自动运行 unified-sync | 同（CI ubuntu 也有 hooks） |
| 文件编码 | CRLF | LF |

### 7.2 显式依赖原则

**包 A 直接使用了包 B 的模块 → 包 A 必须在 `dependencies` 中声明包 B。**

即使通过 workspace 别名引用源码，也不能假设依赖传递会自动生效。这在 pnpm 的严格模式下尤其重要。

### 7.3 管线文件的添加必须同步到 CI

每当 `unified-sync.mjs` 新增输出文件（如 `dashboard-data.json`、`_issue_cache.json`），必须同步更新：
- `ci.yml` 的 `git add` 步骤
- 或直接改用 `git add docs/pipeline/`（目录级，一劳永逸）

### 7.4 electron-builder publish 配置的 CI 含义

`package.json` 中的 `build.publish` 配置在 CI 环境下会自动触发发布行为。如果不想要自动发布（已经通过独立的 Create Release step 处理），有两种方案：

| 方案 | 做法 | 优劣 |
|------|------|------|
| A. 设 GH_TOKEN | 加 `env: GH_TOKEN` | ✅ 简单，与独立 Release step 协同 |
| B. 禁自动发布 | `--publish never` | ⚠️ 如果本地需要用 electron-builder 发布则冲突 |

本项目选择方案 A，因为 `Create GitHub Release` step 是独立的发布流程，electron-builder 的 token 仅用于满足构建时的认证要求。

### 7.5 CI 修复验证流程

今后对 CI 配置的修改，建议按以下顺序验证：

1. **本地预检**：确认 `pnpm install --frozen-lockfile` 通过
2. **push 到 main**：触发 `ci.yml`，观察 Sync Pipeline 是否通过
3. **push tag**：触发 `release.yml`，观察 Build Desktop / Build Web 是否通过（或手动 dispatch）
4. **首次发版**：完整验证推送链路无竞态

## 第八章：快速排查手册

### 8.1 看到 `GH_TOKEN is not set`

```
→ 检查 .github/workflows/release.yml
→ Package (electron-builder) 步骤是否设置了 env: GH_TOKEN
→ 检查 packages/desktop/package.json 的 build.publish 配置
→ 确认 secrets.GITHUB_TOKEN 在仓库 Settings 中可用
```

### 8.2 看到 `resolve import "react/jsx-runtime" 失败`

```
→ 检查报错的包（如 packages/web）的 package.json
→ dependencies 中是否有 "react" 和 "react-dom"
→ 如果没有，添加之
→ 运行 pnpm install 更新 lockfile（如在 CI 则用 --frozen-lockfile 验证）
```

### 8.3 看到 `cannot pull with rebase: unstaged changes`

```
→ 检查 CI 的 git add 是否覆盖了所有 unified-sync 输出文件
→ 推荐改用 git add docs/pipeline/ 替代显式列举
→ 确保 push 前有 git stash → pull --rebase → stash pop 链
→ 检查 git hooks (post-commit) 是否在 commit 后产生新文件
```

### 8.4 看到 `[rejected] main -> main (non-fast-forward)`

```
→ 检查 scripts/release.mjs 和 release-publish.bat
→ 确保 push 前有 git stash → pull --rebase → stash pop 链
→ 如果仍失败：检查是否有未追踪的管线文件变化
→ 参考 docs/47_发版推送竞态条件分析与根治.md
```
