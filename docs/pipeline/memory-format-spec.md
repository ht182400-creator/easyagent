# Memory 目录文件记录规范 v1.0

**适用范围**: `.codebuddy/memory/` 目录下所有 `.md` 文件  
**生效日期**: 2026-06-23 起新编写的内容  
**目的**: 使 `server.mjs` 解析器能准确提取模块-问题-解决方案数据，支持增量缓存

---

## 1. 文件命名

| 文件            | 用途              | 更新策略                 |
| --------------- | ----------------- | ------------------------ |
| `YYYY-MM-DD.md` | 每日工作日志      | 追加式，**严禁覆盖删减** |
| `MEMORY.md`     | 长期记忆/项目约定 | 就地更新，保持精简       |

---

## 2. 问题记录格式（核心规范）

每个需要被管线页面索引的问题/修复，使用以下**标准化 Section 模板**：

```markdown
## [模块:F1] 简短标题 (HH:MM)

- **问题**: 清晰的问题描述（一句话即可）
- **根因**: 根因分析（可选，有则写）
- **修复**: 解决方案描述
- **状态**: ✅ resolved
```

### 2.2 字段说明

| 字段         | 必选    | 格式                                     | 说明              |
| ------------ | ------- | ---------------------------------------- | ----------------- |
| `- **问题**` | ✅ 必选 | `- **问题**: 描述`                       | 问题描述，1-2句话 |
| `- **根因**` | ⭕ 可选 | `- **根因**: 分析`                       | 技术根因分析      |
| `- **修复**` | ✅ 必选 | `- **修复**: 方案`                       | 解决方案说明      |
| `- **状态**` | ✅ 必选 | `✅ resolved` / `⏳ pending` / `❌ open` | 问题当前状态      |

### 2.3 完整示例

```markdown
# 2026-06-23

## [模块:F1] Kimi API URL 双重 `/v1` 前缀 (15:30)

- **问题**: Kimi API 请求报 404，URL 变成 `https://api.moonshot.cn/v1/v1/chat/completions`
- **根因**: `setApiKey()` 从预设取 baseURL 时拼接了第二层 `/v1` 后缀
- **修复**: 在 `ConfigManager.ts` 的 `setApiKey()` 方法中增加 URL 去重逻辑，检测已有 `/v1` 则不再追加
- **状态**: ✅ resolved
```

### 2.4 多问题 Section

一个 Section 可以包含多个问题（用 `###` 子标题分隔），每个子标题也应有模块标签：

```markdown
## [模块:F9] Desktop 打包问题集 (16:00)

### [模块:F9] asar 内 better-sqlite3 加载失败

- **问题**: 打包后的 EXE 启动报 `Cannot find module 'better-sqlite3'`
- **根因**: asar 归档内路径无法解析原生 .node 模块
- **修复**: `extraResources` 复制 better-sqlite3 到 asar 外
- **状态**: ✅ resolved

### [模块:F9] electron-updater 传递依赖缺失

- **问题**: 自动更新检查时报 `Cannot find module 'lodash.escaperegexp'`
- **修复**: 在 desktop/package.json 显式声明 8 个传递依赖
- **状态**: ✅ resolved
```

## 3. 不需要被索引的内容

以下类型的 `##` Section 不会被解析器索引（自动跳过）：

- 启动/重启类：`前后端启动`、`后端重启`、`前端启动`
- 文档类：`文档更新`、`MEMORY.md 更新`
- 发布类：`GitHub Release`、`发布`、`打包`、`编译`
- 验证类：`测试通过`、`构建验证`、`清理`
- 流程类：`workflow`、`CI/CD 运行`

这些 Section **无需**加 `[模块:ID]` 标签。

---

## 4. 存量文件处理

2026-06-22 及之前的 memory 文件**不需要**重新格式化。解析器对存量文件使用关键词匹配进行模块分类，结果会通过文件级缓存持久化。

新增的 `[模块:ID]` 标签优先级**高于**关键词匹配：如果 Section 标题中有标签，直接映射，不再做关键词猜测。

---

## 5. 缓存策略

| 文件日期             | 缓存行为       | 解析频率                                 |
| -------------------- | -------------- | ---------------------------------------- |
| **今天** (`current`) | 每次都重新解析 | 每次 `/api/issues` 请求                  |
| **昨天及之前**       | 按 mtime 判断  | mtime 不变 → 用缓存；mtime 变 → 重新解析 |
| **MEMORY.md**        | 按 mtime 判断  | 同上                                     |

缓存文件存储在 `docs/pipeline/.pipeline-cache.json`，可通过删除该文件强制全量重建。

---

## 6. AI 助手生成记录时的约定

当 AI 助手在 memory 文件中记录问题时，应遵循以下规则：

1. **每个问题/修复作为独立的 `## [模块:ID]` Section**，不要混在流程描述中
2. **模块 ID 必须准确**，参考第 2.1 节的模块表
3. **字段名严格一致**：`- **问题**:` / `- **根因**:` / `- **修复**:` / `- **状态**:`
4. **状态用 emoji 标记**：`✅ resolved` / `⏳ pending` / `❌ open`
5. **纯操作流程**（启动/构建/发布）**不加** `[模块:ID]` 标签
6. **非问题内容**（设计决策、架构讨论、规范更新）不放入此格式
