## 管线解析器深入修复 —— 消除"详见原文" (01:08-01:12)

### [模块:p5c] 解析器 solution 提取失败导致大量"详见原文"

- **问题**: 管线页面中大量条目显示"详见原文"（44/64 = 69%），而非具体解决信息；用户点击无法展开原文
- **根因**: 解析器的 solution 提取 regex 无法匹配多种实际使用的格式：
  1. **`### 修复内容`** 不被识别为 solution section（只认 "解决""解决方案""修复过程"）
  2. **`- **修复** (\`path\`):`** 格式中括号后有冒号，原 regex 要求 `**修复**:`（紧挨冒号）
  3. **`- **修复1**:`** / **`- **修复2**:`** 编号修复项不匹配 `**修复**`（多了数字）
  4. **`- **修复方案**:`** / **`- **正确方案**:`** / **`- **最终方案**:`** 冒号位置不兼容
  5. **`### 问题回顾`→`### 解决`** 过渡时 entry 被提前 push（只有 problem 无 solution）
  6. **`\b` 对中文字符无效**，导致 `### 解决` 的 section 识别完全失败（`解决\b` 始终为 false）
  7. **solution 块内 `- **key**: value`** 粗体键值对不捕获
- **修复**:
  1. solution section 模式扩展为 `解决|解决方案|修复(?:过程|内容|措施|方案)?|处理[方式方案]?`，用 `(?![一-龥\w])` 替代 `\b`
  2. `fixStart` regex 改为匹配任何含"修复/正确方案/核心方案/最终方案/解决"的粗体标签，允许括号内容：`\*\*(?:.*(?:修复|...).*)\*\*`
  3. `fixHeader` 同样扩展，识别标题行（无行内内容）进入 solution 块采集模式
  4. solution 块内新增 `- **key**: value` 粗体键值对捕获
  5. solution 块内新增 `**N. 描述**` 编号条目捕获
  6. `### ` 问题/解决子段之间不 reset `currentProblem`，`### 解决` 时不提前 push entry
  7. problem 模式新增 `- **非标准键**: value` 格式捕获（如"提供商不存在"等自定义字段）
- **状态**: ✅ resolved
- **效果**:
  - 总问题数: 39 → 64 → 87 → 143 → 189 (+385%)
  - "详见原文": 44 → 25 → 6 → 5 → **1** (-97.7%)
  - 覆盖模块: 13 → 22 → **21**（几乎所有模块都有问题记录）
  - 方案覆盖率: 31%→69%→91%→99.5%
