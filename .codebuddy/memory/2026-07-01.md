## [模块:lg1] LangGraph 普通聊天触发 benchmark 工具循环至死锁 (00:36)
- **问题**: 用户发送普通问候"再次你好"，LangGraph 引擎却反复调用 benchmark_report → benchmark_run → benchmark_load 工具，最终 `Recursion limit of 25 reached` 报错；前端显示红色错误气泡。
- **根因**: 四重缺陷叠加：1) 系统提示词未划定工具使用边界；2) AgentFactory 把 69 个工具全量下发（含 benchmark_* 评测工具）；3) observeNode 工具失败后无条件继续循环；4) LangGraph 默认 `recursionLimit=25` 按图步数计算，而 `maxTurns=25` 按轮次计算，导致 maxTurns 还未生效就被 LangGraph 截断。
- **修复**:
  - `packages/langgraph/src/Agent.ts`: 默认系统提示词增加"普通聊天不调用工具"等行为准则；`streamEvents` 运行时传入 `recursionLimit = maxTurns * 3 + 10`。
  - `packages/langgraph/src/bridge/AgentFactory.ts`: `getToolDefinitions` 默认过滤 `benchmark_*` 工具，避免普通聊天暴露评测工具。
  - `packages/langgraph/src/state/AgentState.ts`: 新增 `consecutiveFailures` 字段。
  - `packages/langgraph/src/nodes/actNode.ts`: 统计每轮工具成功/失败数，更新 `consecutiveFailures`（成功则重置为 0）。
  - `packages/langgraph/src/nodes/observeNode.ts`: 当 `consecutiveFailures >= 3` 时停止循环并返回友好提示。
  - `packages/core/src/tools/BenchmarkTools.ts`: `benchmark_run` / `benchmark_report` 失败结果补充 `error` 字段，避免 actNode 显示"未知错误"。
- **构建**: `pnpm --filter core build && pnpm --filter langgraph build && pnpm --filter server build` 全部成功。

- **九种方案对比**:
  1. 文件拆分（当前方案）— 主流基础做法，零成本，检索弱
  2. SQLite 数据库 — 失去 Git 版本控制，不推荐
  3. 向量数据库 + RAG — 最前沿，AI 项目天然适配，中期推荐
  4. 结构化 CMS（Notion）— 过重，不适合单人开发
  5. Docs-as-Code（VitePress）— 长期推荐，需构建步骤
  6. Git Wiki — 收益低，不推荐
  7. Obsidian 图谱式 — 个人使用高度实用（加权 4.35/5）
  8. Markdown Chunker — 自动化辅助，不能作为主方案
  9. Hybrid 混合 — 业界最佳实践，分层叠加
- **Obsidian 九维度分析**: 功能完整性5、学习成本4、性能扩展5、离线可用性5、Git兼容性3（核心权衡）、AI友好度4、可视化导航5、生态插件5、成本迁移5；加权总评 4.35/5；结论：作为辅助浏览工具开箱即用，不替代 VS Code 做 Git 操作。
