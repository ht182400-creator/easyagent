- **问题**: `GitHub 下载失败 (415): Unsupported Media Type`

- **实测证据**: 头部 8 字节 `3c 64 69 76 20 69 64 3d` = `<div id=`
- **修复**: 重写 `getReadmeHtml()` 绕过 `request<T>()`，直接用 `res.text()` → 拿到 11555B 完整 HTML
- **教训**: GitHub 的 `+json` 媒体类型不一定返回 JSON；用真实 fetch 探测比看文档可靠

- **教训**: tsup `splitting: false` 会内联所有 import 子模块，任何**路径敏感**代码（worker 入口、native binding 等）必须独立打包

### 协议设计决策
- **官方协议**保持不变：对象式 `export default { name, version, register?, getTools?, ... }`（已用 2 个多月）
- **函数式为兼容入口**：`export default function (context) { ... }`，元信息从 manifest.json 读
- 两种签名是**等价**的 —— 函数式是对象式 + `register` 字段的语法糖
- Worker 内部统一用同一个 `sandboxContext` API（`registerTool` / `registerTools` / `registerSkill` / `registerHook`）
- **教训**：定义 SDK 协议时，函数式签名比"必须是对象"更友好；可以同时支持多种导出形式

## [模块:F10] 插件市场显示"已安装"但按钮仍提示安装 (23:46)
- **问题**: 截图显示 `obsidian-doc-viewer` 已经在"已安装插件"列表（带绿色"已启用"标签），但"社区插件市场"标签下的对应卡片仍显示"使用"按钮而非"已安装"徽标。
- **根因**: `fetchInstalled` 把 `id` 字段回填为 `local:<name>` 形式（兼容 id 缺失的本地插件），而 `fetchMarketplace` 注入"已安装"状态时只按 `installedIds.has(p.id)` 严格匹配。marketplace 接口返回的 `p.id` 是 `obsidian-doc-viewer`，与 `local:obsidian-doc-viewer` 永远不相等 → 按钮全部错误显示"使用/安装"。
- **修复**:
  - `packages/frontend/src/stores/pluginsStore.ts:188-203` `fetchMarketplace` 改用 `installedNames` Set 兼容 `id` 和 `name` 两种匹配形式（`installedIds.has(p.id) || installedNames.has(p.id) || installedNames.has(p.name)`）。
- **测试**: `pluginsStore.test.ts` 新增 1 用例（`fetchMarketplace` 应通过 name 匹配已安装插件），共 8/8 通过。
- **教训**: 数据合并/去重场景下，单一字段匹配（尤其是 `id` 形式可能被人改写的情况下）极易踩坑。应同时按多个稳定字段匹配（id + name），或统一规范化 id 形式。
- **状态**: ✅ resolved
