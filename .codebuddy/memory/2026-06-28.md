## 发版 push 竞态条件根治 (00:25-00:28)
- 问题：release-publish.bat 每次 git push origin main 都被 rejected (non-fast-forward)
- 根因：发版构建期间（3-5分钟），CI pipeline sync 推了管线数据 commit 到 main
- 第二层问题：git hooks(post-commit)自动运行 unified-sync 修改 docs/pipeline/*.json，导致 pull --rebase 失败
- 修复：scripts/release.mjs 和 release-publish.bat 两处的 push 前加 `git stash → git pull --rebase origin main → git stash pop`
- 安全：远程 CI 只改 pipeline JSON，本地只改 version/CHANGELOG，文件集不相交，永无冲突

## CI Sync Pipeline Data rebase 冲突根治 (10:32)
- 问题：发版时两个 commit 几乎同时推送 → 两个 CI sync job 并行 unified-sync → pull --rebase merge conflict
- v0.6.14 CI 验证：Release 全绿（Build Desktop/Build Web 修复生效），CI sync-pipeline-data 因冲突失败
- 根治方案：ci.yml Commit pipeline data 步骤改为冲突时 rebase --abort → reset --hard origin/main → 重跑 unified-sync → commit → push
- 核心原理：管线文件是全量生成的，reset 后重跑 unified-sync 得到基于最新远程 + 本轮测试报告的完整数据，无数据丢失

## D方案：CI/CD可复用Workflow优化 (12:07)
- 问题：每次发版 3 个 workflow（2 CI + 1 Release），CI 为 artifacts commit 重复跑 8 个 job
- 方案：提取测试 job 为 `_test.yml`（workflow_call），ci.yml + release.yml 共同引用
- 改动：新建 `_test.yml`（7 job），重写 `ci.yml`（删测试 job，引用 _test.yml），`release.yml`（加 tests 质量门禁 + sync-pipeline），`scripts/release.mjs`/`release-publish.bat`（commit 加 [skip ci]）
- 效果：发版 workflow 3→1，测试配置唯一真相来源，质量门禁（测试不通过不构建）
- 日常代码同步（push main 不发版）：零影响，ci.yml 触发机制不变
- 文档：docs/49_CI_CD工作流优化_D方案_可复用Workflow.md
- 验证：YAML结构 × 工作流引用 × job依赖链 × artifact一致性 × [skip ci]标记 × lint检查 → 全部通过
