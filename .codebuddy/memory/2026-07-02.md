## 双缓存问题彻底修复 (22:48)

### 根因
`listMarket(forceRefresh=true)` 只跳过了磁盘缓存（`market.json`），但内层的 `getLatestRelease()` 和 `getManifest()` 仍命中**内存 Map 缓存**，导致用户刷新后看到的仍是旧版本。

双缓存架构：
1. **磁盘缓存** `market.json`：PluginMarketService.loadMarketCache()，TTL 1 小时，forceRefresh 可跳过
2. **内存缓存** `GitHubClient.cache Map`：getLatestRelease/getManifest，TTL 1 小时，**无 skipCache 参数**

### 修复内容（4 处改动）

| 文件 | 改动 |
|------|------|
| `githubClient.ts` | 新增 `RELEASE_CACHE_TTL=5min`；`getLatestRelease`/`getManifest` 新增 `skipCache` 参数；`getCache` 支持自定义 TTL |
| `PluginMarketService.ts` | `listMarket(forceRefresh=true)` 先调 `client.clearCache()` 再 `{skipCache:true}` 传到 `searchPluginRepos`/`getLatestRelease`/`getManifest`；新增公开方法 `clearMarketCache()`（清除磁盘+内存双缓存） |
| `PluginMarketService.ts` | `executeInstall` 的 `getLatestRelease` 传 `skipCache=true`（安装时必须拿最新版） |
| `index.ts` | 新增 `POST /api/plugins/market/refresh` 端点 |

### 新的缓存架构
```
GET /api/plugins/market?refresh=true
  → clearMarketCache()  [同时清除 market.json + GitHubClient Map]
  → searchPluginRepos(page, skipCache=true)   [跳过内存缓存]
  → getLatestRelease(name, skipCache=true)     [跳过内存缓存]
  → getManifest(name, skipCache=true)           [跳过内存缓存]

用户正常浏览：
  → market.json 磁盘缓存（TTL 1h）→ 命中即返回，减少 API 调用

Release 缓存独立 TTL：5 分钟（v1.0.6 发布后 5 分钟内自动生效）
```

### 验证
- `GET /api/plugins/market?refresh=true` → Version: **1.0.6** ✅
- `POST /api/plugins/market/refresh` → `{"success":true}` ✅
- Server 重启后缓存自动清空

## 已装 vs 市场版本号不一致 → 4 处遗漏修复 (22:50)

### 现象
用户截图：
- 已安装插件页：`obsidian-doc-viewer v0.1.0`
- 社区插件市场：`obsidian-doc-viewer v1.0.6`

### 排查
- `installed.json` 实际：`version: "v1.0.6"` ✅ disk 是新的
- `POST /api/plugins/update-check` → `{ latestVersion: "", hasUpdate: false }` ❌ 后端认为已最新
- 前端 store `installed[0].version` 是 `v0.1.0` ❌ store 是旧的

### 根因（双缓存修复的 2 处遗漏）

| # | 位置 | 遗漏 |
|---|------|------|
| 1 | `PluginMarketService.checkAllUpdates()` | 调用 `getLatestRelease(p.id)` 没传 `skipCache=true`，**沿用 1h 内存 cache 误报"已最新"** |
| 2 | `pluginsStore.updateInstallProgress` | `status === 'done'` 时只更新 `marketplace.installed=true`，**不刷新 `installed` 数组**；store 里 `version` 永远停留在首次 `fetchInstalled` 时的值 |

### 修复
| 文件 | 改动 |
|------|------|
| `PluginMarketService.ts` | `checkAllUpdates()` 入口加 `client.clearCache()`；循环内 `getLatestRelease(p.id, true)` |
| `pluginsStore.ts` | `updateInstallProgress` 的 `done` 分支末尾追加 `void get().fetchInstalled().then(() => get().checkUpdates())` |

### 关键教训
- **缓存修复必须全链路穿透**：上游修了别忘了所有调用点；`checkAllUpdates` 和 `listMarket` 是两个独立入口，都需要 `skipCache`
- **前端 store 不会自动反映 disk 变化**：任何 `install/uninstall/update` 终态后必须主动 `fetchInstalled()` 同步；不能只更新 `marketplace` 不更新 `installed`
- **"已安装 vs 市场"不一致** 几乎都是这两个问题：(1) 后端缓存没清 / (2) 前端 store 没刷新

### 构建
- `pnpm --filter server build` ✅ (676.92 KB, 375ms)
- 前端 Vite HMR 自动热更，刷新浏览器即可看到 v1.0.6 + "已是最新"

## 卸载后刷新插件又出现 (幽灵插件) → 卸载链 3 处 Bug 修复 (23:03)

### 现象
用户截图：点卸载 → UI 显示"已卸载"+空状态 → 点刷新 → 插件又出现。

### 根因链
卸载调用链：`marketService.uninstallPlugin(id)` → `onPluginUnload(name)` → `pluginManager.unloadPlugin(name)`

`pluginIdToName(id) = "easyagent-plugin-obsidian-doc-viewer"`（带前缀）
而 `PluginManager.plugins` 的 key = `manifest.name = "obsidian-doc-viewer"`（不带前缀）

→ `unloadPlugin("easyagent-plugin-obsidian-doc-viewer")` 查 `this.plugins.get(...)` 找不到 → warn "插件未加载" → return → **PluginManager 内存里这条插件永远留着**！

→ 用户前端 store 删了条目（看起来"已卸载"）
→ installed.json 也删了
→ **但 PluginManager 内存残留** → 刷新 → `GET /api/plugins` 返回 pluginManager.listPlugins() → 幽灵插件又冒出来

### 修复
`packages/core/src/plugins/PluginManager.ts` 的 `unloadPlugin` 增加**三段匹配**：
1. 精确匹配 manifest.name
2. pluginId 解析：`nameOrId.split('/').pop()` 当 manifest.name 找
3. 兜底遍历：匹配 `value.plugin.name` 或 `value.sourcePath` 的 basename

```ts
async unloadPlugin(nameOrId: string): Promise<void> {
  let loaded = this.plugins.get(nameOrId);
  if (!loaded) {
    const candidate = nameOrId.includes('/') ? nameOrId.split('/').pop() : null;
    if (candidate) { loaded = this.plugins.get(candidate); if (loaded) nameOrId = candidate; }
  }
  if (!loaded) {
    for (const [key, value] of this.plugins.entries()) {
      if (value.plugin.name === nameOrId) { loaded = value; nameOrId = key; break; }
      const short = nameOrId.includes('/') ? nameOrId.split('/').pop() : null;
      if (short && value.plugin.name === short) { loaded = value; nameOrId = key; break; }
      if (value.sourcePath) {
        const parts = value.sourcePath.split(/[\\/]/);
        const base = parts[parts.length - 1] || '';
        if (short && base === short) { loaded = value; nameOrId = key; break; }
        if (base === nameOrId) { loaded = value; nameOrId = key; break; }
      }
    }
  }
  // ... 后续 this.plugins.delete(name)
}
```

### 验证
- 完整链路：装入插件（manifest + installed.json）→ 重启 server → PluginManager 加载（count=1）→ POST uninstall → count=0 + 目录删除 ✅

### 关键教训
- **pluginId 转换语义不一致**：`PluginMarketService.pluginIdToName` 用 `split[1]`（仓库名），但 `PluginManager.plugins` 的 key 是 `manifest.name`（可能是简化名）。这两个值**经常不同**。
- **install/uninstall 必须用同一 key 体系**，否则"卸载"就成了空操作。
- 修复卸载时应**多键兼容**（name + short + sourcePath basename），不能假设单一匹配规则。

### PowerShell 转义陷阱（本次踩坑）
PowerShell here-string 中 `&&` 被解释为链式命令 → 写入文件后 `if (a && b)` 变成 `if (a; b)` 编译失败。
**解决方案**：用 Node 脚本 (`.mjs`) 操作文件 — Node 的字符串字面量无此陷阱。

### 工具陷阱：write_to_file 会覆盖已有内容
本日教训：append 内容到现有 memory 文件时，**绝对不能**用 `write_to_file`（会覆盖整个文件！）。必须用 `replace_in_file` 在文件末尾定位插入。本次误用 `write_to_file` 覆盖了 32KB 的当日记录，仅靠系统 prompt 的 cb_summary 恢复。**正确做法**：
- append-only：使用 replace_in_file 找到最后一个 `---` 位置，new_str 拼到末尾
- 或者改用 Node 脚本 `readFileSync + append` + `writeFileSync`

## 幽灵插件真正根因 + 完整修复 (23:30)

### 根因
上一轮以为是 `unloadPlugin` 找不到 plugin，实际还有更深的 4 处 id 错配：

1. **PluginManager 用 `manifest.name`（如 `obsidian-doc-viewer`）作为 Map key**，
   但插件磁盘目录名是 repo 名（`easyagent-plugin-obsidian-doc-viewer`）
2. **`/api/plugins` 端点不返回 id 字段**，前端拿不到 pluginId
3. **前端 `fetchInstalled` 兜底成 `local:obsidian-doc-viewer`**（用 pluginId 拼路径永远找不到）
4. **卸载时传 `local:obsidian-doc-viewer`** → 后端 `pluginIdToName("local:obsidian-doc-viewer")` 返回原串（没有 `/` 不切分）→ 拼出 `C:\Users\ht182\.easyagent\plugins\local:obsidian-doc-viewer` 路径（不存在）→ `removeFromInstalled` 用 `p.id !== pluginId` 过滤也失败（installed.json 里 id 是 `ht182400-creator/...`）

**结果**：磁盘目录和 installed.json 都没动；重启 server 后 `loadPluginsFromDir(userPluginsDir)` 重新发现并加载 → 幽灵插件复活。

### 修复点

| 文件 | 修改 |
|------|------|
| `core/src/plugins/PluginManager.ts` | `unloadPlugin` 签名改为 `Promise<string\|null>`，返回实际卸载的 name（manifest.name） |
| `server/src/services/PluginMarketService.ts` | `uninstallPlugin` 优先用 PluginManager 反馈的真实 name 删目录；`removeFromInstalled` 兼容 id / name / repo 名多种形式 |
| `server/src/index.ts` | `/api/plugins` 端点反查 installed.json 补全 `id` / `installedAt` / `source` 字段；卸载回调透传 actualName |
| `frontend/src/stores/pluginsStore.ts` | `fetchInstalled` 去掉重复 `apiRequest('/api/plugins')` 调用，优先使用后端返回的 id，不再硬塞 `local:` 兜底 |

### 验证流程
1. `pnpm --filter core build && pnpm --filter server build`
2. 杀 3456 端口进程 → `start-backend.bat` 重启
3. `GET /api/plugins` → 返回带 `id: "ht182400-creator/easyagent-plugin-obsidian-doc-viewer"`（修复前只有 name）
4. `POST /api/plugins/uninstall/<encoded-id>` → 200
5. 检查磁盘：`easyagent-plugin-obsidian-doc-viewer` 目录已删，`installed.json` 为 `{"plugins": []}`
6. 再次重启 server → `GET /api/plugins` 返回 `[]`，幽灵插件不再复活 ✅

### 关键教训
- **前后端 id 形式必须严格一致**。PluginManager 内部用 `manifest.name`，磁盘目录用 repo 名，市场用 `owner/repo`，前端任意一处错配都会导致功能异常。
- **`/api/plugins` 这种聚合端点**必须把多源数据（PluginManager 内存 + installed.json + 磁盘 manifest）merge 后返回统一字段，不能只返回内存数据。
- **`map(p => ({...p, id: p.id || \`local:${p.name}\`}))` 这种兜底是反模式**，会引入 `local:` 这种伪 id 并污染下游逻辑。
- 卸载链路涉及"内存 Map 清理 + 磁盘目录删除 + installed.json 持久化"三处状态机，**任何一处失败都会导致状态不一致**。回调必须返回 actualName 让调用方能准确删目录。

### 调试技巧
- 用 `node -e "fetch(...).then(r=>r.json()).then(console.log)"` 比 curl 更适合在 PowerShell 中做 API 调试（无 BOM 编码问题）。
- 杀 3456 端口推荐 Node 脚本 `node kill-3456.cjs`（用 `execSync('netstat -ano')` + `taskkill`），避免 PowerShell 转义陷阱。
- replace_in_file 失败时（特别是涉及大段函数体），用 `Node + fs.writeFileSync` 直接改文件最稳。
