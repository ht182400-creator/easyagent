## LangGraph Demo 场景 #7/#8/#9 实现 + 类型安全系统性修复

### 新增场景

### 系统性类型修复（根本性解决）

**根源问题**：
1. `ToolResult` 接口在 `actNode.ts` 中未导出，公共 API `ToolExecutor` 依赖私有类型 → 消费者无法正确实现接口
2. `mockExec` 辅助函数使用非标准形状 `{ ok, out }` 代替标准 `ToolResult { success, content, error? }`
3. 根 tsconfig.json 无 `include` 限制，导致 TS 爬入 `packages/` 目录，触发 rootDir 冲突
4. 内联 executor 对象缺少显式类型注解

**修复清单**：
| 文件 | 修改内容 |
|------|---------|
| `src/nodes/actNode.ts` | `interface ToolResult` → `export interface ToolResult` + JSDoc |
| `src/index.ts` | 新增 `ToolResult` re-export |
| `scripts/run-demo.ts` | mockExec 签名为 ToolResult 标准形状；内联 executor 添加 ToolExecutor 类型注解；移除未使用 import；所有调用点 `{ok,out}` → `{success,content}` |
| `scripts/demo-server.ts` | 同上全部修复 |
| `tsconfig.json` (根) | 添加 `include` 和 `exclude: ["packages"]` 防止跨项目类型检查 |
| `packages/langgraph/tsconfig.json` | exclude 添加 `scripts`, `demo` |
| `docs/02_约束规范.md` | 新增第八章 "类型安全与 Linter 零容忍"（7条子规则 + 质量门禁清单） |

## 场景卡片迷你有向流转图功能

### 实现方案
1. **SCENARIOS 数据扩展**: 每个场景新增 `traversalPath`（节点ID序列）、`flowDesc`（流转描述）、可选的 `keyEdges`、`parallelAct`、`isResume`、`isFullGraph`、`retryAct`、`chainAct` 标记

3. **服务端联动**: `ScenarioResult` 新增 `actualPath` 字段，每个场景执行器返回实际遍历路径，前端收到后实时更新流转图

## 流转图重复边拆分为独立箭头线（×N → 多条平行箭头 + 唯一序号）

### 需求
节点上有多次循环时，去掉节点内的 ×N 徽标，将每次流转画成独立箭头线，每条标注唯一全局序号。

### 实现
1. **`getMiniEdgePath`**：新增 `xOffset` / `yOffset` 参数（默认 0），垂直边偏移 x、水平边偏移 y 使平行线不重叠
2. **`renderMiniEdges`** 重写：
   - 统计每对 `from→to` 的出现次数，用 `computeOffsets(count)` 计算对称均匀偏移数组（间距 6px）
   - **第一遍**：绘制未被遍历的边（暗色背景，展示完整图结构）
   - **第二遍**：为每次流转独立绘制箭头线 + 唯一序号标签（标签位置跟随偏移）

4. 放大弹窗 `buildZoomSvg` 复用 `renderMiniEdges` / `renderMiniNodes`，自动受益

## Phase A：LangGraph ↔ EasyAgent Core 引擎桥接

### 目标
让 LangGraphAgent 能使用 EasyAgent 的模型适配器（BaseAdapter）和工具系统（ToolRegistry）。

### 新增文件（3 个桥接文件）

| 文件 | 行数 | 职责 |
|------|------|------|
| `packages/langgraph/src/bridge/adapterBridge.ts` | 80 | 包装 BaseAdapter.chat() → ThinkNodeConfig.chat 回调 |
| `packages/langgraph/src/bridge/toolBridge.ts` | 48 | 包装 ToolRegistry.execute() → ActNodeConfig.toolExecutor |
| `packages/langgraph/src/bridge/AgentFactory.ts` | 130 | createLangGraphAgent() 工厂函数 |
| `packages/langgraph/__tests__/integration.test.ts` | 345 | 9 个集成测试覆盖全部桥接场景 |

### 技术要点

1. **adapterBridge**：解决两套类型体系差异
   - core `Message.content: string | ContentBlock[]` ↔ langgraph `ChatMessage.content: string`（thinkNode 内部已保证为 string）
   - core `ChatResponse` 多了 `id`/`model` 字段 → bridge 剥离
   - core `ToolDefinition.parameters.properties: Record<string, ToolParameter>` ↔ langgraph `ToolDef: Record<string, unknown>`（运行时兼容）

2. **toolBridge**：解决参数个数差异
   - core `ToolRegistry.execute(name, params, context)` 需要 3 参数
   - langgraph `ToolExecutor.execute(name, params)` 只需 2 参数
   - bridge 通过闭包工厂函数注入 ToolContext

3. **AgentFactory**：支持两种创建方式
   - 方式1：传入已有 `BaseAdapter` 实例（直接复用）
   - 方式2：传入 `ProviderConfig`（内部通过 `AdapterFactory.create()` 创建适配器）
