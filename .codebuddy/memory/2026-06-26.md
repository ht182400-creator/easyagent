## [F10] Desktop "无法连接到后端服务" — CORS origin 拒绝 file:// 协议 (01:41 → 01:50 改进)

  2. 三元运算符 `corsOrigin ? corsOrigin : callback` 有歧义——CORS_ORIGIN 可能被 Electron 环境继承或设置
- **最终修复 (01:50)**:
  1. `packages/server/src/index.ts:426` — **完全移除三元运算符**，始终使用回调函数；若 `CORS_ORIGIN` 设定了则精确匹配，否则允许 file://、localhost、127.0.0.1

  3. `npx @electron/asar extract` 解包 `app.asar` → 替换 server dist → `npx @electron/asar pack` 重新打包 (mtime: 01:47)

- **真正根因**: 
  1. `config.tsx` 中 `ConfigProvider` 用 `useEffect` 同步 `apiBase`/`isDesktop` 到 `request.ts` 模块级变量

  1. `config.tsx:8,59` — `useEffect` → `useLayoutEffect`（保证在子组件 effect 之前同步执行）

## [F12] 字节扫描假阳性 + Desktop EXE 修复 (04:46-05:10)

### 问题
desktop EXE 启动后"无法连接到后端服务"。字节扫描 MODULE_VERSION 始终返回 116，看似始终是错误版本。

### 根因
1. **字节扫描陷阱**: better-sqlite3 静态链接 sqlite3.c (13万行)，`readUInt32LE` 扫描总是碰到 sqlite3 源码中的常量 116，与实际 NODE_MODULE_VERSION 无关。这导致每次都以为 "版本又被覆盖了"，白白重复编译 2 小时。
2. **prebuild-install 覆盖**: `pnpm install` 时 better-sqlite3 的 `prebuild-install` 下载 pnpm store 中的老二进制覆盖编译产物。

### 修复
1. **`.pnpmfile.cjs`**: 删除 better-sqlite3 的 `install` 脚本，阻止 prebuild-install 
2. **重写 `rebuild-sqlite3.mjs`**: 
   - 移除不可靠的字节扫描
   - SHA256 比对 Electron≠System 来验证编译正确性
   - System 版本用 `createRequire` 直接加载验证
   - 新增 `--verify` 模式
3. **验证 Desktop EXE**: 
   - 确保 `better_sqlite3_electron.node` SHA256 与 System 版本不同
   - 打包前切换 → 打包 → 恢复
   - EXE 中 `app.asar.unpacked/.../better_sqlite3.node` SHA256=13751beb7c07167d ✅ 匹配 Electron 30 版本

### 结果
- System 版本: require 加载成功 ✅
- Electron 版本: SHA256 与 system 不同 ✅
- Desktop EXE: 打包验证通过 ✅
- 分析文档: `docs/35_MODULE_VERSION问题分析与根治方案.md`

### 关键教训
**绝不要用字节扫描方式验证 .node 的 MODULE_VERSION！** 静态链接 sqlite3 源码的二进制中充满巧合值。正确方式：
- System: `require()` + 内存数据库测试
- Electron: SHA256 比对系统版本
- 命令: `node scripts/rebuild-sqlite3.mjs --verify`
