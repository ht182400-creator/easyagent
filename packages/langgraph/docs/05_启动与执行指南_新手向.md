# 启动与执行指南（新手向）

> 目标读者：第一次接触 LangGraph 模块的开发者  
> 前置要求：已安装 Node.js ≥20、pnpm  
> 文档版本：2026-06-29

---

## 一、快速开始 —— 3 秒跑起来

在项目根目录或 `packages/langgraph` 下打开终端：

```batch
cd packages\langgraph
start-demo.bat
```

脚本会依次执行：
1. 检查配置文件 `langgraph.config.json`
2. 检查 `dist/` 构建产物（不存在则自动 `pnpm build`）
3. 运行 9 个 Demo 场景，在终端中展示有向图流转过程

**首次运行耗时约 10-20 秒**（含首次构建），后续运行 2-5 秒。

## 二、启动模式详解

`start-demo.bat` 支持两种运行模式，通过参数切换：

| 命令 | 模式 | 说明 |
|------|------|------|
| `start-demo.bat` | 终端模式 | PowerShell/CMD 中输出 ANSI 彩色日志 |
| `start-demo.bat --web` | Web UI 模式 | 自动打开浏览器，可视化有向图和场景卡片 |
| `start-demo.bat --skip-build` | 终端 + 跳过构建 | 当 `dist/` 已存在时使用，跳过 tsup 编译 |
| `start-demo.bat --web --skip-build` | Web UI + 跳过构建 | 参数可组合使用 |

### 2.1 终端模式输出内容

运行后终端会依次展示 9 个场景：

```
场景 1: 纯文本对话           → START → think → END
场景 2: 工具调用循环           → START → think → act → observe → think → END
场景 3: 多工具并行执行         → think → act(并行两个工具) → observe → think → END
场景 4: maxTurns 安全终止      → think → act → observe (×3) → END
场景 5: Checkpoint 持久化      → run → 保存快照 → resume → 恢复上下文
场景 6: 有向图 ASCII 结构      → 打印完整图结构
场景 7: 上下文摘要与压缩        → 长对话 → 自动摘要 → 压缩后继续 → END
场景 8: 工具失败自动重试        → act(失败) → observe → think(重试) → act(修正) → END
场景 9: 链式工具调用            → act(tool-A) → observe → think → act(tool-B依赖A) → END
```

日志使用 **ANSI 颜色**区分（青色=标题、绿色=成功、黄色=场景分隔、红色=错误）。

### 2.2 Web UI 模式功能

启动后自动打开 `http://localhost:3455`，页面包含：

- **顶部 SVG 有向图** — 6 个节点（START / think / route / act / observe / END）用彩色连线表示流转关系
- **9 张场景卡片** — 网格布局，显示序号、图标、名称、路径描述；每张卡片内含**迷你有向流转图**（展示该场景的节点遍历路径）
- **点击卡片展开** — 右侧弹出终端风格日志面板，节点级详细执行日志
- **流转图点击放大** — 点击卡片内迷你流转图 → 全屏弹窗放大，支持鼠标滚轮缩放（30%~300%）+ 拖拽平移 + ESC 关闭
- **重复边拆分显示** — 有多个循环的场景（如 3 轮 loop），每条重复边绘制为独立平行箭头线，各标注唯一全局序号（而非节点上显示 ×N）
- **「执行全部场景」按钮** — 一键运行全部 9 个场景（含场景 6 图结构展示），执行完成后自动展开卡片详情
- **设计风格**：暗色终端风（#080c12 背景 + JetBrains Mono 等宽字体 + 霓虹色系）

## 三、等效 pnpm 命令

如果不使用批处理，也可以直接用 pnpm 命令运行：

```batch
# 终端模式 —— 直接运行 Demo 脚本
pnpm demo

# Web UI 模式 —— 启动 HTTP 服务 + 打开浏览器
pnpm demo:web

# 构建主库（demo 脚本引用 dist/index.js）
pnpm build

# 类型检查
pnpm typecheck

# 运行单元测试
pnpm test
```

这些命令定义在 `package.json` 的 `scripts` 字段中：

```json
{
  "demo": "tsx scripts/run-demo.ts",
  "demo:web": "tsx scripts/demo-server.ts",
  "build": "tsup",
  "test": "vitest run",
  "typecheck": "tsc --noEmit"
}
```

## 四、配置文件 langgraph.config.json

位于 `packages/langgraph/` 目录下，控制 Demo 运行时的日志行为：

```json
{
  "logger": {
    "level": "debug",
    "file": "",
    "modules": [],
    "console": true
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `level` | string | 日志等级：`debug` / `info` / `warn` / `error` |
| `file` | string | 日志文件路径，留空则不写文件 |
| `modules` | string[] | 模块过滤，空数组表示全部模块。例如 `["thinkNode", "actNode"]` 只输出这两个模块的日志 |
| `console` | boolean | 是否输出到控制台 |

> **Web UI 模式注意**：demo-server.ts 启动后会将日志等级强制设为 `error`，避免日志输出干扰 HTTP 响应。

## 六、内部实现详解

### 6.1 start-demo.bat 批处理流程

```
┌───────────────────────────────────────┐
│  参数解析                              │
│  --web  → MODE=web                    │
│  --skip-build → SKIP_BUILD=1          │
├───────────────────────────────────────┤
│  Step 1: 配置检查                      │
│  检测 langgraph.config.json 是否存在    │
├───────────────────────────────────────┤
│  Step 2: 构建产物检查                   │
│  若 dist/index.js 不存在 → pnpm build  │
│  若 --skip-build → 跳过               │
├───────────────────────────────────────┤
│  Step 3: 运行                          │
│  MODE=terminal → pnpm demo            │
│       → tsx scripts/run-demo.ts       │
│  MODE=web       → pnpm demo:web       │
│       → tsx scripts/demo-server.ts    │
└───────────────────────────────────────┘
```

**关键设计**：
- `chcp 65001` 设置 UTF-8 代码页，确保中文正常显示
- `cd /d "%~dp0"` 确保在批处理所在目录执行（无论从何处调用）
- `call pnpm demo` 用 `call` 调用，确保 pnpm 子进程报错时 `%errorlevel%` 能正确传递

### 6.2 run-demo.ts（终端 Demo 脚本）

**核心架构**：纯 TypeScript 脚本，脱离 vitest 测试框架，使用 Mock 工厂模拟 LLM 和工具。

```
run-demo.ts
├── Mock 工厂
│   ├── mockChat(responses[])  → 模拟 LLM 调用
│   │   返回 { content, toolCalls, finishReason, usage }
│   └── mockExec(results{})    → 模拟工具执行
│       返回 { success, content }
│
├── 9 个场景函数
│   ├── scenario1()  纯文本对话
│   │    mockChat([{ c: '你好！...' }])
│   │    createAgentGraph({ think, act }) → invoke
│   ├── scenario2()  工具调用循环
│   │    mockChat(2 步: tool_calls + stop)
│   │    模拟 get_weather(city="北京")
│   ├── scenario3()  多工具并行
│   │    mockChat(2 tool_calls: get_weather + get_time)
│   │    act 节点用 Promise.all 并行执行
│   ├── scenario4()  maxTurns 安全终止
│   │    无限 tool_calls 但 maxTurns=3 强制终止
│   ├── scenario5()  Checkpoint 持久化
│   │    new LangGraphAgent({ checkpointerConfig: { dbPath: ':memory:' } })
│   │    agent.run() → 自动保存 checkpoint
│   │    agent.resume() → 恢复上下文继续对话
│   │    agent.clearHistory() → agent.close()
│   ├── scenario6()  图结构展示
│   │    打印 ASCII 图
│   ├── scenario7()  上下文摘要与压缩
│   │    LangGraphAgent + MemoryManager 模拟长对话触发摘要压缩
│   ├── scenario8()  工具失败自动重试
│   │    mockChat(3 步: tool_calls失败 → 修正重试 → 成功)
│   │    mockExec 第一次返回失败 → observeNode 检测 → thinkNode 修正重试
│   └── scenario9()  链式工具调用
│        mockChat(3 步: tool-A → tool-B(依赖A输出) → 最终回复)
│        验证工具间数据传递的流转路径
│
├── COLOR 辅助函数
│   ├── title()   青色分隔标题
│   ├── scene()   黄色场景分隔
│   ├── ok()      绿色完成标识
│   └── info()    青色键值对
│
└── main()
    1. loadLogConfig() 加载日志配置
    2. 依次执行 scenario1~6
    3. 打印总结
```

**Mock 工厂的优势**：
- **零外部依赖** — 不需要真实 API Key，不联网
- **确定性输出** — 每次运行结果完全一致，适合 CI/Demo
- **覆盖边界场景** — 无限循环、工具失败等难以用真实 LLM 复现

### 6.3 demo-server.ts（Web UI 服务器）

**核心架构**：Node.js 内置 `http` 模块实现的轻量 HTTP 服务器，端口 3455（LangGraph 专用，避免与主后端 3456 冲突）。

```
demo-server.ts (619 行)
├── 日志配置
│   loadLogConfig() + setGlobalLevelByName('error')
│   降低日志等级，避免 vs code 干扰 HTTP 响应
│
├── 数据结构
│   ├── GRAPH_DATA    → 6 节点 + 6 边（JSON 格式）
│   └── SCENARIOS[]   → 9 场景元数据（名称/路径/图标/描述）
│
├── Mock 工厂（同 run-demo.ts）
│   ├── mockChat()
│   └── mockExec()
│
├── 场景执行器
│   ├── runScenario1() ~ runScenario9()
│   │   与 run-demo.ts 中对应场景逻辑相同
│   │   额外产出 StepLog[]（节点级详细日志）
│   └── scenarioRunners = { 1: ..., 9: ... }
│
├── HTTP 路由（handleRequest）
│   ├── GET  /api/graph        → GRAPH_DATA JSON
│   ├── GET  /api/scenarios    → SCENARIOS[] JSON
│   ├── POST /api/run/:id      → 执行单个场景，返回 ScenarioResult
│   ├── POST /api/run-all      → 依次执行 1-9，返回 ScenarioResult[]
│   ├── GET  / 或 /index.html  → 返回 demo/index.html
│   └── OPTIONS                → CORS 预检响应
│
└── 启动
    server.listen(3455) + 自动 open browser
    输出服务地址，Ctrl+C 停止
```

**API 响应格式**（以 `/api/run/2` 为例）：

```json
{
  "id": 2,
  "success": true,
  "turnCount": 2,
  "messageCount": 4,
  "output": "北京今天晴天，气温 25°C，适合出行。",
  "logs": [
    { "node": "START",  "type": "enter",    "message": "接收用户输入" },
    { "node": "think",  "type": "info",     "message": "LLM 返回 tool_calls: get_weather" },
    { "node": "route",  "type": "decision", "message": "决策: 有 tool_calls → act" },
    { "node": "act",    "type": "info",     "message": "工具结果: 晴天 25°C" },
    { "node": "observe","type": "enter",    "message": "观察执行结果" },
    { "node": "think",  "type": "info",     "message": "LLM 返回最终自然语言回答" },
    { "node": "route",  "type": "decision", "message": "决策: 无 tool_calls → END" }
  ],
  "duration": "5ms"
}
```

### 6.4 demo/index.html（可视化前端）

纯静态 HTML 页面，无框架依赖，通过 `fetch` API 与后端交互：

| 功能 | 实现方式 |
|------|----------|
| 页面加载 | `GET /api/graph` + `GET /api/scenarios` → 渲染图和卡片 |
| 卡片点击 | 根据场景 ID → `POST /api/run/:id` → 右侧展示日志 |
| 全部执行 | `POST /api/run-all` → 依次执行 9 个场景，执行完成后自动展开卡片详情 |
| SVG 图 | 纯 CSS 动画 + 内联 SVG，节点对应的 CSS 变量控制颜色 |
| 卡片入场 | `@keyframes cardEnter` 逐个延迟（`animation-delay: calc(var(--i) * 0.08s)`） |
| **迷你有向流转图** | 每张卡片内嵌迷你 SVG（`renderMiniFlowGraph`），根据 `traversalPath` 高亮遍历节点和边 |
| **重复边拆分** | `renderMiniEdges` 为每次流转绘制独立平行箭头线（6px 间距均匀分布），每条标注唯一全局序号；`renderMiniNodes` 不再显示 ×N |
| **流转图放大** | `openFlowZoom` 全屏弹窗展示放大 SVG（viewBox 290px vs 卡片内 250px）；`handleZoomWheel` 滚轮缩放 (0.3x~3.0x, 以鼠标位置为中心)；`handleZoomMouseDown/Move/Up` 拖拽平移；ESC/遮罩点击/✕按钮关闭 |
| 执行状态联动 | 服务端 `ScenarioResult.actualPath` → 前端收到后实时更新流转图 |

## 七、有向图结构速查

```
         ┌──────────┐
         │  START   │  ← 接收用户输入的入口节点
         └────┬─────┘
              │ addEdge
              ▼
       ┌─────────────┐
       │   think     │  ← LLM 调用节点：分析输入、决定是否调用工具
       │ (LLM 思考)   │
       └──────┬──────┘
              │ routeAfterThink（条件边）
              │
     ┌────────┴──────────┐
     │                   │
 有 tool_calls      无 tool_calls
     │                   │
     ▼                   ▼
┌─────────┐          ┌──────┐
│   act   │          │ END  │  ← 停止，返回最终回答
│(执行工具)│          └──────┘
└────┬────┘
     │ addEdge
     ▼
┌──────────┐
│ observe  │  ← 观察结果：turnCount < maxTurns → 继续
│ (观察)    │                        turnCount ≥ maxTurns → END
└────┬─────┘
     │ addEdge（循环边）
     ▼
(回到 think)
```

**关键规则**：
- `think` → `act` 的条件：LLM 返回了 `tool_calls`
- `think` → `END` 的条件：LLM 返回纯文本，无工具调用
- `observe` → `think` 循环继续的条件：`turnCount < maxTurns`
- `observe` → `END` 终止的条件：`turnCount ≥ maxTurns`（安全阀）

## 八、常见问题 FAQ

### Q1: 首次运行报错 `tsx: command not found`？

确保已安装依赖：
```batch
cd packages\langgraph
pnpm install
```

### Q2: Web UI 页面空白 / 一直加载？

1. 确认服务器已启动（应显示 `服务地址: http://localhost:3455`）
2. 检查端口是否被占用：`netstat -ano | findstr 3455`
3. 尝试手动访问：`http://localhost:3455/api/graph` 应返回 JSON

### Q3: 终端模式中文乱码？

确保终端使用 UTF-8 编码。在 PowerShell 中先执行：
```powershell
chcp 65001
```
或直接用 `start-demo.bat`（批处理内置了 `chcp 65001`）。

### Q4: 如何只看某个场景的日志？

终端模式下，手动执行单个场景需要改代码。推荐用 Web UI 模式，点击对应卡片单独查看 9 个场景中的任意一个。

### Q5: 构建报错 `tsup: command not found`？

tsup 安装在 `devDependencies` 中。运行 `pnpm install` 后应可用。
如果仍不行，从项目根目录执行：
```batch
pnpm install --filter @easyagent/langgraph
```

### Q6: 如何调整日志详细程度？

编辑 `langgraph.config.json`：
- Demo/调试：`"level": "debug"`
- 生产环境：`"level": "error"`
- 只看特定模块：`"modules": ["thinkNode", "actNode"]`

### Q7: Web UI 端口 3455 被占用？

修改 `scripts/demo-server.ts` 中的 `const PORT = 3455;` 为其他值。
常见冲突：确认 EasyAgent 主后端是否在 3456 运行（LangGraph Demo 已改为 3455，两者不冲突）。

### Q8: 终端模式的 Mock 场景和真实 LLM 有什么区别？

终端 Demo 使用 Mock 工厂**模拟** LLM 和工具调用，目的是：
- 验证有向图的节点流转、条件路由、循环控制逻辑
- 不依赖外部 API，离线可运行

要测试真实 LLM 集成，需运行 `pnpm test`（vitest 测试包含真实 API 集成测试）。

### Q9: 流转图放大后如何操作？

点击卡片内的迷你有向流转图即可全屏放大。放大弹窗支持：
- **🖱️ 滚轮** — 缩放 30%~300%，以鼠标位置为缩放中心
- **🖱️ 拖拽** — 平移视图（grab 光标）
- **ESC** — 关闭弹窗
- **点击遮罩区域 / ✕按钮** — 关闭弹窗

卡片内流转图 hover 时右下角会显示 `🔍 点击放大` 提示。

### Q10: 为什么有的场景卡片内有 11 条箭头线？

多轮循环场景（如场景4 maxTurns=3）会重复经过 think→route→act→observe 节点 3 次。
系统将每次流转绘制为独立的平行箭头线（间距 6px），每条标注唯一全局序号（1~11）。
这比旧版的节点内显示 ×3 更清晰地展示了完整执行路径和顺序。

### Q11: 为什么场景6（图结构）执行后不展开？

已在 v2026-06-29 修复。`runAllScenarios()` 中场景6走特殊分支（直接渲染，不调 API），
之前漏掉了自动展开逻辑。现已补充 `detail6.classList.add('card-detail--open')`。

## 九、文件编码注意事项

**Windows 批处理 (.bat) 不要使用 Unicode 制表符**：

```batch
REM ❌ 错误 —— ╔╗╚╝║ 等字符在 CMD 中可能乱码或被解释为命令
echo ╔════════════════╗
echo ║  标题          ║
echo ╚════════════════╝

REM ✅ 正确 —— 使用 ASCII 字符
echo ====================
echo   标题
echo ====================
```

即使设置了 `chcp 65001`，某些多字节 Unicode 字符在 CMD 解析器中仍不稳定。
