## 一、通用编码规范

### 1.1 JSDoc 注释

```typescript
/**
 * 每个导出函数/类必须有 JSDoc 注释
 * @param paramName - 参数说明（中文）
 * @returns 返回值说明
 */
export function myFunction(paramName: string): ResultType { }
```

### 1.2 错误处理

```typescript
/**
 * 始终使用 try-catch 包裹可能出错的代码
 * 记录详细的错误日志
 */
try {
  const result = await riskyOperation();
  return result;
} catch (error) {
  logger.error({ err: error }, '[模块名] 操作失败描述');
  throw new AgentError('友好的错误信息', { cause: error });
}
```

### 1.3 行内注释

```typescript
// 关键决策点需要中文注释解释 WHY
if (turnCount >= maxTurns) {
  // 达到最大轮次，强制终止以防止无限循环
  return { shouldContinue: false };
}
```

## 二、LangGraph 特定约定

### 2.1 State Reducer

```typescript
import { Annotation } from '@langchain/langgraph';

// ✅ 推荐：使用 Annotation API
const AgentState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (prev, next) => [...(prev ?? []), ...next],
    default: () => [],
  }),
  turnCount: Annotation<number>({
    reducer: (_, next) => next,  // 替换
    default: () => 0,
  }),
});

// ❌ 避免：直接修改 state（不可变！）
// state.messages.push(newMsg);  // 错误！
```

### 2.2 节点函数签名

```typescript
// ✅ 推荐：接收完整 state，返回部分更新
export async function thinkNode(
  state: typeof AgentState.State
): Promise<Partial<typeof AgentState.State>> {
  // 处理逻辑
  return { turnCount: state.turnCount + 1 };
}
```

### 2.3 条件路由

```typescript
// ✅ 推荐：返回下一个节点名称的字符串
export function routeAfterThink(
  state: typeof AgentState.State
): 'act' | '__end__' {
  const lastMessage = state.messages[state.messages.length - 1];
  if (lastMessage instanceof AIMessage && lastMessage.tool_calls?.length) {
    return 'act';
  }
  return '__end__';
}
```

### 2.4 图命名约定

- 图变量名：`camelCase` + `Graph` 后缀 → `agentGraph`
- 编译后的图：`camelCase` + `App` 后缀 → `agentApp`
- 节点名：`kebab-case` (string) → `'think'`, `'act'`, `'observe'`

### 2.5 文件结构

```
src/
├── state/
│   └── AgentState.ts       ← 状态定义（单个文件，Annotation API）
├── nodes/
│   ├── thinkNode.ts        ← ❌ 不做 side-effect 导入
│   ├── actNode.ts          ← ❌ 不修改全局状态
│   └── observeNode.ts      ← ❌ 不执行网络请求之外的副作用
├── edges/
│   └── routeAfterThink.ts  ← 纯函数，基于 state 返回路由决策
├── graph/
│   └── agentGraph.ts       ← 图构建 + 编译（唯一有副作用的地方：注册节点/边）
├── memory/
│   ├── Checkpointer.ts     ← 实现 BaseCheckpointSaver
│   └── Memory.ts           ← 长期记忆接口
├── Agent.ts                ← 主类，封装图的 invoke/stream/resume
└── index.ts                ← 只做 re-export
```

## 四、命名约定汇总

| 范畴 | 约定 | 示例 |
|------|------|------|
| 图变量 | `XxxGraph` (未编译) | `agentGraph` |
| 编译后 | `XxxApp` | `agentApp` |
| 节点函数 | `xxxNode` | `thinkNode`, `actNode` |
| 路由函数 | `routeAfterXxx` | `routeAfterThink` |
| 状态注解 | `XxxState` | `AgentState` |
| 类 | `PascalCase` | `LangGraphAgent` |
| 文件名 | `PascalCase` 或 `camelCase` | `AgentState.ts`, `thinkNode.ts` |

## 五、错误类型定义

```typescript
/**
 * Agent 错误基类
 * 所有 LangGraph Agent 抛出的错误统一使用此类型
 */
export class AgentError extends Error {
  constructor(
    message: string,
    public readonly code: AgentErrorCode,
    public readonly details?: Record<string, unknown>
  ) {
    super(message);
    this.name = 'AgentError';
  }
}

export type AgentErrorCode =
  | 'MAX_TURNS_EXCEEDED'    // 超过最大轮次
  | 'ADAPTER_ERROR'         // LLM 调用失败
  | 'TOOL_EXECUTION_ERROR'  // 工具执行失败
  | 'CHECKPOINT_ERROR'      // 状态恢复失败
  | 'ABORTED'               // 用户取消
  | 'INVALID_STATE';        // 非法状态
```

## 六、测试规范

### 6.1 测试框架与工具

```typescript

import { MemorySaver } from '@langchain/langgraph';
import { describe, it, expect, beforeEach } from 'vitest';

describe('agentGraph', () => {
```

### 6.2 新增模块必须同步测试案例文档 🔴 强制

> **核心规则：每次新增模块/测试文件，必须立即同步更新测试案例文档和 CI 配置。**
>
> 这是强制要求，不得延后或省略。

| # | 必须同步的内容 | 目标文件 | 操作 |
|---|-------------|---------|------|
| 1 | **新增测试案例**添加到测试案例文档 | `docs/03_测试案例文档.md` | 新增子章节（3.29/3.30...）列出所有用例 |
| 2 | **更新模块统计表** | `docs/03_测试案例文档.md` §四 4.1 | 在"按模块统计"表中新增行 |
| 3 | **更新测试数据统计** | `docs/03_测试案例文档.md` §八 | 更新测试文件数、用例总数、分类统计 |
| 4 | **更新概览页** | `docs/03_测试案例文档.md` §一 | 更新文件头部的测试总数/通过率/覆盖模块 |
| 5 | **CI 工作流同步** | `.github/workflows/_test.yml` | 新增 test 作业（build + vitest run + upload artifact） |

**同步时机**：测试代码编写完成且 `pnpm exec vitest run` 全通过后，立即执行上述 5 项更新。

**示例**（LangGraph 引擎新增）：

```
新增模块: packages/langgraph (5 个测试文件, 57 用例)
↓ 立即同步
1. docs/03_测试案例文档.md → 新增 §3.29 + §3.30, 更新 §四/§八
2. .github/workflows/_test.yml → 新增 test-langgraph job
```

### 6.3 CI 测试作业模板 🔴 强制

新增包的 CI job 必须遵循以下模板（复用 `_test.yml` 的模式）：

```yaml
test-langgraph:
  name: Test LangGraph
  needs: test-core
  runs-on: windows-2022
  steps:
    - uses: actions/checkout@v4
    - uses: pnpm/action-setup@v4
    - uses: actions/setup-node@v4
      with:
        node-version: ${{ env.NODE_VERSION }}
        cache: 'pnpm'
    - uses: ilammy/msvc-dev-cmd@v1
    - name: Install dependencies
      run: pnpm install --frozen-lockfile
    - name: Build core + langgraph
      run: |
        cd packages/core && npx tsup --dts
        cd ../langgraph && npx tsup
    - name: Run langgraph tests
      run: cd packages/langgraph && npx vitest run
```

> **关键约束**：`needs: test-core` 确保 Core 先通过；`Build core` 是必须的（langgraph 依赖 `@easyagent/core: workspace:*`）。

## 八、类型安全与 Linter 零容忍

### 8.1 根本性修复原则

> **核心规则：发现 linter/编译器错误必须根本性修复，禁止"只修新增问题"。**
>
> 所有类型不兼容、未使用变量、rootDir 配置错误等问题，无论是否为本次修改引入，
> 一旦发现必须根治。允许的例外：经过评审确认属于外部依赖 bug 或 tsconfig 有意为之的豁免项。

### 8.2 公共 API 类型必须导出

```typescript
// ✅ 正确：公共接口依赖的所有类型都要导出
export interface ToolExecutor {
  execute(name: string, params: Record<string, unknown>): Promise<ToolResult>;
}

export interface ToolResult {
  success: boolean;
  content: string;
  error?: string;
}

// ❌ 错误：ToolResult 未导出，消费者无法正确实现 ToolExecutor 接口
// interface ToolResult { ... }  // 私有 — 外部无法使用此类型
```

### 8.3 Mock / 测试辅助函数类型对齐

```typescript
// ✅ 正确：mock 函数返回类型与接口完全对齐
import type { ToolExecutor, ToolResult } from '../src/index';

function mockExec(results: Record<string, ToolResult> = {}): { execute: ToolExecutor['execute'] } {
  return {
    execute: async (name: string, _params: Record<string, unknown>): Promise<ToolResult> => {
      const r = results[name];
      if (r) return r;
      return { success: true, content: `${name} 完成` };
    },
  };
}

// ❌ 错误：使用非标准形状 { ok, out } 替代标准 ToolResult { success, content, error? }
// function mockExec(results: Record<string, { ok: boolean; out: string }> = {}) { ... }
```

### 8.4 内联类型声明

```typescript
// ✅ 正确：显式类型注解确保类型兼容
const retryExec: ToolExecutor = {
  execute: async (name: string, _params: Record<string, unknown>) => { ... },
};

// ❌ 错误：依赖类型推导，可能在接口变更时静默失败
// const retryExec = { execute: async (name, _params) => { ... } };
```

### 8.5 未使用导入零容忍

```typescript
// ✅ 正确：精确导入，不引入未使用的符号
import type { ToolResult, ToolExecutor } from '../src/index';
import { createAgentGraph, LangGraphAgent, loadLogConfig } from '../src/index';

// ❌ 错误：导入后未使用（即使运行时无害）
// import { setGlobalLevelByName, LogLevel } from '../src/index';  // 未使用！
```

### 8.6 tsconfig 作用域控制

```json
// ✅ 正确：通过 include/exclude 精确控制 tsconfig 作用域
{
  "include": ["src/**/*.ts"],
  "exclude": ["node_modules", "dist", "__tests__", "scripts", "demo"]
}

// ❌ 错误：不限制 include 导致 ts 爬入未预期的目录（如 packages/ 或 scripts/）

```

### 8.7 质量门禁检查清单

修改完成后必须逐项验证：

| # | 检查项 | 验证方式 |
|---|--------|---------|
| 1 | `read_lints` 零 ERROR | 对修改文件执行 `read_lints` |
| 2 | 公共类型全部导出 | 检查 `index.ts` re-export 列表 |
| 3 | mock/测试函数类型对齐 | 确认返回类型与接口签名一致 |

| 5 | tsconfig 作用域正确 | include/exclude 不跨项目边界 |
| 6 | 运行验证通过 | `npx tsx scripts/run-demo.ts` 全场景通过 |
